# TiKi 해결 가이드 (2026-06-24)

이번 [필수] 두 가지 해결법입니다. 우선순위는 "없는 경로 500" → "주문 재고/취소"입니다.

---

## 1. 모르는 경로가 500 대신 404 나게 하기 (`[eventId]`)

### 왜 이렇게 되나

`/admin`처럼 진짜 이벤트가 아닌 경로가 `src/app/[eventId]/page.tsx`로 흘러들어가면 `eventId = "admin"`인 채로 쿼리들이 실행됩니다. `getEventDetail`엔 UUID 가드가 있는데 같이 부르는 `getSlots`/`getGrades`/`getReviews`엔 가드가 없다 보니, Postgres가 "uuid 형식 아님(22P02)" 에러를 던지고 그게 `Promise.all`을 깨뜨려서 페이지가 통째로 500이 나는 겁니다.

### 방법 A (권장) — 페이지 진입 시 UUID가 아니면 바로 404

쿼리를 부르기 전에 막는 게 가장 깔끔합니다.

```ts
// src/app/[eventId]/page.tsx
import { notFound } from "next/navigation";

const UUID_RE =
  /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;

export default async function EventDetailPage({ params }: { params: Promise<{ eventId: string }> }) {
  const { eventId } = await params;
  if (!UUID_RE.test(eventId)) notFound();   // ← 모르는 경로는 404

  const [event, slots, grades, reviewData] = await Promise.all([ ... ]);
  if (!event) notFound();                    // 형식은 맞지만 없는 이벤트도 404
  ...
}
```

`notFound()`를 부르면 Next가 `not-found.tsx`(없으면 기본 404 화면)를 보여줍니다. 쿼리 자체를 아예 안 부르니 DB 에러가 날 일도 없고, 응답으로 새어 나갈 것도 없습니다.

### 방법 B — 네 쿼리에 가드를 통일

페이지를 안 건드리고 싶으면, `getEventDetail`처럼 나머지 셋에도 같은 가드를 넣습니다.

```ts
// src/lib/event/queries.ts
export async function getSlots(eventId: string): Promise<Slot[]> {
  if (!UUID_RE.test(eventId)) return [];     // ← 추가
  ...
}
export async function getGrades(eventId: string): Promise<Grade[]> {
  if (!UUID_RE.test(eventId)) return [];     // ← 추가
  ...
}
export async function getReviews(eventId: string): Promise<ReviewListData> {
  if (!UUID_RE.test(eventId)) return { averageRating: 0, totalCount: 0, reviews: [] }; // ← 추가
  ...
}
```

둘 중 하나만 적용하면 되는데, 방법 A 쪽이 "없는 페이지는 404"라는 의도를 코드에서 더 분명하게 드러내 줍니다.

> UUID 정규식이 `queries.ts`에 이미 `UUID_RE`로 있으니, 페이지에서도 import해서 같은 걸 쓰면 중복이 없습니다.

## 2. 주문 재고 차감을 원자적으로 + 취소 복원 (`orders/route.ts`)

### 왜 필요한가

지금은 재고를 SELECT로 읽고 주문을 INSERT만 하는 구조입니다. 같은 좌석에 두 요청이 거의 동시에 들어오면 둘 다 "재고 있음"을 읽은 뒤 둘 다 INSERT 해버려서 초과 판매가 나고요, 취소 핸들러도 없다 보니 한 번 나간 재고가 영영 돌아오지 않습니다. 이런 재고 차감은 "읽고-쓰기"를 두 단계로 나누지 말고 **조건부 UPDATE 한 방**으로 처리하는 게 정석입니다.

### 재고 차감을 조건부 UPDATE로

```sql
-- 주문 + 재고 차감을 한 트랜잭션으로
create or replace function public.create_ticket_order(
  p_user_id uuid,
  p_grade_id uuid,
  p_qty int
)
returns uuid
language plpgsql
as $$
declare
  v_order_id uuid;
begin
  -- 재고가 충분할 때만 차감 (조건이 안 맞으면 0 rows)
  update ticket_grade
    set remaining = remaining - p_qty
    where id = p_grade_id and remaining >= p_qty;
  if not found then
    raise exception 'SOLD_OUT';   -- 매진/재고부족 → 전체 롤백
  end if;

  insert into orders (user_id, grade_id, quantity, status)
  values (p_user_id, p_grade_id, p_qty, 'ordered')
  returning id into v_order_id;

  return v_order_id;
end;
$$;
```

```ts
// orders/route.ts
const { data: orderId, error } = await supabase
  .rpc("create_ticket_order", { p_user_id: userId, p_grade_id: gradeId, p_qty: qty });
if (error) {
  const soldOut = error.message.includes("SOLD_OUT");
  return NextResponse.json(
    { success: false, message: soldOut ? "매진되었습니다." : "주문 처리 중 오류" },
    { status: soldOut ? 409 : 500 },
  );
}
```

`where remaining >= p_qty` 조건이 있으면 동시 요청이 몰려도 재고가 음수로 빠질 일이 없습니다. UPDATE가 도는 동안 DB가 행 잠금을 알아서 잡아 주기 때문입니다.

### 취소 시 복원

```sql
create or replace function public.cancel_ticket_order(p_order_id uuid, p_user_id uuid)
returns void language plpgsql as $$
declare r record;
begin
  select grade_id, quantity into r from orders
    where id = p_order_id and user_id = p_user_id and status = 'ordered';
  if not found then raise exception 'NOT_FOUND'; end if;

  update orders set status = 'cancelled' where id = p_order_id;
  update ticket_grade set remaining = remaining + r.quantity where id = r.grade_id;
end;
$$;
```

PATCH/DELETE 핸들러에서 이 함수를 부르면 취소와 재고 복원이 한 번에 됩니다.

### 매진 판정 status 통일

코드는 `"ordered"`(영문)를 쓰는데 DB에 한글("공개")이 섞여 들어가 있으면 매진 판정이나 집계가 어긋나게 됩니다. 저장값을 한쪽(영문을 권장합니다)으로 통일하고, 기존 데이터가 남아 있으면 아래처럼 마이그레이션으로 한 번 정리해 주세요.

```sql
update orders set status = 'ordered' where status = '공개';
```
