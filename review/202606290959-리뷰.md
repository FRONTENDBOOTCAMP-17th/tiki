## 공개 페이지 SEO 점검

먼저 칭찬부터 하겠습니다. SEO에서 제일 중요한 토대인 **SSR이 공개 페이지 전반에 잘 깔려 있습니다.** 메인(`src/app/page.tsx`)이 async 서버 컴포넌트로 데이터를 받아 `HeroSlider`·`HorizontalCardSection` 같은 `'use client'` 자식에게 props로 넘기는 구조라, 클라이언트 컴포넌트를 쓰면서도 데이터는 서버 HTML에 박혀 내려갑니다. 카테고리 상세(`category/[slug]/page.tsx`)도 async 서버 컴포넌트이고, 이벤트 상세(`[eventId]/page.tsx`)는 `getEventDetail`·`getSlots`·`getGrades`·`getReviews`를 `Promise.all`로 서버에서 병렬 조회한 뒤 `<h1>`에 공연 제목을 박아 내려보냅니다(page.tsx:54-60, 97-99). 크롤러가 자바스크립트를 실행하지 않아도 본문을 볼 수 있다는 뜻이라, 이 구조 자체는 아주 잘 잡혀 있습니다. 이 규율은 계속 지켜 주세요.

문제는 그 위에 얹혀야 할 메타데이터 층이 통째로 비어 있다는 점입니다.

### [필수] 루트 metadata가 create-next-app 기본값 그대로입니다

`src/app/layout.tsx:17-19`:

```tsx
export const metadata: Metadata = {
  title: 'Create Next App',
  description: 'Generated by create next app',
};
```

거기에 `layout.tsx:29`도 `<html lang='en'>`이고요. 이게 이번 SEO 점검에서 가장 큰 손해입니다. 한국어 티켓 예약 서비스인데 메인은 물론이고 **모든 이벤트 상세까지 구글 검색결과에 "Create Next App"으로 뜹니다.** `lang`도 `en`이라 검색엔진과 스크린리더가 언어를 영어로 잘못 인식합니다. `lang="ko"`로 바꾸고, `title.template`을 `'%s | TiKi'`로 잡아 하위 페이지 제목 뒤에 서비스명이 자동으로 붙게 하고, 실제 설명과 `metadataBase`(배포 도메인)를 채워주세요. 해결 코드는 가이드 1번에 적어뒀습니다.

### [필수] 이벤트 상세에 generateMetadata가 없어 모든 상세가 같은 제목입니다

`src/app/[eventId]/page.tsx`에는 `generateMetadata`가 없습니다. 그래서 공연이 100개여도 검색결과 제목·설명이 전부 동일합니다. 이벤트 상세야말로 URL이 각각 다르고 개수가 많은, SEO에서 가장 중요한 페이지입니다. 이미 `getEventDetail(eventId)`로 데이터를 가져오고 있으니(page.tsx:55), 같은 데이터로 `generateMetadata`를 하나 더 만들어 제목·설명·OG를 공연별로 내려주면 됩니다. 같은 함수를 두 번 호출하는 게 걸리면 `React.cache`로 감싸면 DB는 한 번만 칩니다(가이드 2번).

카테고리 상세(`category/[slug]/page.tsx`)도 `generateStaticParams`는 있는데 `metadata`가 없습니다. 고정 카테고리라 정적 `metadata`만 export해도 충분합니다.

### [필수] sitemap.ts / robots.ts가 없습니다

`src/app/sitemap.ts`와 `src/app/robots.ts`가 둘 다 없습니다. 사이트맵은 공개 이벤트 URL을 DB에서 펼쳐 크롤러에게 목록을 한 번에 건네주고, robots는 `/admin`·`/mypage`·`/seller`·`/payment` 같은 비공개 영역을 크롤에서 빼고 사이트맵 주소를 알려줍니다. 둘 다 App Router에선 파일 하나로 끝납니다(가이드 3번).

> SEO 개념 전반(크롤러의 눈, Metadata API, OG, JSON-LD, 검증법)은 별도로 공유한 `seo-가이드.md`에 처음부터 끝까지 정리해뒀으니 함께 읽어보면 좋겠습니다.

## 팀 공통

- **서버액션 = 공개 엔드포인트**라는 감각을 팀 전체가 공유하면 좋겠습니다. 강재훈님 항목의 admin 액션이 대표 사례인데, "이 함수는 admin 화면에서만 부르니까 안전하다"는 가정은 성립하지 않습니다. 권한이 필요한 액션은 함수 첫 줄에서 호출자를 재검증하는 걸 팀 규칙으로 두면 같은 실수가 반복되지 않습니다. 마침 `lib/auth.ts:20`에 `requireUser`, `getCurrentProfile`(role 조회)이 이미 있으니 재료는 갖춰져 있습니다.
- **SEO 메타 층은 한 사람이 layout/sitemap/robots를 한 번에 깔고**, 이후 상세 페이지 담당자가 각자 `generateMetadata`를 채우는 식으로 나누면 빠르게 정리됩니다.
- 날짜 포맷·주문 상태값처럼 **여러 파일에 흩어진 상수·유틸을 공용 모듈로 모으는 습관**을 들이면, 화면마다 표기가 어긋나거나 상태 문자열 오타로 버그가 나는 걸 줄일 수 있습니다.
