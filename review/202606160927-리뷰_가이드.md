## 1. 회원가입 API — Supabase 에러를 자체 코드로 감싸기 [제안]

`src/app/api/auth/join/route.ts:56-57`가 지금 이렇습니다.

```ts
if (error) {
  return fail(error.message); // Supabase 원문이 그대로 나감
}
```

왜 바꾸면 좋은가:

- `error.message`는 영어 원문이라 그대로 화면에 못 보여줍니다.
- "이미 가입된 이메일"인지 같은 정보가 응답에 실리면, 공격자가 어떤 이메일이 가입돼 있는지 떠볼 수 있습니다(계정 존재 노출).

자체 코드로 감싸고 원문은 서버 로그에만 남깁니다.

```ts
if (error) {
  console.error('[join] signUp 실패:', error.message); // 원문은 서버에만
  return fail('signup_failed'); // 클라이언트엔 자체 코드만
}
```

이러면 프론트는 `signup_failed`를 받아 "가입에 실패했습니다" 같은 안내 문구로 매핑하면 됩니다. 검증 실패에서 쓰는 `empty_email` 같은 자체 코드 컨벤션과도 결이 맞습니다.

같은 맥락으로 `src/app/auth/callback/route.ts:13-15`의 `console.log(data)`/`console.log(error)`도 마무리 단계에선 지우거나 개발 환경에서만 찍게 감싸 주세요(세션/유저 정보가 로그에 그대로 쌓입니다).

```ts
if (process.env.NODE_ENV === 'development') {
  console.log('endpoint: /auth/callback', { error });
}
```

## 3. Modal 포커스 접근성 [제안]

`src/components/modal/Modal.tsx`에 `role="dialog"`·`aria-modal`·ESC 닫기·스크롤 락은 이미 잘 들어가 있습니다. 남은 건 키보드 셋입니다. Modal 한 곳에 넣으면 이걸 쓰는 모든 모달이 같이 좋아집니다.

### 3-1. 포커스 트랩 + 복원

모달이 열려도 지금은 Tab을 누르면 포커스가 뒤 배경으로 빠져나가고, 닫아도 원래 버튼으로 안 돌아옵니다.

```tsx
import { useEffect, useRef, type ReactNode } from "react";

function Modal({ open, onClose, children, position = "center", className }: ModalProps) {
  const dialogRef = useRef<HTMLDivElement>(null);

  useEffect(() => {
    if (!open) return;

    // 열기 직전에 포커스를 가졌던 요소 기억(닫을 때 복원용)
    const previouslyFocused = document.activeElement as HTMLElement | null;

    const onKey = (e: KeyboardEvent) => {
      if (e.key === "Escape") onClose();

      if (e.key === "Tab" && dialogRef.current) {
        const focusables = dialogRef.current.querySelectorAll<HTMLElement>(
          'a[href], button:not([disabled]), textarea, input, select, [tabindex]:not([tabindex="-1"])',
        );
        if (focusables.length === 0) return;
        const first = focusables[0];
        const last = focusables[focusables.length - 1];

        // 모달 안에서만 Tab이 돌도록 가둠
        if (e.shiftKey && document.activeElement === first) {
          e.preventDefault();
          last.focus();
        } else if (!e.shiftKey && document.activeElement === last) {
          e.preventDefault();
          first.focus();
        }
      }
    };

    document.addEventListener("keydown", onKey);
    document.body.style.overflow = "hidden";

    // 모달이 열리면 첫 포커스를 모달 안으로
    dialogRef.current?.focus();

    return () => {
      document.removeEventListener("keydown", onKey);
      document.body.style.overflow = "";
      previouslyFocused?.focus(); // 닫을 때 원래 자리로 복원
    };
  }, [open, onClose]);

  if (!open) return null;

  return createPortal(
    <div onClick={onClose} className={/* 기존 그대로 */}>
      <div
        ref={dialogRef}
        role="dialog"
        aria-modal="true"
        aria-labelledby="modal-title"   // 3-2 참고
        tabIndex={-1}
        onClick={(e) => e.stopPropagation()}
        className={/* 기존 그대로 */}
      >
        {children}
      </div>
    </div>,
    document.body,
  );
}
```

### 3-2. aria-labelledby로 제목 연결

`role="dialog"`만 있고 제목 연결이 없으면 스크린리더가 "무슨 다이얼로그인지"를 못 읽습니다. `Modal.Header`에 `id`를 주고 루트의 `aria-labelledby`로 잇습니다.

```tsx
function ModalHeader({ children }: { children: ReactNode }) {
  return (
    <h2 id="modal-title" className="px-6 pt-6 text-lg font-bold text-mirage">
      {children}
    </h2>
  );
}
```

한 화면에 모달이 여러 개라면 `id`를 prop으로 받아 고유하게 주면 됩니다.

## 4. 토스트 다중 표시 — 스택 레이아웃 [제안]

E2E에서 토스트를 여러 개 띄우니 겹쳐 보였습니다. 토스트들을 감싸는 컨테이너를 하나 두고 세로로 쌓이게 합니다.

```tsx
// 토스트 컨테이너 (한 번만 렌더)
<div className="fixed bottom-6 right-6 z-50 flex flex-col gap-2">
  {toasts.map((t) => (
    <Toast key={t.id} variant={t.variant}>{t.message}</Toast>
  ))}
</div>
```

핵심은 (1) 컨테이너에 `flex flex-col gap-2`로 세로 정렬, (2) 개별 토스트에서 `fixed` 위치 지정을 빼고 컨테이너가 위치를 잡게 하는 것입니다. 그래야 여러 개가 차곡차곡 쌓입니다.

## 5. 다음 단계 — 인증 + 컴포넌트를 실제 도메인 화면에 연결

인증이 모양을 갖췄으니, 다음은 만들어둔 부품(Modal·Filter·인증 가드)을 실제 예매 흐름 한 화면에서 합쳐 실제로 돌아가는지 확인하는 단계라 생각합니다. 예를 들면:

- `/home`을 지금의 "로그인 성공 🎉" 데모 대신 실제 목록/예매 진입 화면으로 채우기.
- 그 화면을 서버 컴포넌트로 두고 `getUser()` 가드를 그대로 재사용(이미 `home/page.tsx`에 패턴이 있으니 복사 가능).
- Filter/SortContext로 목록을 거른 다음, 예매 확인은 합성 Modal로 띄우고 결과는 색 고친 Toast로 알리기.

부품을 한 화면에 모아두면, 여러 멤버의 코드가 한곳에서 만나고 통합 버그도 일찍 드러납니다. 큰 화면 하나를 통째로 만들기보다, 목록→상세→예매확인 토스트처럼 작은 흐름 하나를 끝까지 이어보는 걸 권합니다.
