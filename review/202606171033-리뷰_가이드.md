우선순위 순서로 정리했습니다. 1번만 풀어도 검색 기능 전체가 화면에 다시 살아납니다.

## 1. [필수] `/search` 500 복구 — next/image 외부 호스트 등록

### 무슨 일이 일어났나
인기공연 더미가 `https://picsum.photos/...` 이미지를 `next/image`로 띄우는데, 정작 `next.config.ts`에는 그 호스트가 빠져 있습니다. `next/image`는 허용되지 않은 외부 호스트를 만나면 에러를 던지고, 그게 서버 렌더 단계에서 터지면서 페이지 전체가 500이 됩니다.

### 해결 — 둘 중 하나

**(A) 호스트 등록 (가장 빠름)**
```ts
// next.config.ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  images: {
    remotePatterns: [
      { protocol: "https", hostname: "picsum.photos" },
      // 검색결과 썸네일이 외부 호스트면 그 호스트도 함께 등록
    ],
  },
};
export default nextConfig;
```

**(B) 더미 이미지를 일반 `<img>`로 (데모 데이터라 최적화 불필요할 때)**
```tsx
// EventCard.tsx — 데모/외부 임의 이미지엔 next/image 대신
<img src={item.image} alt={item.title} width={160} className="..." />
```

> 검색결과 카드(`thumbnail`)도 외부 도메인이면 같은 이유로 터질 수 있습니다. 실제로 쓰는 외부 이미지 호스트를 전부 `remotePatterns`에 등록하거나, 신뢰하기 어려운 외부 이미지엔 그냥 `<img>`를 쓰면 됩니다. next/image를 쓸 땐 "호스트를 config에 등록한다"까지가 한 세트라고 기억해 두면 좋습니다.

적용한 뒤 `/search`를 다시 띄워 200으로 뜨는지 확인해 주세요.

## 2. [제안] 서버 응답 타입을 하나로 (단일 출처)

지금은 API가 만드는 응답(`route.ts:66-78`)과 클라가 받는 형태(`page.tsx:19-27`의 `as` 단언)가 서로 따로 정의돼 있습니다. 이러면 한쪽만 바뀌어도 컴파일은 그냥 통과하고, 정작 런타임에서 깨집니다.

```ts
// types/event.ts — 서버·클라가 공유하는 한 곳
export interface EventSearchItem {
  event_id: number;
  title: string;
  startDate: string;
  thumbnail: string | null;
  // ...실제 응답 필드
}
export interface EventSearchResponse {
  items: EventSearchItem[];
  total: number;
  page: number;
  limit: number;
}
```
```ts
// route.ts — 응답을 이 타입으로 만들고
return ok<EventSearchResponse>({ items, total, page, limit });

// page.tsx — fetch 결과를 이 타입으로 받으면 toSortItems의 'as' 단언이 사라진다
const data: EventSearchResponse = await res.json();
```
이렇게 두면 서버가 필드를 바꾸는 순간 클라에서 타입 에러가 떠서, 런타임에 가기 전에 잡힙니다.

## 3. [제안] 정렬 로직 이원화 정리

`filterSort.ts`의 `useSort`/`compare`(클라 정렬)와 `search/page.tsx`의 서버 정렬이 같이 살아 있고, `changeSort`도 양쪽에 거의 똑같이 들어가 있습니다. 실제 검색은 서버 정렬로 도니까 `compare` 함수는 사실상 죽은 코드입니다.

- 실서비스가 서버 정렬이라면, `page.tsx`는 `SORT_OPTIONS`(키·라벨)만 가져다 쓰고 정렬 토글 상태 관리는 한 곳(작은 `useSearchSort` 훅)으로 모으면 됩니다. `compare`는 데모 전용이라고 분명히 표시하거나 아예 지웁니다.
- 핵심은 "정렬"이라는 한 가지 일에 진실 원천을 하나만 두는 것입니다.

## 4. [제안] API 에러 래핑

```ts
// route.ts
if (error) {
  console.error("event search failed:", error);          // 원문은 로그로
  return fail("검색 중 문제가 발생했습니다.", 500);        // 사용자에겐 자체 메시지
}
```
입력값이 잘못돼서 난 에러는 400대로 내려주면, 클라에서 구분해 처리할 수 있습니다.

## 5. [제안] proxy 인증 가드 전략

앞으로 보호 페이지가 늘어날 거라면, 페이지마다 서버 컴포넌트에서 `getUser` 가드를 반복하는 것보다 `proxy.ts`에서 한 번에 막는 쪽이 결합도 면에서 깔끔합니다.
```ts
// proxy.ts (예시 방향)
const { data: { user } } = await supabase.auth.getUser();
const protectedPaths = ["/home", "/mypage"];
if (!user && protectedPaths.some((p) => request.nextUrl.pathname.startsWith(p))) {
  return NextResponse.redirect(new URL("/login", request.url));
}
```
다만 어느 쪽으로 갈지는 팀이 상의해서 한 방식으로 통일하는 게 더 중요합니다.
