# TiKi 9차 코드 리뷰 (2026-06-19)

다음 주 월요일 사용자·관리자 기능 중간 시연을 앞두고, 8차(2026-06-18) 이후 `develop`에 들어온 변경을 봤습니다.
이번 회차는 화면만 붙던 단계를 넘어 **실제 백엔드가 연결되기 시작한** 회차입니다. 이벤트 상세·리뷰·주문 API,
판매자 이벤트 등록/수정/조회 API와 관리 화면, 그리고 로그인 미들웨어 가드까지 한꺼번에 올라왔네요.
앱을 먼저 띄워 사용자에서 판매자로 이어지는 흐름을 순서대로 돌려 보고(아래 "E2E 결과"), 거기서 드러난 동작을 근거로 코드를 봤습니다.

시연을 기준으로 보면 큰 진전이 있었고, 시연 전에 손봤으면 하는 것은 두 가지입니다. 하나는 마이페이지 로그인 가드(8차에서 이어진 것),
다른 하나는 이벤트 상세 API가 잘못된 주소를 받았을 때 데이터베이스 내부 에러를 그대로 흘리는 점입니다.

## 1. 지난(8차) 지적 반영 현황

## 2. 시연 전 우선 점검 — [필수] 두 가지

### 2-1. 마이페이지 로그인 가드 (8차 이월)

E2E로 비로그인 상태에서 `/mypage`에 들어가 봤더니, 8차와 똑같이 `/mypage/profile`이 그대로 열렸습니다(T11).
화면에는 "티키님 / tiki@gmail.com / 010-1234-5678" 프로필 폼, 친구 목록(`/mypage/friends`, T23), 라이브러리·예매내역까지
전부 비로그인으로 보입니다. 헤더도 로그인한 사용자용으로 떠 있습니다.

이번에 `/seller`는 미들웨어로 막혔는데, 그 설정이 이렇게 돼 있습니다.

```ts
// src/lib/auth/routes.ts
export const PROTECTED_ROUTES = ['/seller'];
export const AUTH_ROUTES = ['/login', '/join'];
```

`proxy.ts`가 이 목록을 보고 비로그인을 `/login`으로 보냅니다. 즉 가드 구조 자체는 이미 잘 갖춰져 있고,
**`PROTECTED_ROUTES`에 `'/mypage'` 한 줄만 더해주면** 마이페이지 전체가 같은 방식으로 닫힙니다. 가이드에 그대로 적어 두었습니다.

지금은 마이페이지가 전부 플레이스홀더라 실제 개인정보가 새는 건 아닙니다. 다만 시연에서 "로그인 안 한 사람이 내 프로필을 본다"는
모습이 그대로 보이고, 나중에 이 폼을 진짜 사용자 데이터에 연결하는 순간 가드가 없으면 곧장 정보 노출로 이어집니다. 시연 전에 닫는 걸 권합니다.

추가로, 가드를 미들웨어에서 처리하더라도 `mypage/layout.tsx`의 `<Header loggedIn ... />`은 여전히 하드코딩입니다.
레이아웃을 서버 컴포넌트로 두고 `getCurrentUser()` 결과로 `loggedIn`을 넘기면, 헤더 표시도 실제 로그인 상태와 맞습니다.

### 2-2. 이벤트 상세 API의 원문 에러 노출 [필수·신규]

E2E에서 잘못된 주소(`/api/events/1`)로 상세 API를 호출해 봤더니 이렇게 나왔습니다(T26).

```
HTTP 500
{"success":false,"message":"invalid input syntax for type uuid: \"1\"","data":null}
```

`eventId`가 uuid가 아닐 때 데이터베이스가 던진 원문 에러가 그대로 클라이언트로 나갑니다. 원인은 상세 라우트의 이 부분입니다.

```ts
// src/app/api/events/[eventId]/route.ts:42
if (error) return fail(error.message, 500);   // DB 원문을 그대로 노출
```

존재하지 않는 정상 uuid를 넣으면 `404 event_not_found`로 깔끔하게 나오니 그건 좋고, 문제는 형식이 틀린 입력을 만났을 때입니다.
데이터베이스 컬럼 타입(`uuid`)이나 내부 에러 문구가 외부로 드러나면 공격자에게 스키마 힌트를 주는 셈이고, 사용자한테는 영문 에러가 그대로 보입니다.
회원가입 API의 `error.message` 노출과 정확히 같은 패턴이라, 두 곳을 같은 방식으로 정리하면 좋겠습니다. 즉 **DB 원문은 서버 로그에만 남기고,
클라이언트에는 우리 쪽 코드(`'invalid_event_id'` 같은)와 적절한 상태코드(400/404)만 내려보내는** 식입니다(가이드 참고).

## 3. 빌드(tsc)·빌드 산출 — 건강합니다 [칭찬]

`npx tsc --noEmit` 0건, `npm run build` 성공(라우트 38개, 미들웨어 `ƒ Proxy` 인식)입니다. 백엔드 API가 한꺼번에 늘었는데도
타입이 깨지지 않은 건 좋은 신호입니다. CI(`build` + `tsc`)도 PR에서 돌고 있어 3~4차의 casing 사고가 develop으로 다시 새어들 일은 막혀 있습니다.

다만 두 가지는 계속 이월 중입니다(이번에도 [제안]).
- CI 트리거가 `pull_request`만이라, 실수로 develop에 직접 push하면 검사를 안 거칩니다. `push:` 트리거도 함께 두면 안전합니다.
- `npm run lint`(eslint) 단계가 없습니다. 타입 외에 미사용 변수·hooks 규칙도 자동으로 걸리니, 시연 뒤 여유가 생기면 넣어두면 좋습니다.

> 참고: 제 로컬 `.env.local`에 `NEXT_PUBLIC_SITE_URL`이 없어 OAuth/가입 콜백 주소 조합용으로 로컬에서만 임시로 채워 띄웠습니다.
> 학생 코드 문제는 아니고 제 실행 환경 보정이라, 사실만 적어 둡니다. 배포 환경에는 이 변수가 들어가 있어야 콜백 주소가 정상 조립됩니다.

## 4. 멤버별 — 새 코드 리뷰

지난 리뷰 이후 새로 생긴 페이지·API를 빠짐없이 봤습니다. 네 사람이 도메인을 하나씩 끝까지 끌고 가는 분담이 코드에 잘 드러납니다.

### 4-1. 이벤트 상세·주문 — 강재훈

상세 화면이 이제 `/api/events/${eventId}`·`/slots`·`/reviews`를 각각 `useEffect`로 가져오도록 연결됐습니다.
세 fetch 모두 `ignore` 플래그로 cleanup을 걸어 언마운트 후 setState 경합을 막은 점이 일관되게 좋습니다. 검색 무한스크롤 때 보였던 습관이 여기서도 이어졌네요.
주문 API(`/api/orders`)도 인증(`getUser` 401)·등급/회차 존재 확인·재고(`grade.quantity < quantity`)·회차 마감(`is_closed`) 검증에
수수료 계산(`FEE_RATE`)까지 갖춰져 있어 흐름이 탄탄합니다. E2E에서 비로그인 주문이 401로 정확히 막혔습니다(T27).

- [필수] 상세 API의 원문 에러 노출은 2-2에 정리했습니다.
- [제안] 상세 화면이 슬롯·리뷰는 API에서 받는데, **등급(grades)은 여전히 `MOCK_GRADES` 하드코딩**입니다(`page.tsx:421`의 `grades={MOCK_GRADES}`).
  그런데 예매 버튼은 실제 `selection.gradeId`를 주문 API로 보내거든요(`createOrder`). 실데이터가 들어오면 mock 등급 id(`grd_1`)로는
  주문이 `ticket grade not found`(404)로 떨어집니다. 등급을 가져오는 경로(`/slots`에 함께 싣거나 등급 API 추가)를 붙여 mock을 걷어내면,
  상세에서 주문까지 한 줄로 이어집니다. `/api/events/[eventId]/slots` 라우트가 아직 없는 점도 같이 챙기면 좋겠습니다.
- [사소] 상세 페이지 상단의 `loading` 초기값이 `false`라, 첫 진입에 "불러오는 중" 표시가 거의 안 뜹니다(곧장 mock 표시). 동작엔 문제없으나
  실데이터를 붙일 때 초기 `true`로 두면 깜빡임을 줄일 수 있습니다.

### 4-2. 회원가입·로그인 — 이선우

로그인이 서버 액션(`login/action.ts`)으로 정리됐습니다. `signInWithOAuth`(google/kakao)·`signInWithEmail` 모두 서버에서 처리하고
콜백 주소를 `NEXT_PUBLIC_SITE_URL`로 조립하는 흐름이 깔끔합니다. 미들웨어 인증 가드(`proxy.ts`)를 설치해 `/seller`를 닫고,
이미 로그인한 사용자가 `/login`·`/join`에 가면 `/`로 돌려보내는 `AUTH_ROUTES` 처리까지 넣은 게 좋습니다(E2E T20~T22로 가드 동작 확인).

- [제안·이월] `api/auth/join/route.ts:57`의 `fail(error.message)`는 6차부터 이어진 항목입니다. Supabase 원문(계정 존재 여부·영문 메시지)이
  그대로 나가지 않게 우리 코드로 매핑하는 정리를 2-2의 이벤트 상세 API와 함께 하면 한 번에 끝납니다.
- [사소] `login/page.tsx`의 "아이디 찾기"(`/find-id`)·"비밀번호 찾기"(`/find-password`) 링크는 아직 라우트가 없습니다. 시연에서 누군가 누르면 404가 나니,
  당장 만들 게 아니라면 링크를 잠시 비활성화하거나 "준비 중" 안내로 두는 편이 안전합니다.

### 4-3. 마이페이지 — 김연수

마이페이지가 프로필·설정·친구·라이브러리·예매내역까지 넓게 채워졌습니다. 설정 화면은 이용약관·개인정보처리방침을 모달로 연결하고,
회원 탈퇴를 "의도 확인 → 비밀번호 재인증" 2단계로, 비밀번호 변경도 모달로 분리한 구성이 꼼꼼합니다. 실수 탈퇴를 막는 장치라 방향이 좋습니다(E2E T24).

- [제안] `WithdrawButton`·`PasswordChangeButton`의 제출 핸들러가 아직 `// TODO: ... server action`으로 비어 있습니다.
  화면은 완성됐으니, 시연에서 "탈퇴/변경 버튼을 눌렀을 때 아무 일도 안 일어난다"가 어색하게 보일 수 있습니다. 실제 서버 액션 연결 전까지는
  토스트로 "준비 중입니다" 정도라도 피드백을 주면, 시연 중 동작이 끊겨 보이지 않습니다.
- [제안] 친구 목록·공유 티켓이 컴포넌트 상단 더미 배열입니다(`friends`, `sharedTickets`). 주석에 "나중에 Supabase 조회로 교체"라 적어둔 건 좋습니다.
  이 영역은 2-1 가드가 붙어야 진짜 "내" 데이터를 안전하게 보여줄 수 있으니, 가드를 먼저 얹는 걸 권합니다.

### 4-4. 판매자 이벤트 관리 — 방효진

이번 회차에서 가장 손이 많이 간 영역입니다. 이벤트 등록 폼(`EventCreateForm`)이 회차 생성 로직(기간×시간 조합, 금지일 제외,
회차 시간 겹침 검사)까지 직접 처리하고, 이미지를 Supabase Storage에 올려 URL을 받아 `/api/seller/event`로 보내는 흐름이 완성도 높습니다.
이벤트 목록·상세·수정 화면도 서버 컴포넌트에서 데이터를 모아 통계(주문 수·매출·잔여석)를 계산해 보여줍니다.

특히 좋은 점은 **모든 판매자 쿼리가 `seller_id`로 소유권을 좁힌다**는 것입니다. 목록(`list/page.tsx:14`), 상세(`events/[id]/page.tsx:22`),
수정 API(`event/[id]/route.ts:41`의 `.eq("seller_id", user.id)`) 모두 남의 이벤트를 건드릴 수 없게 막혀 있습니다. 권한 처리를 빠뜨리지 않은 게 인상적입니다.

- [제안] 서버 컴포넌트들이 `user!.id`로 비로그인 가능성을 단언하고 넘어갑니다(`list/page.tsx:14`, `events/[id]/page.tsx:22`).
  지금은 미들웨어가 `/seller`를 막아 주니 실제로 user가 없을 일은 없지만, 미들웨어 matcher가 바뀌거나 예외가 생기면 `user!.id`에서 런타임 에러가 납니다.
  `lib/auth.ts`에 이미 만들어 둔 `requireUser()`(없으면 `redirect('/login')`)를 페이지 맨 앞에서 부르면, 단언 없이도 안전하게 user를 확보할 수 있습니다.
  좋은 헬퍼를 만들어 두고 정작 페이지에선 안 쓰는 상태라 살짝 아깝습니다.
- [제안·이월] 이벤트 등록 API(`/api/seller/event`)는 `getUser` 인증은 하지만 "이 사용자가 정말 판매자(seller) 역할인지"는 확인하지 않습니다.
  일반 구매자 계정으로도 이벤트가 생성될 수 있어서, 가입 시 저장한 역할을 한 번 확인하면 더 안전합니다. 스토어 정보 API의 형식 검증(사업자번호·계좌)도 이월 상태입니다.

- **분담이 잘 굴러갑니다.** 이벤트·주문(강재훈), 인증·로그인(이선우), 마이페이지(김연수), 판매자(방효진)로 네 사람이 각자 도메인을 백엔드까지 끌고 갔습니다.
  팀장 공백이 있었던 걸 생각하면 팀이 제 속도를 잘 찾았다고 봅니다. 화면 단계 → 백엔드 연결 단계로 넘어온 진전이 또렷합니다.
- **인증 가드 패턴이 한 군데로 모이는 중입니다.** 8차엔 home은 서버 가드, mypage는 무방비, seller는 어중간한 상태로 제각각이었는데,
  이번에 미들웨어로 `/seller`를 묶으면서 방향이 잡혔습니다. 여기에 `/mypage`만 합류시키면 "보호가 필요한 경로는 미들웨어 한 곳에서 관리한다"는 규칙이 완성됩니다.
- **에러 응답 정책을 팀에서 한 번 합의하면 좋겠습니다.** 회원가입·이벤트 상세 두 곳에서 DB 원문이 새는 같은 패턴이 반복됩니다.
  "외부에는 우리 코드와 상태코드만, 원문은 서버 로그에"를 공통 규칙으로 정해 `api-response.ts` 근처에 에러 매핑 한 곳을 두면, 앞으로 API가 늘어도 일관되게 갑니다.
- 커밋·PR 흐름은 계속 건강합니다. 이벤트 상세가 mock으로 화면을 먼저 완성하고 API를 나중에 붙이는 식이라면, PR 설명이나 주석에 "이 부분은 아직 mock/연결 예정"을 적어두면
  리뷰하는 사람도 의도를 바로 압니다.
