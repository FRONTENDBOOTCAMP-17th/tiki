## 3. delta 주요 발견

### [필수] 비-UUID `/[eventId]` 진입 시 500 (16차 이월)

`getEventDetail`(`src/lib/event/queries.ts:55`)에는 `UUID_RE.test`로 UUID 가드가 들어가 잘못된 값이면 null을 돌려주도록 되어 있습니다. 여기까지는 좋습니다. 문제는 같은 상세 페이지에서 `Promise.all`로 **함께 호출되는 형제 쿼리들** — `getSlots`(115행), `getGrades`(175행), `getReviews`(194행) — 에는 같은 가드가 없다는 점입니다. 잘못된 UUID로 `.eq("event_id", ...)`를 던지면 Postgres가 `22P02`(invalid input syntax for uuid)를 뱉고, `if (error) throw error`에서 그대로 던져지면서 `Promise.all` 전체가 reject → 페이지 500이 됩니다.

대비되는 지점이 하나 있습니다. `/payment/[orderId]`는 `page.tsx:19`에서 `notFound()` 가드가 제대로 들어가 있거든요. 상세 페이지도 이 패턴을 그대로 가져오면 됩니다. 해결 방법은 가이드에 코드로 적어 뒀습니다.

### [제안] 정책/안내 페이지가 두 트리로 갈라져 있음

이번에 방효진님이 만든 `/info/privacy`·`/info/terms`·`/info/about`… 트리와, 앞서 김연수님이 SEO 정비 때 만든 루트 `/privacy`·`/terms`·`/support`가 지금 동시에 살아 있습니다. 그런데 **Footer는 `/info/*`로만 링크**(`Footer.tsx`)하는 반면 **sitemap에는 루트 `/terms`·`/privacy`·`/support`만 등재**돼 있어요. 그러다 보니 사용자가 실제로 보는 `/info/*`는 색인에서 빠지고, 정작 색인에 올라간 루트 페이지는 화면 어디서도 도달할 수 없는 상태가 됐습니다. 콘텐츠도 두 벌로 관리해야 하니 마감 전에 한쪽으로 합치는 게 좋겠습니다. 어느 쪽을 정본으로 할지 팀에서 먼저 합의하면 정리가 빠릅니다.

### [제안] sitemap·robots의 슬래시 중복

`metadataBase`가 `/`로 끝나는데 경로도 `/`로 시작하다 보니 `...vercel.app//support`, robots의 Sitemap 줄도 `...vercel.app//sitemap.xml`처럼 슬래시가 겹칩니다. 동작에 치명적이진 않지만 크롤러가 중복 URL로 볼 수 있으니, `baseUrl`에서 끝 슬래시를 떼거나 URL 결합을 `new URL()`로 처리하면 깔끔합니다.

### [제안] generateMetadata 부재 — 상세 페이지 SEO 약함

인프라(sitemap·robots·metadataBase)는 갖췄는데, 이벤트 상세가 전부 기본 타이틀을 공유하고 있습니다. 상세 페이지에 `generateMetadata`로 이벤트명·설명을 넣어 주면 검색 결과에서 각 공연이 자기 이름으로 노출됩니다.

### [제안] 정산 신청 중복 제출 여지

`seller/settlement/actions.ts`의 `requestSettlement`가 covered 집합을 확인한 뒤 insert하는데, 정작 `settlement_request` 테이블에는 unique 제약이 보이지 않습니다. 버튼 연타나 동시 요청 때 같은 기간에 대해 중복 행이 생길 수 있어요. 16차에 리뷰 삭제 연타를 막은 것과 같은 방식(버튼 비활성 + DB 유니크)이면 안전합니다.

### [사소] 그 외

- `/info/terms` 본문 조항 번호 밀림(제3조 중복).
- `SettlementCharts`의 축·그리드 색이 라이트 고정이라 다크에서 대비가 약합니다(주석에 의도라고 적혀 있긴 합니다).
- anon 키로 `review`·`order_seat`를 읽어 보니 `user_id`·`order_id` 같은 식별자가 함께 노출됩니다. 민감 테이블(users/orders/seller_profiles)은 전부 anon 차단이 잘 돼 있어 직접 악용은 어렵지만, 공개 조회에 굳이 필요 없는 식별자는 뷰에서 빼는 걸 권합니다.

### 칭찬하고 싶은 부분

- 다크모드를 요소마다 `dark:`를 다는 대신 `globals.css`에서 `.dark :where(...)`로 전역 리맵한 접근이 영리합니다. 손이 덜 가고 누락도 줄어듭니다.
- 검색 페이지의 Suspense 스트리밍, 정산의 `.in(eventIds)` 배치 조회(N+1 회피), 정산 신청/승인 액션의 인가(`requireUser`+소유자 검증, `requireAdmin`+상태 재검증)가 견고합니다.
- 좌석 수 상한을 RPC 레벨에서 막은 건 방어를 한 겹 더 쌓은 좋은 선택입니다.

## 4. 조원별

### 김연수 (harilog)
SEO를 세 회차 만에 종결지었습니다. `layout.tsx` 메타데이터, `sitemap.ts`·`robots.ts`, CI lint 단계 추가까지 한 흐름으로 정리했고, 루트 정책 페이지도 metadata를 갖춰 만들었습니다. 리뷰 연타 방지도 반영됐네요. 남은 숙제라면 sitemap/robots 슬래시 중복과 `generateMetadata` 도입, 그리고 방효진님의 `/info/*`와 정책 페이지가 겹치는 걸 팀에서 어느 쪽으로 합칠지 정하는 일입니다.

### 방효진 (lllillly)
앱 전역 다크모드가 이번 delta의 얼굴입니다. 전역 리맵 전략이 특히 좋았고, 헤더 토글 배치부터 푸터와 `/info/*` 안내 트리, 검색 페이지 Suspense화까지 폭넓게 손댔습니다. 개선점은 `/info/*`가 정책 페이지와 내용이 겹치면서 푸터/사이트맵 링크가 어긋난 점, 그리고 정산 차트의 다크 대비입니다.

### 강재훈 (bbalabuya, 리더)
판매자 매출 그래프(recharts)와 검색 최적화가 눈에 띕니다. 상세 진입에 `getEventDetail` UUID 가드를 넣은 것도 강재훈님 작업으로 보이는데, 이 가드를 형제 쿼리(`getSlots`/`getGrades`/`getReviews`)까지 확장하면 이번 [필수]가 바로 닫힙니다. recharts는 새로 들어온 의존성이라, 배포/CI에서 `npm ci` 후 설치가 보장되는지만 한 번 확인해 두면 좋겠습니다.

### 이선우 (zzz664)
보안·안정성 감각이 가장 도드라집니다. 결제 취소 경로 await+catch, 좌석 상한 RPC 방어, 주문 수량 상한 + 상수 일원화, 회원가입 에러 한글 매핑까지 마감에 꼭 필요한 것들을 잘 챙겼습니다. 다만 상한 8이 SQL과 TS 양쪽에 하드코딩돼 있어서 한쪽만 바꾸면 조용히 어긋납니다. 상수 출처를 하나로 두는 방법은 가이드에 적어 뒀습니다.

## 5. 팀 공통

- **비-UUID 상세 500**이 이번 팀 차원의 [필수]입니다. `/payment/[orderId]`가 쓰는 `notFound()` 패턴을 상세 페이지에도 그대로 옮기면 됩니다.
- 정책/안내 페이지 이원화는 두 분(김연수·방효진)의 작업이 만나는 지점이라, **정본을 하나 정하는 짧은 합의**가 먼저입니다. 그 뒤에 sitemap·Footer 링크를 그쪽으로 맞추면 SEO와 도달성이 한 번에 정리됩니다.
- CI에 lint가 들어간 건 좋은 변화입니다. 여유가 되면 `tsc --noEmit`도 게이트에 넣어 타입 깨짐이 머지 전에 걸리게 하면 마감 막판 사고를 줄일 수 있습니다.

정적 검증은 `npm install` 후 `tsc --noEmit` 0건, `eslint` 0 errors, `next build` 성공까지 확인했습니다(recharts가 새 의존성이라 pull 후 설치가 필요합니다 — 코드 결함은 아닙니다). 남은 나흘, 우선순위는 비-UUID 500 → 정책 페이지 정리 → 상세 generateMetadata 순으로 보면 충분합니다. 화이팅입니다.
