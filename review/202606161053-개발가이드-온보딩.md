# TiKi 개발 온보딩 가이드 (2026-06-16)

새로 합류한 분을 위한 안내입니다. 이 문서를 따라가면 우리 디자인 시스템 컴포넌트로 화면을 조립하고, 인증과 데이터를 어떻게 다루는지 이해해서 실제 페이지(예: 예매 흐름)를 만들 수 있습니다. 이미 있던 분들도 한 번 훑어두면 좋습니다. 모든 경로와 함수명은 지금 `develop` 브랜치 코드 기준입니다.

---

## 0. 프로젝트 한눈에

TiKi는 티켓 예약 오픈마켓입니다. 기술 스택은 Next.js 16(App Router, Turbopack) + React 19 + Tailwind v4 + cva, 백엔드는 Supabase입니다.

지금 단계는 두 축이 먼저 자리를 잡았습니다.

1. **디자인 시스템** — Button, Input, Modal, Toast, Sidebar, Toggle, Filter 등 공통 컴포넌트가 `src/components/`에 있고, 각각을 눈으로 확인하는 데모가 `src/app/example/*`에 있습니다.
2. **인증** — Supabase OAuth 로그인이 막 붙었습니다. 용도별 Supabase 클라이언트 헬퍼, 세션 갱신 미들웨어, 서버에서 로그인 가드를 거는 패턴이 준비돼 있습니다.

실데이터 페이지는 아직 적습니다. 그래서 새 페이지를 만드는 게 지금 가장 흔한 작업이고, 이 가이드의 핵심입니다.

### 폴더 구조

```
src/
  app/                 App Router. page.tsx 가 곧 라우트.
    layout.tsx         루트 레이아웃. ToastProvider 가 여기 들어 있음.
    page.tsx           "/" (지금은 임시 홈)
    home/page.tsx      "/home" — 서버에서 로그인 가드를 거는 모범 예시
    login/page.tsx     "/login" — OAuth 버튼 (providers 배열로 렌더)
    auth/callback/route.ts   OAuth 리다이렉트를 받는 콜백
    api/auth/join/route.ts   회원가입 API (입력 검증 + signUp)
    action.ts          서버 액션 (logout)
    example/*          공통 컴포넌트 데모 페이지
  components/          공통 컴포넌트 (디자인 시스템)
  hooks/useToast.ts    Toast Context 접근 훅
  lib/
    cn.ts              className 병합 유틸 (clsx + tailwind-merge)
    supabase/          용도별 클라이언트 (client/server/admin/middleware)
    api/api-response.ts  API 응답 헬퍼 (success / fail)
  types/               api·domain·database 타입
  proxy.ts             미들웨어 (모든 요청에서 세션 갱신)
```

`@/`는 `src/`를 가리킵니다(`tsconfig.json`의 `paths`). 그래서 `import { cn } from '@/lib/cn'`처럼 절대 경로로 가져옵니다. 상대경로(`../../lib/cn`)는 쓰지 않는 게 팀 관례입니다. 깊이가 바뀌어도 안 깨지니까요.

## 1. 새 페이지 만드는 법

### 라우트가 되는 조건 — 파일 이름이 `page.tsx` 여야 한다

App Router에서는 폴더가 경로이고, `page.tsx`가 그 경로의 화면입니다. 폴더만 있다고 라우트가 생기지 않고, 그 폴더 안에 `page.tsx`가 있어야 비로소 접근 가능한 주소가 됩니다.

지금 코드에 헷갈리기 쉬운 부분이 있어 미리 짚어둡니다. 아래 파일들은 라우트가 아닙니다.

```
src/app/category/category.tsx        ← "/category" 가 아님 (page.tsx 아님, 게다가 빈 파일)
src/app/pay/pay.tsx                  ← "/pay" 가 아님
src/app/search/search.tsx            ← "/search" 가 아님
src/app/mypage/mypage.tsx            ← "/mypage" 가 아님
src/app/seller/registration/registration.tsx  ← 라우트 아님
...
```

이것들은 화면 자리를 잡아둔 빈 스캐폴딩입니다. 폴더 이름과 같은 `.tsx`라서 라우트처럼 보이지만, Next는 `page.tsx`만 라우트로 인식합니다. 새로 `/category` 페이지를 만든다면 `src/app/category/page.tsx`를 만들어야 합니다. 기존 `category.tsx` 안의 내용을 컴포넌트로 빼서 `page.tsx`에서 불러 쓰거나, 그냥 `page.tsx`로 옮기면 됩니다.

표로 보면 이렇습니다.

| 파일 | 주소 |
|------|------|
| `src/app/page.tsx` | `/` |
| `src/app/home/page.tsx` | `/home` |
| `src/app/seller/registration/page.tsx` | `/seller/registration` |
| `src/app/category/category.tsx` | (없음 — 라우트 아님) |

`route.ts`는 화면이 아니라 API 엔드포인트입니다(예: `api/auth/join/route.ts` → `POST /api/auth/join`). 화면은 `page.tsx`, 데이터 엔드포인트는 `route.ts`로 구분합니다.

### "use client" vs 서버 컴포넌트 — 기본은 서버

App Router에서 컴포넌트는 **기본이 서버 컴포넌트**입니다. 파일 맨 위에 `'use client'`를 적은 것만 클라이언트 컴포넌트가 됩니다.

판단 기준은 간단합니다.

- 서버 컴포넌트로 둔다(기본): `useState`/`useEffect`/`onClick` 같은 게 필요 없는 화면, 그리고 로그인 확인이나 DB 읽기가 필요한 화면. 서버에서 직접 Supabase를 호출할 수 있어 빠르고 안전합니다. `home/page.tsx`가 이 경우입니다.
- `'use client'`를 붙인다: 버튼 클릭, 입력 상태, 모달 열고 닫기처럼 브라우저 상호작용이 있는 화면. `login/page.tsx`, `example/*`가 이 경우입니다.

실전에서는 서버 컴포넌트가 데이터를 읽어 와서 상호작용이 필요한 부분만 작은 클라이언트 컴포넌트로 떼어 넘기는 구조를 권장합니다. 화면 전체를 통째로 `'use client'`로 만들면 서버에서 데이터를 미리 채워 보내는 이점을 잃습니다.

## 2. 공통 컴포넌트로 화면 조립

### cn 과 cva — 이 두 가지만 알면 됩니다

- `cn(...)` (`src/lib/cn.ts`)은 클래스 문자열을 합치는 유틸입니다. 내부에서 `clsx`로 조건부 클래스를 정리하고 `tailwind-merge`로 충돌(예: `px-3`와 `px-4`)을 뒤엣것 우선으로 합쳐줍니다. 컴포넌트에 `className`을 덧붙여 쓸 때 이 함수를 거칩니다.
- `cva`(class-variance-authority)는 variant(종류)별 클래스를 한곳에 정의하는 도구입니다. 버튼의 색·크기 조합 같은 걸 깔끔하게 관리합니다.

`Button.styles.ts`를 보면 패턴이 보입니다.

```ts
export const buttonVariants = cva(
  ['inline-flex items-center justify-center ...'], // 공통 클래스
  {
    variants: {
      variant: { primary: '...', secondary: '...', danger: '...', outline: '...', /* 등 */ },
      size: { sm: '...', md: '...', lg: '...' },
      fullWidth: { true: 'w-full' },
    },
    defaultVariants: { variant: 'primary', size: 'md' },
  },
);
```

그리고 `Button.tsx`에서 `cn(buttonVariants({ variant, size, fullWidth }), className)`으로 합쳐 씁니다. 새 컴포넌트를 만들 때 이 분리(스타일은 `*.styles.ts`의 cva, 로직은 `*.tsx`)를 그대로 따라가면 팀 코드와 결이 맞습니다.

### Button

```tsx
import Button from '@/components/Button';

<Button variant="primary">예매하기</Button>
<Button variant="outlinePrimary" size="sm">취소</Button>
<Button variant="danger" fullWidth>삭제</Button>
<Button loading>저장 중</Button>   // loading 이면 스피너 + disabled 자동
```

variant 종류는 `primary` `secondary` `danger` `accent`와 outline 계열(`outline` `outlinePrimary` `outlineSecondary` `outlineDanger` `outlineAccent`), size는 `sm` `md` `lg`입니다. `loading`을 주면 스피너가 돌고 자동으로 비활성화되니, 별도로 `disabled`를 챙길 필요가 없습니다. 전체 목록은 `src/app/example/button/page.tsx`에서 눈으로 볼 수 있습니다.

### Input

```tsx
import { Input } from '@/components/Input';

<Input label="이메일" type="email" placeholder="you@example.com"
  helperText="로그인에 쓸 이메일" />

<Input label="비밀번호" type="password" error="8자 이상 입력하세요" />
```

`label`/`error`/`helperText`를 받습니다. `id`를 안 넘기면 `useId()`로 자동 생성해 `label`과 연결하고, `error`가 있으면 `aria-invalid`와 에러 메시지를 붙여줍니다. 접근성이 기본으로 챙겨져 있으니 그냥 props만 채우면 됩니다. `Input`은 **named export**라 중괄호로 가져오는 점만 주의하세요(`import { Input }`). Button은 default export(`import Button`)입니다.

### Modal — 합성(compound) 패턴

`Modal`은 `Modal.Header` / `Modal.Body` / `Modal.Footer`를 조립해 쓰는 합성 컴포넌트입니다. 바깥을 클릭하거나 Esc를 누르면 닫히고, 열려 있는 동안 배경 스크롤을 잠급니다.

```tsx
'use client';
import { useState } from 'react';
import Modal from '@/components/modal/Modal';
import Button from '@/components/Button';

export default function Example() {
  const [open, setOpen] = useState(false);

  return (
    <>
      <Button onClick={() => setOpen(true)}>예매하기</Button>

      <Modal open={open} onClose={() => setOpen(false)} position="sheet">
        <Modal.Header>좌석 선택</Modal.Header>
        <Modal.Body>
          {/* 내용: 날짜·좌석 목록 등 */}
        </Modal.Body>
        <Modal.Footer>
          <Button variant="outline" onClick={() => setOpen(false)}>닫기</Button>
          <Button onClick={() => setOpen(false)}>결제</Button>
        </Modal.Footer>
      </Modal>
    </>
  );
}
```

`position`은 `center`(기본)와 `sheet`(아래에서 올라오는 바텀시트) 두 가지입니다. 예매처럼 모바일에서 아래에서 올라오는 느낌이면 `sheet`를 씁니다. `open`/`onClose` 상태는 쓰는 쪽에서 `useState`로 들고 있어야 하니, Modal을 쓰는 페이지는 `'use client'`가 됩니다.

확인/취소 같은 단순 모달은 이미 만들어둔 `Dialog`(`src/components/modal/Dialog.tsx`)가 더 편합니다. 예매 바텀시트는 `OrderContent`라는 전용 컴포넌트도 있습니다. 실제 사용 예는 `src/app/example/modal/page.tsx`에 전부 모여 있으니 그걸 출발점으로 삼으면 좋습니다.

### Toast — Context + useToast

Toast는 전역 알림입니다. `ToastProvider`가 이미 `src/app/layout.tsx`에서 앱 전체를 감싸고 있어서, 어디서든 `useToast()`만 부르면 됩니다.

```tsx
'use client';
import useToast from '@/hooks/useToast';

export default function SomeButton() {
  const toast = useToast();
  return (
    <button onClick={() => toast.success('예매가 완료되었습니다')}>
      예매
    </button>
  );
}
```

`toast.success` / `toast.error` / `toast.warning` / `toast.info` 네 가지가 있고, 두 번째 인자로 표시 시간(ms)을 줄 수 있습니다(`toast.error('실패했습니다', 1000)`). 한 번에 보이는 개수와 위치는 Provider에서 관리하고, 넘치면 큐에 쌓였다가 순서대로 나옵니다. `useToast`는 Provider 바깥에서 부르면 에러를 던지므로 반드시 컴포넌트 트리 안(즉 `'use client'` 컴포넌트)에서 호출합니다. 데모는 `src/app/example/toast/page.tsx`입니다.

## 3. 인증

### 용도별 Supabase 클라이언트 — 어디서 어느 걸 쓰나

`src/lib/supabase/` 아래에 네 가지가 있습니다. **이걸 헷갈리면 인증이 조용히 깨지니** 표로 외워두는 걸 권합니다.

| 헬퍼 | 파일 | 어디서 쓰나 | 키 |
|------|------|------------|-----|
| `createClient()` (browser) | `client.ts` | `'use client'` 컴포넌트(브라우저) | anon |
| `createClient()` (server) | `server.ts` | 서버 컴포넌트 / 서버 액션 / Route Handler | anon |
| `createMiddlewareClient()` | `middleware.ts` | 미들웨어(`proxy.ts`)에서만 | anon |
| `supabaseAdmin` | `admin.ts` | 서버 전용, RLS 우회가 꼭 필요할 때만 | **service role** |

핵심 구분 기준은 쿠키를 누가 다루느냐입니다.

- 브라우저 client는 브라우저가 알아서 쿠키를 들고 다니므로 단순합니다.
- 서버 client는 `next/headers`의 쿠키를 읽습니다. 단 서버 컴포넌트는 쿠키를 쓸 수 없어서, `server.ts`의 `setAll`은 일부러 `try {} catch {}`로 감싸 조용히 넘어갑니다. 쿠키 갱신은 미들웨어가 대신합니다.
- 미들웨어 client는 쿠키를 쓸 수 있는 유일한 곳이라, 만료된 세션 토큰을 갱신해 브라우저로 돌려보냅니다.
- admin은 `service role` 키를 쓰므로 RLS(행 수준 보안)를 통째로 우회합니다. 절대 클라이언트로 내려가면 안 됩니다. 그래서 `admin.ts` 맨 위에 `import 'server-only'`가 붙어 있어, 실수로 클라이언트 번들에 섞이면 빌드가 막힙니다. 평소엔 쓸 일이 거의 없고, 정말 필요할 때만 서버 코드에서 씁니다.

### 보호 페이지 만드는 법 — 서버에서 getUser 가드

로그인한 사람만 볼 수 있는 페이지는 서버 컴포넌트에서 `getUser()`로 막는 게 가장 확실합니다. `home/page.tsx`가 그 모범입니다.

```tsx
import { redirect } from 'next/navigation';
import { createClient } from '@/lib/supabase/server';

export default async function Home() {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();

  if (!user) {
    redirect('/login');   // 비로그인이면 로그인 페이지로
  }

  // 여기부터는 user 가 보장됨. user.id 로 데이터 조회 가능.
  return <main>...{user.email}...</main>;
}
```

왜 `getSession()`이 아니라 `getUser()`인지가 중요합니다. `getUser()`는 Supabase 서버에 토큰을 검증받습니다. `getSession()`은 쿠키에 든 값을 그대로 믿어서, 위조된 쿠키로 우회될 수 있습니다. 서버에서 신뢰가 필요한 가드에는 항상 `getUser()`를 씁니다.

새 보호 페이지를 만들 때는 이 다섯 줄(`createClient` → `getUser` → `if (!user) redirect`)을 그대로 복사해 시작하면 됩니다.

### proxy(미들웨어) 세션 한 줄

`src/proxy.ts`는 모든 요청에서 `supabase.auth.getUser()`를 한 번 호출해 만료된 토큰을 갱신하고, 갱신된 쿠키가 실린 `response`를 반환합니다. 덕분에 한 시간짜리 토큰이 만료돼도 세션이 끊기지 않습니다. 새로 합류한 분이 여기를 건드릴 일은 거의 없지만, "세션이 자동으로 살아 있는 건 이 미들웨어 덕분"이라는 것만 알아두면 됩니다.

### 로그인 버튼 — providers 배열 패턴

`login/page.tsx`는 소셜 로그인 버튼을 하나하나 적지 않고 `providers` 배열을 `map`으로 돌려 렌더합니다.

```tsx
const providers = [
  { id: 'google', label: 'Google로 시작하기', color: '...' },
  { id: 'kakao',  label: '카카오로 시작하기', color: '...' },
] as const;

// ...
{providers.map((p) => (
  <button key={p.id} onClick={() => signIn(p.id)}>{p.label}</button>
))}
```

버튼을 추가·삭제할 때 JSX를 복붙하지 않고 배열만 고치면 되는, 좋은 패턴입니다. 클릭하면 `supabase.auth.signInWithOAuth`가 제공자로 보내고, 로그인이 끝나면 `/auth/callback`(`auth/callback/route.ts`)으로 돌아와 `exchangeCodeForSession`으로 세션을 만든 뒤 원래 가려던 곳으로 리다이렉트합니다.

## 4. 데이터 다루기

### 읽기 — 서버 컴포넌트에서

보호 페이지 가드 다음에 바로 이어서 읽으면 됩니다. DB 타입(`src/types/database.ts`)에 현재 `users`, `seller_profiles` 테이블이 정의돼 있습니다.

```tsx
import { createClient } from '@/lib/supabase/server';

export default async function MyPage() {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) redirect('/login');

  const { data: profile, error } = await supabase
    .from('users')
    .select('name, email, role')
    .eq('id', user.id)
    .single();

  if (error) {
    // 화면에서 에러 처리 (예: 안내 메시지)
  }

  return <div>{profile?.name}</div>;
}
```

`error`를 무시하지 말고 항상 확인하세요. Supabase는 예외를 던지지 않고 `{ data, error }`로 돌려주기 때문에, `error`를 안 보면 실패해도 그냥 빈 화면이 나옵니다.

### 쓰기 — Route Handler + 입력 검증 + ApiResponse

데이터를 만들거나 바꾸는 건 `api/.../route.ts`(Route Handler)나 서버 액션에서 합니다. 회원가입 API(`api/auth/join/route.ts`)가 우리 팀의 표준 형태입니다.

```ts
import { fail, success } from '@/lib/api/api-response';
import { createClient } from '@/lib/supabase/server';

export async function POST(req: NextRequest) {
  const body = await req.json();

  // 1) 입력 검증 — 빠진 값/형식 오류를 먼저 막는다
  if (!body.email) return fail('empty_email');
  // ... 비밀번호 정규식 검증 등

  // 2) 실제 작업
  const supabase = await createClient();
  const { error } = await supabase.auth.signUp({ /* ... */ });
  if (error) return fail(error.message);

  // 3) 일관된 형식으로 응답
  return success(null, 'verification_email_sent');
}
```

여기서 두 가지가 우리 컨벤션입니다.

- 응답 헬퍼 `success(data, message?, status?)` / `fail(message, status?)` (`src/lib/api/api-response.ts`). 모든 API가 `{ success, message, data }` 같은 모양으로 응답하도록 통일합니다. 프론트에서 받을 때 타입은 `ApiResponse<T>`(`src/types/api/common.ts`)로 받으면 됩니다.
- 입력 검증을 맨 앞에서 합니다. 작업을 시작하기 전에 빠진 값과 형식을 먼저 거릅니다. 클라이언트 검증은 사용자 편의일 뿐, 서버 검증이 진짜 방어선입니다. 누가 API를 직접 호출해도 막혀야 하니까요.

새 API를 만들 때 이 세 단계(검증 → 작업 → `success`/`fail`)와 타입(`types/api/`에 요청·응답 타입 추가)을 그대로 따라가면 됩니다.

### 한 가지 더 — RLS는 기본으로 켜둔다

Supabase 테이블은 RLS(행 수준 보안)를 켜고, "본인 데이터만 읽고 쓸 수 있다" 같은 정책을 거는 게 원칙입니다. anon 키는 누구나 가진 키라, RLS가 없으면 남의 데이터가 그대로 읽힙니다. 새 테이블을 만들 때는 RLS 정책도 같이 만든다고 생각해주세요. 정책으로 풀기 어려운 특수한 작업에서만 `supabaseAdmin`을 서버에서 씁니다.

## 5. 라우트 규칙·커밋 컨벤션·CI

### 라우트 규칙 (4번까지 나온 것 정리)

- 화면은 `page.tsx`, API는 `route.ts`. 그 외 이름의 `.tsx`는 라우트가 아니다(지금 비어 있는 `category.tsx` 등은 자리만 잡아둔 것).
- 기본은 서버 컴포넌트, 상호작용이 필요한 곳만 `'use client'`.
- import는 `@/`(=`src/`) 절대경로.
- 컴포넌트는 default export(Button)와 named export(Input)가 섞여 있으니, 가져올 때 데모 페이지를 보고 맞춥니다.

### 커밋 컨벤션

최근 커밋 로그를 보면 형식이 일정합니다. `타입(범위): 설명` 형태입니다.

```
feat(auth): 로그아웃 서버액션 추가
fix(toast): Toast.styles info 대괄호 2개 -> 1개 수정
fix(proxy): 세션 갱신 코드 수정
feat(supabase): 용도별 supabase client 생성헬퍼 추가
```

타입은 `feat`(기능 추가) / `fix`(버그 수정) 등을 쓰고, 범위(괄호 안)에 어디를 건드렸는지 적습니다. 새 작업도 이 형식을 맞춰주세요. 한눈에 "무엇을 왜 바꿨는지" 읽히는 게 목적입니다.

작업은 `develop`에 직접 푸시하지 않고 **브랜치 → PR**로 들어갑니다(`develop`은 보호 브랜치). 브랜치 이름도 `이름/feat/...`, `이름/fix/...` 형태로 쓰고 있습니다.

### CI — 머지 전 빌드 통과 문화

`.github/workflows/ci.yml`이 `develop`으로 가는 PR마다 자동으로 돕니다. Ubuntu(리눅스)에서 `npx tsc --noEmit`(타입 검사)와 `npm run build`(빌드)를 돌립니다. 리눅스를 쓰는 이유가 있습니다. 맥은 파일명 대소문자를 구분하지 않아서, 로컬에선 멀쩡하던 `import './Button'` vs `import './button'` 같은 사고가 배포 서버(리눅스)에서만 터집니다. CI를 리눅스로 돌려 이런 걸 머지 전에 잡습니다.

그래서 PR을 올리기 전에 로컬에서 미리 한 번 돌려보는 습관을 권합니다.

```bash
npx tsc --noEmit
npm run build
```

여기서 빨간불이 나면 CI에서도 막힙니다. 미리 확인하면 PR을 다시 손볼 일이 줄어듭니다. "CI 통과 = 일단 안전"이라는 합의가 있으니, 통과를 깨는 PR은 머지하지 않습니다.

## 6. 새 페이지 추가 체크리스트

예매 페이지 하나를 만든다고 할 때 순서입니다.

1. **경로 폴더에 `page.tsx`를 만든다.** 예: `src/app/booking/page.tsx`. (폴더만 만들거나 `booking.tsx`로 만들면 라우트가 안 생긴다.)
2. **서버/클라이언트를 정한다.** 로그인 확인·데이터 읽기가 필요하면 서버 컴포넌트(기본), 클릭·상태·모달이 있으면 그 부분만 `'use client'` 컴포넌트로 분리.
3. **보호가 필요하면 `getUser()` 가드를 건다.** `home/page.tsx`의 다섯 줄을 복사해 시작.
4. **데이터는 서버에서 `supabase.from(...).select(...)`로 읽고 `error`를 확인한다.** 쓰기는 `route.ts` + `success`/`fail` 헬퍼.
5. **화면은 공통 컴포넌트로 조립한다.** Button/Input/Modal/Toast. 사용법이 막히면 `src/app/example/*`에서 같은 컴포넌트의 실제 예시를 본다.
6. **타입을 맞춘다.** 새 API면 요청·응답 타입을 `src/types/api/`에, 응답은 `ApiResponse<T>`로.
7. **올리기 전 `npx tsc --noEmit && npm run build`로 확인.** 통과하면 `타입(범위): 설명` 형식으로 커밋하고 브랜치 → PR.

막히면 비슷한 일을 이미 한 파일을 찾아 그대로 따라가는 게 가장 빠릅니다. 인증은 `home/page.tsx`와 `login/page.tsx`, 컴포넌트는 `example/*`, API는 `api/auth/join/route.ts`가 좋은 출발점입니다. 천천히 그렇지만 꾸준히 갑시다. 화이팅입니다.
