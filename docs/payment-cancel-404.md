# 결제 취소 시 404 노출 해결

## 문제 상황

결제창에서 결제를 취소한 뒤 다시 TiKi 결제 페이지로 돌아오면 404 페이지가 보이는 경우가 있었다.
사용자 입장에서는 결제가 취소된 것인지, 예매가 남아있는 것인지 알기 어려운 흐름이었다.

## 원인

`/payment/[orderId]` 페이지는 주문 상태가 `ordered`가 아니면 모두 `notFound()`로 처리하고 있었다.

결제 취소 또는 실패가 발생하면 주문 상태는 `cancelled` 또는 `failed`가 될 수 있다. 이 상태로 다시
`/payment/[orderId]`에 접근하면, 실제로는 내 주문이고 정상적인 취소/실패 결과인데도 결제 페이지가
404로 보내고 있었다.

또한 포트원 리디렉션 취소 응답은 상황에 따라 `paymentId` 없이 `code`만 붙어서 돌아올 수 있는데,
기존 코드는 `paymentId`가 없으면 취소 처리를 시작하지 않았다.

## 해결 방법

- 잘못된 주문 ID, 존재하지 않는 주문, 다른 사용자의 주문은 기존처럼 404로 처리한다.
- 내 주문이 `cancelled` 또는 `failed` 상태라면 404 대신 안내 화면을 보여준다.
- 결제 취소/실패 응답에 `code`가 있으면 `paymentId` 유무와 관계없이 예약 취소 처리를 시도한다.
- 취소 처리 후 `/payment/[orderId]?result=cancelled`로 이동시켜 서버가 최신 주문 상태를 다시 읽게 한다.

## 왜 결제 직전 화면으로 돌려보내지 않았을까요?

TiKi는 티켓/좌석 재고가 있는 예약 서비스라서, 결제 전에 먼저 `orders`를 만들고 해당 수량 또는 좌석을 잡아둔다.
결제창에서 취소하면 이 주문은 `cancelled` 상태로 바뀌고, 잡아두었던 재고나 좌석도 다시 풀린다.

이 상태에서 사용자를 다시 결제 직전 화면으로 돌려보내면, 화면상으로는 다시 결제할 수 있을 것처럼 보이지만
실제로는 이미 취소되어 재고가 풀린 주문이다. 사용자가 같은 `orderId`로 다시 결제를 누르면 상태와 재고가 맞지 않는
혼란이 생길 수 있다.

그래서 취소된 주문을 다시 결제 화면으로 보여주기보다는, "결제가 취소되었습니다" 안내를 보여주고
필요하면 공연 상세에서 다시 예매하도록 처리했다. 쇼핑몰처럼 장바구니를 유지하는 서비스라면 결제 직전 화면으로
돌아가는 방식도 괜찮지만, TiKi처럼 좌석/수량을 임시로 선점하는 구조에서는 취소 후 재예매 흐름이 더 안전하다.

## 확인 방법

1. 결제 페이지에서 결제창을 연다.
2. 결제창에서 취소를 누른다.
3. 404가 뜨지 않고 결제 취소 안내 화면이 보이는지 확인한다.
4. 예매내역에 취소된 예매가 잘 정리되는지 확인한다.
5. 정상 결제 완료 흐름은 기존처럼 예매내역으로 이동하는지 확인한다.
