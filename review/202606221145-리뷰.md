# TiKi 10차 코드 리뷰 (2026-06-22) — 시연 대비 집중 점검

오늘 오후 시연을 앞두고, 9차(2026-06-19) 이후 들어온 변경 가운데 **시연 동선에 직결되는 것 위주로** 빠르게 봤습니다. 이번엔 양도 많고 무게도 큽니다. 오늘 새벽 작업한 결제(PortOne) 연동이 새로 들어왔고, 회차·등급 API가 연결되면서 예매가 주문까지 이어지게 됐고, 마이페이지와 카테고리도 채워졌네요. 앱을 먼저 띄워 사용자 흐름을 돌려 본 뒤(아래 "E2E 결과") 그 동작을 근거로 코드를 봤습니다.

시연 기준으로 추리면 **막아야 할 [필수]가 두 가지**입니다. 9차에서 이어진 마이페이지 가드, 그리고 이번에 새로 확인한 홈 페이지의 환경변수 강결합인데, 둘 다 고칠 손은 작습니다.

## 2. 시연 전 막아야 할 [필수] 두 가지

### 2-1. 마이페이지 로그인 가드 (9차 이월·미해결) [필수]

`/seller`는 미들웨어(`proxy.ts`)로 완전히 막혀 있습니다(E2E T20~T22 — `/seller/*` 다섯 경로 모두 `/login`으로 리다이렉트). 가드 구조 자체는 잘 갖춰져 있다는 뜻인데, 그 보호 목록에 마이페이지만 빠져 있습니다.

```ts
// src/lib/auth/routes.ts
export const PROTECTED_ROUTES = ["/seller", "/api/seller"];
```

그래서 비로그인 상태로 `/mypage`에 들어가면 프로필·친구·라이브러리·예매내역이 그대로 보입니다(T11). 여기에 **`'/mypage'` 한 줄만 더하면** 마이페이지 전체가 판매자와 똑같은 방식으로 닫힙니다. 9차에 드린 안내 그대로라 가이드 1번에 적어 두었습니다.

### 2-2. 홈(`/`)이 `SUPABASE_SERVICE_ROLE_KEY`에 강결합 — 키 없으면 사이트 루트 전체 500 [필수·신규]

홈 페이지가 "전체 예매 랭킹"을 집계하려고 service-role 클라이언트(`supabaseAdmin`)를 직접 가져다 씁니다.

```tsx
// src/app/page.tsx
import { supabaseAdmin } from "@/lib/supabase/admin";   // 공개 홈페이지가 service-role 을 import
```

```ts
// src/lib/supabase/admin.ts — 모듈이 로드되는 순간 키를 읽음
export const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,   // 없으면 여기서 throw
);
```

`createClient`는 두 번째 인자(키)가 비어 있으면 모듈 로드 시점에 `Error: supabaseKey is required`를 던집니다. 그런데 이 모듈을 **공개 홈페이지(`/`)가 import**하다 보니, `SUPABASE_SERVICE_ROLE_KEY`가 없는 환경에서는 홈뿐 아니라 사이트 루트 전체가 500으로 떨어집니다. 리뷰 환경에서 이 500을 실제로 봤습니다.

```
⨯ Error: supabaseKey is required.
    at module evaluation (src/lib/supabase/admin.ts:4)
    at module evaluation (src/app/page.tsx:6)
```

솔직히 말씀드리면 이건 제 리뷰 env에 service-role 키가 없어서 난 것이라, **배포 환경에 그 키가 들어가 있다면 정상 동작합니다.** 다만 뒤집어 보면 "그 키가 빠지는 순간 홈이 통째로 죽는다"는 뜻이기도 합니다. 오늘 `chore(ci.yml): 시크릿 키 추가` 커밋이 올라온 걸 보면 비밀 키 설정을 한창 맞추는 중인 듯한데, 배포든 로컬이든 **시연 환경에 `SUPABASE_SERVICE_ROLE_KEY`가 확실히 들어가 있는지** 시연 전에 꼭 확인해 주세요. 이번 시연의 가장 큰 리스크라 생각합니다.

랭킹 집계에 service-role을 쓴 의도(`orders`는 본인 주문만 보이는 RLS라 전체 랭킹은 service-role로)는 주석에 잘 설명돼 있고 납득됩니다. 다만 공개 페이지가 service-role 키에 목을 매는 구조는 위태로우니, 시연 이후엔 집계를 DB 쪽(SECURITY DEFINER 함수나 집계 뷰)으로 옮겨 anon 키로도 읽히게 바꾸는 편을 권합니다. 그러면 공개 홈에서 service-role을 import할 일이 사라집니다(가이드 2번).

## 3. 잘된 부분 — 결제 승인 검증 [칭찬]

이번에 들어온 결제 승인 API(`/api/payments/confirm`, 이선우)가 결제에서 제일 위험한 함정을 정확히 피했습니다. 결제창에서 돌아온 클라이언트 응답을 그대로 믿지 않고 **포트원 서버에 직접 조회해 상태와 금액을 다시 검증**합니다.

```ts
const isPaid =
  payment.status === "PAID" && payment.amount.total === order.total_price;
```

- 로그인 확인(`getUser` → 401), 본인 주문 확인(`.eq("user_id", user.id)`)을 먼저 하고,
- 클라가 보낸 금액이 아니라 **DB의 `order.total_price`와 포트원이 알려준 실제 결제액을 맞춰 봅니다.** 결제 금액 위변조는 초보 단계에서 가장 자주 뚫리는 곳인데 여기서 막았습니다.
- 이미 `paid`면 재조회 없이 끝내는 멱등 처리, 검증 실패 시 `failed`로 기록하는 흐름까지 깔끔합니다.

E2E T28에서 비로그인 `/api/payments/confirm` 호출이 **401**로 정확히 막히는 것도 확인했습니다. 결제처럼 무거운 도메인을 이 정도 완성도로 가져온 건 칭찬할 만하다 생각합니다.

## 6. 시연까지 우선순위

1. **[필수] `SUPABASE_SERVICE_ROLE_KEY`가 시연 환경에 들어가 있는지 확인** — 없으면 홈이 통째로 500(2-2). **시연 전 최우선.**
2. **[필수] 마이페이지 가드** — `PROTECTED_ROUTES`에 `'/mypage'` 한 줄 추가(2-1·가이드 1번).
3. **[제안] 홈 랭킹 집계를 RPC/뷰로** — 공개 페이지의 service-role 의존 제거(2-2·가이드 2번). 시연 이후.
4. **[제안·이월] join API 원문 에러 노출, find-id/find-password 404 링크, CI push 트리거+lint.**

결제까지 들어오면서 "화면만 있던" 단계를 확실히 넘어섰습니다. 특히 결제 금액 검증을 정석대로 막은 건 이번 회차의 가장 큰 성과라 생각합니다. 남은 건 마이페이지 가드 한 줄, 그리고 홈이 의존하는 시크릿 키가 시연 환경에 실려 있는지 확인하는 정도라, 그 둘만 챙기면 시연은 무리 없이 보여줄 수 있는 상태라 봅니다. 화이팅입니다.
