# TiKi 7차 코드 리뷰 (2026-06-17)

6차 이후 develop에 들어온 검색 도메인 작업을 봤습니다. 이번 회차에 크게 바뀐 건 검색 페이지(`/search`), 검색 API(`/api/events/search`), 무한스크롤, 그리고 미들웨어/프록시 supabase 클라이언트 분리입니다. 오늘은 **응집도·결합도**, 그리고 **React·Next·TypeScript를 제대로 쓰고 있는지**에 초점을 맞췄습니다. 새 라우트가 생긴 만큼 E2E도 그쪽을 집중적으로 돌렸습니다.

좋은 소식부터 말하면 **빌드는 깨끗합니다**(`npx tsc --noEmit` 0건, eslint도 0건). 새 검색 코드까지 타입 정합성이 흐트러지지 않았습니다. 다만 E2E를 돌려보니 **검색 페이지가 실제로는 500으로 죽습니다.** 타입은 맞는데 런타임에서 터지는 경우라 2장에서 자세히 보겠습니다.

이번 회차는 6차에 남긴 [제안]들을 직접 손대기보다 **검색 도메인 신규 기능에 집중**한 쪽이었습니다. 6차에서 zzz664가 닫은 proxy/Toast [필수]는 회귀 없이 그대로 유지되고 있습니다.

| 6차 항목 | 등급 | 상태 | 비고 |
|---|---|---|---|
| 회원가입 `signUp` 에러 자체 코드로 래핑 | [제안] | 미해결 | 이번 범위에 인증 영역 변경 없음 |
| CI `push` 트리거 + `npm run lint` | [제안] | 미해결 | ci.yml 변경 없음(새 검색 코드는 lint 통과 상태) |

6차 [제안]을 미룬 건 새 기능에 집중하느라 그런 것으로 보입니다. 검색이 일단락되면 CI lint나 회원가입 에러 래핑처럼 작은 것부터 차례로 닫아가면 좋겠습니다.

## 2. [필수] 검색 페이지(`/search`)가 500으로 죽습니다 — E2E 실측

새 라우트라 제일 먼저 띄워봤는데 데스크톱·모바일 둘 다 **HTTP 500**이 납니다.

```
Invalid src prop (https://picsum.photos/seed/1/200/300) on `next/image`,
hostname "picsum.photos" is not configured under images in your `next.config.js`
  at EventCard.tsx:35  ← next/image
  at search/page.tsx:228 (popularEvents.map)
```

원인을 따라가 보면 이렇습니다.
- `search/page.tsx:235`의 인기공연 더미 데이터가 이미지로 `https://picsum.photos/seed/${event.id}/200/300`을 씁니다.
- 그런데 `next.config.ts`에 `images.remotePatterns` 설정이 아예 없습니다.
- `next/image`는 허용되지 않은 외부 호스트를 만나면 에러를 던지는데, 그게 서버 렌더 단계에서 터지면서 **페이지 전체가 500**이 됩니다.

한 가지 짚어둘 건 **검색 API(`/api/events/search`)는 정상**이라는 점입니다(E2E T7: `200`, `{"success":true,...,"items":[],"total":0}`). 데이터가 없어 빈 결과가 오긴 하지만 API 자체는 잘 동작합니다. 검색 로직이 아니라 **인기공연 더미 이미지 한 줄** 때문에 페이지가 통째로 안 뜨는 상황입니다.

- 캡처: `review/images/2026-06-17/tiki-T6-search-desktop.png` (500 에러 오버레이)

새 라우트가 사용자에게 전혀 안 보이는 상태라 `[필수]`로 둡니다. 고치는 법은 간단합니다(가이드 1번). `next.config.ts`에 picsum 호스트를 등록하거나, 더미 이미지를 등록된 호스트나 일반 `<img>`로 바꾸면 됩니다.

> 참고: 리뷰 환경의 `.env.local`에 `SUPABASE_URL`이 `https://` 스킴 없이 들어 있어 처음엔 dev 서버가 기동조차 못 했습니다. 제 리뷰용 env 파일 문제라 보정하고 진행했습니다(팀 코드 이슈는 아닙니다). 팀에서도 `.env.local`의 URL이 `https://`로 시작하는지 한 번 확인해 두면 좋겠습니다.

## 3. 멤버별 — 응집도·결합도 / React·Next·TS

### 강재훈 (bbalabuya) — 검색 페이지 + 검색 API

이번 회차의 핵심 작업입니다. 설계 감각이 좋아진 게 여러 군데서 보입니다.

**잘한 점**
- **단일 출처(SSOT)를 일관되게 씁니다.** `filterSort.ts:25-37`의 `SORT_OPTIONS`에서 `SortKey` 타입을 `(typeof SORT_OPTIONS)[number]["key"]`로 자동 도출했습니다(`:40`). 정렬 옵션을 한 곳에서 늘리면 타입·버튼·비교함수가 같이 따라옵니다. TS를 제대로 쓰는 패턴입니다.
- **무한스크롤의 경합 방어가 견고합니다.** ① `loadingRef`로 중복 호출을 막고(`page.tsx:116`), ② 디바운스 effect에 `ignore` 플래그와 cleanup을 둬 stale 응답을 버리고(`:84,108-111`), ③ `loadedKey === requestKey` 비교로 "지금 요청과 결과가 같을 때만" `hasMore`를 계산합니다(`:135-136`). 검색어를 빠르게 바꿀 때 옛 결과가 새 결과에 섞이는 전형적인 버그를 미리 막아뒀습니다. 정말 잘한 부분입니다.
- **API route 입력 정규화가 탄탄합니다.** `page`/`limit`을 `Math.max`·`Math.min`으로 클램프하고(`route.ts:30-34`), 특수문자를 제거하고(`:37`), 2차 정렬키(`event_id`)로 페이지네이션을 안정화하고(`:59`), 빈 키워드는 200 빈 결과로 처리합니다(`:40-42`). 정렬(`.order`)을 서버에서 처리해 클라 부담도 줄였습니다.

**[제안] `toSortItems`의 강제 단언이 타입 안전성을 비웁니다 (`page.tsx:19-27`)**
`items as Array<{...}>`로 단언만 하고 런타임 검증이 없습니다. API 응답 형태가 바뀌면 `it.title`/`it.startDate`가 `undefined`가 되고, 그게 `EventCard.tsx:14`의 `date.split("-")`에서 터집니다. API가 만드는 응답 타입을 `types/`에 한 번 정의해 `fetchPage` 반환을 타입드로 받으면 단언 자체가 사라집니다(아래 4-2의 "응답 타입 단일 출처"와 같은 얘기입니다).

**[제안] API 에러를 항상 500으로 (`route.ts:62-63`)**
Supabase `error`를 그대로 `fail(error.message, 500)`로 던집니다. 클라가 보낸 입력 때문에 생긴 에러도 500이 되고, `error.message` 원문을 그대로 노출하면 DB 구조 힌트가 새 나갈 수 있습니다(6차 회원가입에서 짚었던 것과 같은 결입니다). 자체 메시지로 감싸고 원문은 서버 로그로만 남기는 편이 좋습니다.

### 이선우 (zzz664) — 미들웨어/프록시 supabase 클라이언트

**잘한 점 (결합도 분리의 모범)**
- 미들웨어 전용 클라이언트를 `lib/supabase/middleware.ts`(`createMiddlewareClient`)로 떼어내고, `setAll`에서 `request.cookies`와 `response.cookies` 양쪽에 써서 세션 갱신 쿠키가 브라우저로 나가게 했습니다(`:16-24`). `proxy.ts`는 클라이언트 생성이나 쿠키 동기화 세부를 모른 채 `getUser()`만 부릅니다(`proxy.ts:9`). 책임이 깔끔하게 나뉘었고, 이게 `@supabase/ssr` 공식 패턴입니다.
- proxy matcher가 정적 자산을 제외해(`:16`) 불필요한 미들웨어 실행을 피합니다.

**[제안] proxy에 인증 가드는 아직 없습니다 (`proxy.ts:9-11`)**
`getUser()`만 부르고 결과로 분기하지 않습니다(세션 갱신만 합니다). 지금은 보호 라우트 가드를 `home/page.tsx` 서버 컴포넌트가 담당하니 당장 버그는 아닙니다. 다만 보호 페이지가 늘면 미들웨어 한 곳에서 막는 편이 결합도상 유리합니다. 페이지별 가드로 갈지 미들웨어 중앙 가드로 갈지는 팀이 합의해 두면 좋겠습니다.

## 4. 응집도·결합도 핵심 (오늘 주제)

1. **[제안] 정렬 로직이 두 갈래로 나뉘어 있습니다.** `filterSort.ts`의 `useSort`/`SortContext`는 **클라 정렬**용인데, 실제 `search/page.tsx`는 정렬을 **서버에 위임**하면서 `changeSort`만 따로 재구현합니다(`page.tsx:52-61`). 두 `changeSort` 로직이 거의 같고, `SortItem`의 `compare` 함수들(`filterSort.ts:29,35`)은 실제 검색에선 안 쓰이는 죽은 코드가 됩니다. 데모(`/example/filter`)는 클라 정렬, 실서비스는 서버 정렬로 의도한 분리일 수도 있는데, 그렇다면 경계를 분명히 하거나(데모 전용임을 표시) 하나로 합치는 게 좋습니다. 지금은 "한 도메인의 정렬"에 진실 원천이 둘인 상태입니다.

2. **[제안] 서버 응답 타입의 단일 출처가 없습니다.** API route가 만드는 응답 객체(`route.ts:66-78`)와 클라가 받는 형태(`page.tsx:19-27`의 단언)가 서로 따로 정의돼 있습니다. 한쪽만 바뀌면 컴파일은 통과하고 런타임에 깨집니다. `types/`에 `EventSearchItem`을 정의해 서버·클라가 공유하면 결합이 타입으로 묶입니다(이 팀이 이미 쓰는 `ApiResponse<T>` 패턴과도 결이 맞습니다).

3. **[사소] 인기공연 더미 데이터가 페이지 본문에 하드코딩**돼 있습니다(`page.tsx:156-187`). 나중에 실데이터로 바꿀 지점이 흩어지고, 2장의 500도 여기서 났습니다. 상수로 분리하거나 검색 API로 통일하면 좋습니다.

4. **[사소] 검색 헤더의 ChevronLeft/Menu 버튼이 동작이 없습니다**(`page.tsx:194,205`). `cursor-pointer`만 있고 `onClick`이 없어 장식만 된 상태입니다. 추후 연결이 필요합니다.

## 5. React·Next·TypeScript 종합

**잘 쓰고 있는 것**
- `as const` + 인덱스 타입으로 리터럴 유니온을 파생하는 패턴(`SORT_OPTIONS` → `SortKey`)은 TS의 정석입니다.
- 무한스크롤에서 `useRef`(loadingRef)와 effect cleanup(ignore 플래그)을 조합해 경합을 막은 건 React를 잘 이해한 코드입니다.
- 서버/클라 경계가 맞습니다. API route는 서버 클라이언트로 DB를 다루고 정렬도 DB에 위임했습니다.

**더 볼 부분**
- `unknown[]` 단언 대신 응답 타입을 공유하기(3장 1번·4장 2번).
- `next/image` 외부 호스트 등록(2장). next/image를 쓸 거면 `next.config`에 호스트를 반드시 등록해야 한다는 게 Next의 기본 규칙이니 꼭 익혀두면 좋습니다.

## 우선순위 정리 (다음 회차 대조용)

1. **[필수] `/search` 500 복구** — `next.config.ts`에 picsum 호스트 등록(또는 더미 이미지 교체).
2. **[제안] 응답 타입 단일 출처** — `EventSearchItem`을 서버·클라가 공유, `toSortItems` 단언 제거.
3. **[제안] 정렬 로직 이원화 정리** — 클라/서버 정렬 경계 명확히, `changeSort` 중복 제거.
4. **[제안] API 에러 래핑** — 입력 기인 에러는 4xx, 원문 메시지 비노출.
5. **[제안] proxy 인증 가드 전략 합의** — 미들웨어 중앙 가드 vs 페이지별.

검색 페이지는 무한스크롤이며 경합 방어까지 공들여 잘 짜놓고 정작 이미지 설정 한 줄에 막혀 있는 상태입니다. 그 한 줄만 풀면 이번 회차 작업이 한꺼번에 살아나니, 거기부터 갑시다. 화이팅입니다.
