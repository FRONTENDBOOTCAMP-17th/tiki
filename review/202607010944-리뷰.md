## 3. 보안 실증

anon(publishable) 키로 직접 찔러 보고, service-role로 실제 행 수를 교차 확인했습니다. 결론은 **RLS가 견고합니다.**

| 테이블 | anon 결과 | 실제 행수 | 판정 |
| --- | --- | --- | --- |
| `users` | 401 거부 | 16 | 보호 O |
| `orders` | 200 rows=0 | 116 | 필터링 O (누출 없음) |
| `notification` | 200 rows=0 | 28 | 필터링 O |
| `inquiry` | 401 거부 | 1 | 보호 O |
| `event`/`ticket_grade`/`slot`/`seat` | 200 read | — | 좌석맵·상세 렌더용 공개(의도) |

anon으로 쓰기(`notification` INSERT)를 시도하면 `42501 RLS violation`으로 막히고, `event` PATCH는 200이 오긴 하지만 실제로는 0행 수정(원본 그대로)이라 변조가 안 됩니다.

15차에 짚었던 **admin 서버액션 role 검증도 이번에 일괄 적용됐습니다.** `sendNotification`을 비롯해 categories/events/members actions의 export 함수마다 최상단에 `requireAdmin()`이 붙어 있고, `requireAdmin`은 프로필 role이 admin이 아니면 throw합니다. 새로 리팩토링한 알림 기능에도 인가 구멍이 없고요. 잘 닫으셨습니다.

- **[사소]** 알림 `link` 입력이 임의 URL(`type=url`)을 허용합니다. admin 전용이라 위험은 낮지만, 수신자가 클릭했을 때 open-redirect 여지가 있으니 내부 경로(`/`로 시작)만 허용하도록 막아 두면 더 안전합니다.

| admin `/admin/notifications` | 정상 렌더 |
| 없는 이벤트(정상 UUID) `/000...000` | **200 + "존재하지 않는 공연입니다"** (과거 500 개선됨) |
| 비-UUID 문자열 `/not-a-uuid-slug` | **500** (`22P02 invalid input syntax for type uuid`) |

정상 형식 UUID인데 존재하지 않는 이벤트는 이제 `event==null` 분기로 "존재하지 않는 공연입니다"를 깔끔하게 보여 줍니다(개선됐습니다). 그런데 **UUID 형식이 아닌 아무 문자열**을 넣으면 여전히 500이 납니다. Next의 동적 세그먼트(`[eventId]`)는 아무 문자열이나 매칭하다 보니, 그 값이 Postgres uuid 컬럼 쿼리에 그대로 들어가면서 DB가 던진 에러가 페이지를 터뜨리는 겁니다. 봇이나 오타 URL이 그대로 500을 내는 상태라, 처방은 아래 6장에 적었습니다.

이번 회차에서 가장 인상적이었습니다.

- **잘한 점**: 카테고리 전체·상세 리뉴얼(`category/page.tsx`, `[slug]/page.tsx`, `CategoryEventGrid.tsx`, `CategorySectionNav.tsx`)이 전반적으로 깔끔합니다. `Promise.all`로 목록·랭킹·프로필을 병렬 조회하고, `header/section/ul/nav` 시맨틱 마크업에 `aria-hidden`·`aria-current`·`aria-label`, disabled 상태까지 접근성을 두루 챙겼습니다. 특히 `ProfileMenu.tsx`가 좋았는데, `aria-haspopup="menu"`·`aria-expanded`·`role="menu"`에 Esc 닫기, 첫 항목 포커스, 닫힐 때 트리거로 포커스 복원, Tab 포커스 트랩까지 드롭다운 접근성의 정석을 빠짐없이 갖췄습니다. 라우트가 바뀌면 드롭다운을 닫는 처리를 effect가 아니라 `prevPath !== pathname`으로 렌더 중에 파생시킨 것도 React가 권장하는 방식이라, 이 패턴을 스스로 찾아 쓰신 거라면 수준이 상당합니다. 헤더 프로필을 서버에서 받아 클라 재조회를 없앤 것도 좋았고요.
- **[제안]** `[slug]/page.tsx:80-81` — `recommended`와 `newlyOpened`가 둘 다 `eventCards.slice(0,10)`이라 인기랭킹과 사실상 같은 목록이 됩니다. 주석에 "임시 데이터"라고 솔직히 적어 두시긴 했지만, 화면에는 인기·추천·신규오픈 세 섹션이 같은 공연을 반복해서 보여 줍니다. 실데이터가 붙기 전까진 섹션을 하나로 줄이는 편이 사용자 혼란이 덜합니다.
- **[사소]** `CategoryEventGrid.tsx:93` 페이지 버튼을 `Array.from({length: totalPages})`로 전부 렌더하다 보니, 페이지가 많아지면 번호가 끝없이 늘어납니다. 지금 데이터로는 괜찮지만 `…` truncation을 염두에 두세요.
