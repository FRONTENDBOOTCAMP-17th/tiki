## 4. 컴포넌트를 "실제로 잘 동작"시키려면 — Modal로 보는 체크리스트

지금 컴포넌트들은 보기엔 잘 뜨는데, **실제 서비스에 붙이면 드러나는 빈틈**이 몇 가지 있습니다. 가장 많이 재사용될 Modal을 예로 들어 정리해봤습니다. (예시 코드는 가이드에)

1. **[필수 관심] 포커스 관리가 없습니다.** 모달이 열려도 키보드 포커스가 모달 안으로 안 들어가고, Tab을 누르면 뒤 배경의 버튼으로 빠져나갑니다. 닫을 때 원래 누른 버튼으로 포커스가 안 돌아옵니다. → 열릴 때 모달 안 첫 요소로 focus, 닫을 때 직전 요소로 복원, Tab을 모달 안에 가두기(focus trap). 모달에서 가장 중요한 부분입니다.

2. **`role="dialog"`에 이름이 없습니다.** 스크린리더가 "무슨 다이얼로그인지" 못 읽습니다. 제목에 `id`를 주고 `aria-labelledby`로 연결하세요.

3. **닫힘 정책을 고를 수 있게.** 지금은 백드롭 클릭이면 무조건 닫힙니다. "정말 결제할까요?" 같은 확인 모달은 실수로 바깥을 눌러 닫히면 곤란합니다. `closeOnBackdrop` 같은 prop으로 끌 수 있게.

4. **예제 화면 ↔ 실제 데이터의 거리.** 지금 OrderContent·NotifSend 등은 `/example/*`에서 하드코딩 props로 보여줍니다. 잘 만들었지만, 실제로 동작한다는 건 **상태와 데이터에 연결됐을 때** 증명됩니다. 예매 모달이 좌석/수량 상태를 들고 "예매" 버튼이 진짜 요청을 보내고, 알림 모달이 실제 알림 목록을 받는 순간 새 문제(로딩/에러/빈 상태/중복 클릭)가 보입니다. 다음 단계로 **컴포넌트 1~2개를 골라 실제 페이지+상태에 연결**해 보길 권합니다. 그게 "동작하는 컴포넌트"의 진짜 시험대입니다.

5. **제어/비제어 계약을 분명히.** Modal은 `open/onClose`를 부모가 쥐는 제어형이라 좋습니다. 이 규칙을 폼·토글 등 다른 컴포넌트에도 일관되게 적용하면, 어느 컴포넌트든 "상태는 부모, 표시는 컴포넌트"로 예측 가능해집니다.

## 5. 로그인/인증을 구현한다면 — 권장 방향

아직 인증 코드가 없으니(현재 `lib`에 `cn.ts`뿐), 처음부터 안전한 길로 잡아두면 좋습니다. TiKi는 Supabase를 쓸 테니 그 기준으로 정리해볼게요. 깊은 셋업은 가이드에 코드로 적어뒀습니다.

1. **`@supabase/ssr`로 서버/클라 클라이언트를 나눠 만들기.** 브라우저용·서버용을 분리하고, 세션은 **쿠키**로 관리합니다. (구버전 `auth-helpers` 말고 `@supabase/ssr`)

2. **루트 `middleware.ts`에서 세션 갱신.** 이게 없으면 로그인해도 서버 컴포넌트가 세션을 모릅니다. 보호 라우트(예: `/mypage`, 예매하기)는 여기서 미로그인 시 `/login`으로 보냅니다.

3. **"누구인지"는 항상 서버에서 확인.** 화면에서 `user.id`를 믿고 분기하지 말고, 서버 라우트/액션에서 `supabase.auth.getUser()`로 확인한 사용자 기준으로만 DB를 만집니다. 클라이언트가 보낸 user_id는 위조될 수 있습니다.

4. **RLS를 처음부터 켜기.** publishable(anon) 키는 공개됩니다. 테이블을 만드는 순간 RLS를 켜고 "본인 데이터만" 정책을 같이 씁니다. 나중에 켜려면 이미 새는 상태가 됩니다(다른 조들이 지금 이 숙제로 고생 중입니다).

5. **소셜 로그인은 콜백 라우트가 필수.** `/auth/callback`에서 `exchangeCodeForSession`을 처리하고, `redirectTo`에 쓸 `NEXT_PUBLIC_SITE_URL`을 env에 둡니다.

순서는 이렇게 추천합니다. **(1) 클라이언트 분리 → (2) 미들웨어 세션 → (3) 이메일 로그인 한 줄 흐름 → (4) RLS → (5) 소셜**. 한 번에 다 하려 들지 말고, "로그인하면 서버가 나를 안다"까지 먼저 확인한 다음에 넘어가세요.
