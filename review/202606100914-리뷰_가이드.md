# TiKi 3차 리뷰 — 해결 가이드 (2026-06-10)

리뷰에서 짚은 것들을 어떻게 고치는지 단계별로 적어둡니다. 위에서부터 순서대로 가면 됩니다.

---

## 1. [최우선] 파일명 casing 통일 — 앱 전체가 다시 살아납니다

### 문제 한 줄 요약
파일은 `button.tsx`(소문자)인데 import는 `@/components/Button`(대문자). macOS에선 같은 파일로 찾아주지만, CI/배포(Linux)와 Turbopack 빌드는 못 찾아서 **홈까지 전부 500**이 됩니다.

### 왜 git에서 casing 변경이 까다로운가
git도 macOS에선 대소문자를 잘 못 알아챕니다. 그냥 파일 탐색기에서 이름만 바꾸면 git이 "변경 없음"으로 보고 커밋이 안 됩니다. 그래서 **git이 알아듣는 방식**으로 바꿔야 합니다.

### 방법 A — `git mv` 2단계 (안전, 권장)

```bash
# button 예시 (한 파일씩)
git mv src/components/button.tsx src/components/Button.tmp.tsx
git mv src/components/Button.tmp.tsx src/components/Button.tsx

# 나머지도 동일하게
git mv src/components/footer.tsx     src/components/Footer.tmp.tsx && git mv src/components/Footer.tmp.tsx     src/components/Footer.tsx
git mv src/components/navigation.tsx src/components/Navigation.tmp.tsx && git mv src/components/Navigation.tmp.tsx src/components/Navigation.tsx
git mv src/components/header.tsx     src/components/Header.tmp.tsx && git mv src/components/Header.tmp.tsx     src/components/Header.tsx
git mv src/components/roleHeader.tsx src/components/RoleHeader.tmp.tsx && git mv src/components/RoleHeader.tmp.tsx src/components/RoleHeader.tsx
git mv src/components/logo.tsx       src/components/Logo.tmp.tsx && git mv src/components/Logo.tmp.tsx       src/components/Logo.tsx
```

### 그다음 import를 전부 PascalCase로 맞춥니다
바꿔야 할 곳(현재 기준):
- `src/app/example/button/page.tsx` → `@/components/Button`
- `src/app/example/toast/page.tsx` → `@/components/Button`
- `src/components/header.tsx` → `@/components/Button`, `@/components/logo` → `@/components/Logo`
- `src/components/roleHeader.tsx` → `@/components/logo` → `@/components/Logo`
- `src/app/example/header/page.tsx` → `@/components/header`, `@/components/roleHeader` → `@/components/Header`, `@/components/RoleHeader`
- `src/components/Header.styles.ts`의 import 경로(`./Header.styles`)는 그대로 OK

빠르게 찾기:
```bash
grep -rn "@/components/button\|@/components/header\|@/components/roleHeader\|@/components/logo\|@/components/footer\|@/components/navigation" src
```

### 검증 (반드시 셋 다 통과해야 함)
```bash
npx tsc --noEmit     # TS1261 사라져야 함
npm run build        # Build error 없이 통과(에러 0)
npx eslint .         # rules-of-hooks 등 0 error
```

> 한 번에 하기 부담되면 `Button.tsx`만 먼저 바꿔도 됩니다. 그것만으로도 빌드/홈은 복구됩니다(리뷰에서 실측 확인).

## 2. [필수] Input — `useId` 조건부 호출 고치기

지금:
```tsx
const inputId = id ?? name ?? useId();   // [X] id/name 있으면 useId 미호출 → hook 순서 깨짐
```

고친 형태:
```tsx
import { useId, type ComponentPropsWithoutRef } from 'react';
import { cn } from '@/lib/cn';

export function Input({ label, error, helperText, id, name, className, ...props }: InputProps) {
  const autoId = useId();                 // [O] 항상 최상단에서 무조건 호출
  const inputId = id ?? name ?? autoId;   // 그다음 골라 쓰기
  const descId = error || helperText ? `${inputId}-desc` : undefined;
  // ...
}
```
규칙: **Hook은 항상, 같은 순서로, 조건 없이** 부른다. 값을 고르는 건 hook 호출 뒤에.

## 4. [필수] Toast `removeToast` — setState 업데이터 안의 부수효과 빼기

지금은 `setVisible` 업데이터 함수 안에서 `setQueue`/`setTimeout`/`timers.current.set`을 호출합니다. 업데이터는 순수해야 하고 StrictMode에서 두 번 불릴 수 있어, 타이머가 중복으로 걸리거나 큐가 어긋날 수 있습니다.

권장 패턴 — "보이는 토스트가 줄면 큐에서 채운다"를 `useEffect`로 분리:

```tsx
// visible/queue는 그대로 useState
useEffect(() => {
  if (visible.length >= maxVisible) return;
  if (queue.length === 0) return;

  const [next, ...rest] = queue;
  setQueue(rest);
  setVisible((prev) => [...prev, next]);

  const timer = setTimeout(() => {
    setVisible((prev) =>
      prev.map((t) => (t.id === next.id ? { ...t, isClosing: true } : t)),
    );
  }, next.duration ?? 3000);
  timers.current.set(next.id, timer);
}, [visible, queue, maxVisible]);

const removeToast = (id: string) => {
  clearTimeout(timers.current.get(id));
  timers.current.delete(id);
  setVisible((prev) => prev.filter((t) => t.id !== id)); // 지우기만, 큐 승격은 effect가
};
```
핵심: setState 함수 안에서는 **상태 계산만** 하고, "그 결과로 또 무엇을 한다"는 effect로 옮긴다.

## 5. [필수] CI에 build + tsc 걸기 (재발 방지)

`.github/workflows/ci.yml` 새로 만들기:
```yaml
name: CI
on:
  pull_request:
    branches: [develop]
jobs:
  build:
    runs-on: ubuntu-latest   # ★ Linux = 대소문자 구분 → casing 사고를 여기서 잡음
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npx tsc --noEmit
      - run: npm run build
```
이러면 PR 올릴 때마다 빨간불/초록불로 빌드 가능 여부가 보입니다. macOS에선 안 보이던 casing 사고가 여기서 걸립니다.

## 6. [제안] 디자인 토큰으로 색 통일

지금 컴포넌트마다 보라가 제각각입니다:
- Sidebar: `bg-purple-100 text-purple-700`, badge `bg-pink-500`
- Toggle: `peer-checked:bg-purple-500`
- Input: `focus:border-violet-500 focus:ring-violet-200`

`globals.css`에 이미 `primary`/`secondary`/`accent` 100~900 토큰이 있으니 거기에 맞춥니다:
```diff
- bg-purple-100 text-purple-700
+ bg-primary-100 text-primary-800
- bg-pink-500
+ bg-accent-500
- peer-checked:bg-purple-500
+ peer-checked:bg-primary-500
- focus:border-violet-500 focus:ring-violet-200
+ focus:border-primary-500 focus:ring-primary-200
```
이러면 색을 바꿀 때 토큰 한 곳만 고치면 전체가 따라옵니다. 그게 디자인 시스템을 만드는 이유예요.
