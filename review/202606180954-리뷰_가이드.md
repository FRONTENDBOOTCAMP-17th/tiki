## 1. 마이페이지 로그인 가드 [필수]

증상: 로그인 안 한 사람이 `/mypage`에 들어가도 `/mypage/profile`이 그대로 열립니다.
원인: `mypage/layout.tsx`에 인증 확인이 없고, `Header`의 `loggedIn`이 항상 true로 하드코딩돼 있습니다.

마이페이지는 하위 페이지(profile·reservations·library·friends·settings)가 여럿이라 **레이아웃 한 곳에서** 막는 게 제일 깔끔합니다. 레이아웃을 서버 컴포넌트로 두고 거기서 로그인 여부만 확인하면 하위 페이지가 전부 한 번에 보호됩니다.

```tsx
// src/app/mypage/layout.tsx
import { redirect } from "next/navigation";
import { createClient } from "@/lib/supabase/server";
import Header from "@/components/Header";
import Footer from "@/components/Footer";
import Navigation from "@/components/Navigation";
import MyPageSidebar from "@/components/sidebar/MyPageSidebar";
import MobileDrawer from "@/components/mypage/MobileDrawer";

export default async function MyPageLayout({ children }: { children: React.ReactNode }) {
  const supabase = await createClient();
  const {
    data: { user },
  } = await supabase.auth.getUser();

  if (!user) {
    redirect("/login"); // 비로그인은 로그인 페이지로
  }

  return (
    <div className="flex min-h-screen flex-col">
      <Header loggedIn showCategory={false} />   {/* 이제 가드를 통과했으니 항상 로그인 상태가 맞습니다 */}
      {/* ...기존 레이아웃 그대로... */}
    </div>
  );
}
```

함수에 `async`를 붙이고 `await`로 `getUser()`를 부른다는 점만 기억하면 됩니다. 이렇게 두면 `Header loggedIn`도 더는 "거짓 하드코딩"이 아니라 실제로 로그인한 사용자만 도달하는 자리가 되니 의미가 맞아떨어집니다.

판매자 스토어 정보(`seller/storeInfo/page.tsx`)도 같은 패턴입니다. 여기는 이미 `getUser()`를 부르고 있어서 바로 아래에 한 줄만 더하면 됩니다.

```tsx
const {
  data: { user },
} = await supabase.auth.getUser();

if (!user) redirect("/login");   // 추가
```

> 팁: 판매자 영역도 페이지가 여러 개라면 `seller/layout.tsx`를 만들어 거기서 한 번에 가드하는 쪽이 낫습니다. 마이페이지와 똑같은 방식입니다.

## 2. 회원가입 API 에러 메시지 래핑 [제안] (이월)

`api/auth/join/route.ts:57`이 Supabase 에러 원문을 그대로 응답합니다.

```ts
// 지금
if (error) {
  return fail(error.message);   // "User already registered" 같은 영어 원문이 클라이언트로 감
}

// 제안 — 우리 코드로 매핑, 원문은 서버 로그에만
if (error) {
  console.error("[join] signUp error:", error.message); // 원문은 로그로
  const message = error.message.includes("already registered")
    ? "email_already_exists"
    : "join_failed";
  return fail(message);
}
```

이렇게 두면 클라이언트는 우리가 정한 코드(`email_already_exists` 등)만 받고, 화면에서는 그에 맞는 안내 문구를 보여주면 됩니다. 어떤 이메일이 이미 가입돼 있는지 같은 정보가 영어 원문째로 새 나가는 것도 함께 막을 수 있습니다.

## 3. 이벤트 상세 API 연결 [제안]

지금 `eventDetail/[eventId]/page.tsx`는 아래 API들을 부르는데, 아직 라우트가 없어서 mock으로 폴백합니다.

```
GET /api/events/[eventId]          ← 없음 → MOCK_* 로 폴백
GET /api/events/[eventId]/slots    ← 없음
GET /api/events/[eventId]/reviews  ← 없음
```

검색 API(`/api/events/search/route.ts`)가 이미 좋은 본보기입니다. 그 구조를 그대로 따라 단건 조회 라우트를 만들면 됩니다.

```ts
// src/app/api/events/[eventId]/route.ts
import { success, fail } from "@/lib/api/api-response";
import { createClient } from "@/lib/supabase/server";

export async function GET(_req: Request, { params }: { params: Promise<{ eventId: string }> }) {
  const { eventId } = await params;
  const supabase = await createClient();
  const { data, error } = await supabase
    .from("events")
    .select("*")
    .eq("id", eventId)
    .single();

  if (error || !data) return fail("event_not_found", 404);
  return success(data, "ok");
}
```

이렇게 하나씩 붙여 두면 페이지 코드는 손대지 않아도 fetch가 성공하면서 mock 대신 실데이터가 그려집니다. 급한 작업은 아니니 1번을 먼저 끝내고 천천히 가도 됩니다.

## 4. 판매자 스토어 입력 형식 검증 [제안]

`api/seller/store/route.ts`는 들어온 값을 그대로 저장하는데, 정산과 직결되는 값만큼은 형식을 한 번 확인하고 넘어가는 게 좋습니다. 회원가입 때 쓰던 비밀번호 정규식과 같은 방식입니다.

```ts
if (body.businessNumber !== undefined) {
  // 사업자번호: 숫자 10자리(하이픈 제거 후)
  const digits = body.businessNumber.replace(/\D/g, "");
  if (digits.length !== 10) return fail("invalid_business_number");
  fields.business_number = digits;
}
if (body.bankAccountNumber !== undefined) {
  if (!/^\d{10,16}$/.test(body.bankAccountNumber.replace(/\D/g, ""))) {
    return fail("invalid_bank_account");
  }
  fields.bank_account_number = body.bankAccountNumber;
}
```


