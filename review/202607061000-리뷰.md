## 2. 멤버별 리뷰

### 김연수 (harilog) — 스태프 초대·수락·공연별 체크인

이번 회차의 핵심이고, 인가 설계가 이 조에서 나온 코드 중 가장 잘 짜인 축에 듭니다. 좋았던 점부터 짚겠습니다.

- **권한을 화면이 아니라 DB에서 막았습니다.** `event_staff` 테이블에 RLS를 켜 두고 정책을 하나도 안 붙인 뒤(`-- 정책 없음: SECURITY DEFINER RPC로만 접근`), 모든 접근을 `SECURITY DEFINER` RPC로만 통과시켰습니다. 실제로 anon 키로 `event_staff`를 직접 읽어보면 200에 빈 배열(`[]`)이 오고, `invite_event_staff`를 직접 호출하면 401(permission denied)이 납니다. 즉 클라이언트가 아무리 요청을 조작해도 남의 스태프 데이터는 한 줄도 못 봅니다. 초보 단계에서 이 패턴을 스스로 적용한 건 정말 칭찬할 부분입니다.
- **각 RPC가 "누가 부를 수 있는가"를 정확히 검사합니다.** `invite_event_staff`는 공연 소유자(`v_event.seller_id <> v_caller`)만, `accept/reject`는 초대 당사자(`v_row.user_id <> auth.uid()`)만, `remove`는 초대한 판매자나 admin만 통과합니다. 자기 자신 초대·판매자/관리자 초대 차단, 거절했던 사람 재초대 허용까지 엣지 케이스가 촘촘합니다.
- **체크인은 이중으로 막았습니다.** 페이지(`staff/checkin/[eventId]/page.tsx:32`)에서 "그 공연에 accepted로 배정됐는지"를 먼저 보고, 실제 스캔을 처리하는 `checkin_ticket` RPC 안에서도 스태프면 해당 주문의 이벤트에 accepted 배정이 있는지 다시 확인합니다. 화면 가드가 뚫려도 DB가 한 번 더 막는 구조라, E2E에서 배정 안 된 공연(`/staff/checkin/{다른 eventId}`)으로 직접 들어가니 "배정되지 않은 공연입니다"가 정확히 떴습니다.
- 로그인 후 역할 분기(`login/action.ts:83`)도 admin/seller/staff/그 외로 깔끔하게 갈라집니다. `redirect()`가 예외를 던지는 특성상 `case` 사이에 `break`가 없어도 안전하게 동작합니다.

고칠 거리는 크지 않고 대부분 `[사소]`입니다.

- `[사소]` `login/action.ts:83`의 `data!.role` — 로그인 성공 직후 `users` 행이 아직 없는 계정(예: OAuth 최초 가입 타이밍)이면 non-null 단언 때문에 `role` 접근에서 터질 수 있습니다. `data?.role`로 받고 없으면 `default`(홈)로 보내면 더 안전합니다.
- `[사소]` 서버 액션(`lib/staff/actions.ts`)이 RPC 에러를 전부 `{ code: "error" }`로 뭉갭니다. 실제 인가는 RPC가 하니 보안 문제는 없지만, 화면에서 "권한 없음"과 "일시 오류"를 구분해 안내하려면 RPC가 돌려주는 `forbidden`/`unauthorized` 코드를 그대로 전달하는 편이 낫습니다.
- 질문 하나 — `usertest` 계정처럼 role은 `staff`인데 배정이 하나도 없는 상태가 되면 로그인 시 `/staff`로 보내지고 "스태프 전용 페이지입니다"만 뜹니다. 정상 흐름(`remove_event_staff`가 마지막 배정 해제 시 buyer로 되돌림)에서는 안 생기는데, 혹시 테스트하다 role만 수동으로 바꾼 흔적일까요? 발표 전 계정 상태만 한 번 확인해 두면 좋겠습니다.

### 방효진 (lllillly) — 예매관리·CSV·OG·통계·테스트 기반

여러 축을 두루 다뤘고, 하나하나 완성도가 있습니다.

- **테스트 기반을 처음 깔았습니다.** vitest 단위(포맷·날짜·검증·에러 매핑 등 6파일 19테스트)와 Playwright 공개 라우트 스펙을 도입했고, 지금 돌려보면 전부 초록불입니다. 지난 회차에 드렸던 프론트 테스트 이야기를 실제 코드로 옮긴 거라 방향이 아주 좋습니다.
- **CSV 내보내기 인가가 정확합니다.** `orders/export/route.ts`가 판매자 본인 공연(`seller_id = user.id`)으로 주문을 먼저 스코핑하고, `ids` 파라미터가 넘어와도 소유 이벤트 안에서만 교집합을 잡습니다. 임의의 주문 id를 넣어도 남의 주문은 안 빠져나갑니다. `csvCell`에서 콤마·따옴표·줄바꿈 이스케이프도 제대로 처리했습니다.
- 정산 중복은 unique 인덱스(`settlement_request(seller_id, period_start, period_end)`)로 DB에서 막고, 액션에서 `23505`를 잡아 "이미 신청된 기간입니다"로 친절히 안내합니다. 17차 때 봐 둔 중복 제출 걱정이 깔끔히 닫혔습니다.

고칠 거리:

- `[제안]` **CSV 수식 인젝션 방어**: `csvCell`이 콤마·따옴표는 감싸지만, 셀 값이 `=`, `+`, `-`, `@`로 시작하면 엑셀/구글시트가 이걸 수식으로 실행합니다. 이벤트명·구매자 이름처럼 사용자가 넣은 값이 그대로 들어가니, 판매자가 받은 정산 CSV를 열었을 때 의도치 않은 수식이 돌 수 있습니다. 해결은 간단합니다 — 값이 저 네 글자로 시작하면 앞에 작은따옴표(`'`)나 공백을 붙여 문자열로 강제하면 됩니다(가이드에 예시).
- `[사소]` `settlement/actions.ts`가 `23505`가 아닌 에러는 `error.message`(Postgres 원문)를 그대로 화면에 돌려줍니다. 일반 메시지로 감싸고 원문은 로그로만 남기는 편이 좋습니다.
