## 2. 이번 회차에서 가장 급한 것 — 마이페이지 로그인 가드 [필수]

E2E로 비로그인 상태에서 `/mypage`에 들어가 봤더니, `/login`으로 막히지 않고 `/mypage/profile`이 그대로 열렸습니다(T11).
화면에는 "티키님 / tiki@gmail.com / 010-1234-5678" 같은 프로필 폼이 보이고, 헤더도 로그인한 사용자용으로 떠 있었습니다.

원인은 `mypage/layout.tsx`에 인증 가드가 없다는 점입니다. 지금 레이아웃은 이렇게 시작합니다.

```tsx
export default function MyPageLayout({ children }: { children: React.ReactNode }) {
  return (
    <div className="flex min-h-screen flex-col">
      <Header loggedIn showCategory={false} />   {/* 로그인 여부와 무관하게 항상 loggedIn */}
      ...
```

`/home`은 서버 컴포넌트에서 `getUser()`로 비로그인을 막고 있는데(7차에 확인), 마이페이지에는 같은 가드가 없다 보니
로그인 상태가 페이지마다 들쭉날쭉합니다. 지금은 보이는 게 전부 플레이스홀더라 실제 개인정보가 새는 건 아닙니다. 다만
- 로그인 안 한 사람이 "내 프로필" 화면을 보는 건 분명한 UX 결함이고,
- 나중에 이 폼을 진짜 사용자 데이터에 연결하는 순간, 가드가 없으면 그대로 정보 노출이 됩니다.

마이페이지처럼 "로그인한 사람만 들어와야 하는 영역"은 **레이아웃 한 곳에서** 막는 게 가장 깔끔합니다.
`mypage/layout.tsx`를 서버 컴포넌트로 두고 `getUser()` → 없으면 `redirect("/login")` 하면, 하위 5개 페이지가 한 번에 보호됩니다.
덤으로 `Header`의 `loggedIn`도 하드코딩 대신 실제 로그인 여부로 넘길 수 있습니다. 구체적인 코드는 가이드에 적어 두었습니다.

판매자 스토어 정보(`/seller/storeInfo`)도 같은 상황입니다. 서버에서 `getUser()`는 부르는데 비로그인일 때 막지는 않아서
폼이 빈 채로 열립니다. 마이페이지와 묶어서 같은 방식으로 처리하면 됩니다.

## 3. 빌드(tsc)·CI — 건강합니다 [칭찬]

`npx tsc --noEmit` 결과 0건입니다. 3~4차에 casing 때문에 전 라우트가 깨졌던 걸 생각하면 큰 변화고,
무엇보다 **CI(`build` + `tsc`)가 PR에서 돌고 있어서** 그때 같은 사고가 다시 develop까지 새어 들어오지 못합니다.
`runs-on: ubuntu-latest` 주석에 "Linux = 대소문자 구분 → casing 사고를 여기서 잡음"이라고 의도까지 적어 둔 게 좋네요.

남은 건 작은 두 가지입니다(이월 [제안]).
- CI 트리거가 `pull_request`만이라, 실수로 develop에 직접 push 하면 검사를 안 거칩니다. `push:` 트리거도 함께 두면 안전합니다.
- `npm run lint`(eslint) 단계를 추가하면 타입뿐 아니라 hooks 규칙·미사용 변수 같은 것도 자동으로 걸립니다.

## 4. 새 엔드포인트 리뷰 (이번 회차 핵심)

지난 리뷰 이후 새로 생긴 페이지·API를 빠짐없이 봤습니다. 네 사람이 도메인을 하나씩 맡은 분담이 코드에 잘 드러납니다.

### 4-1. 이벤트 상세 — `/eventDetail/[eventId]` (강재훈)

395줄짜리 큰 화면인데 디자인 완성도가 높습니다(E2E T8, 데스크톱·모바일 모두 정상). 회차·등급·리뷰·포스터 영역이 잘 짜여 있습니다.
데이터 fetch에서 `ignore` 플래그로 cleanup을 걸어 언마운트 후 setState 경합을 막은 점이 특히 좋네요. 검색 무한스크롤 때 보였던 습관이 여기도 그대로 이어졌습니다.

- [제안] 이 페이지는 `/api/events/${eventId}`·`/slots`·리뷰 API를 fetch하는데, 지금 존재하는 이벤트 API는 `/api/events/search` 하나뿐입니다.
  나머지 라우트가 없어서 fetch가 실패하고 `MOCK_SLOTS`·`MOCK_GRADES`·`MOCK_REVIEWS`로 폴백한 화면이 보입니다. 폴백 처리 자체는
  깔끔하게 돼 있어 크래시는 안 나지만, **상세 화면이 아직 실데이터에 연결되지 않은 상태**라는 점은 짚어 둡니다. 다음 단계로 이 API들을 붙이면
  화면이 그대로 살아납니다. 폴백을 의도적으로 남겨둔 거라면 그것대로 좋습니다.

### 4-2. 회원가입·로그인 — `/join` · `/login` · `/api/auth/join` (이선우)

`SignupContext`/`useSignup`으로 가입 단계 상태를 관리하고, 가입 API에서 필드 검증 → 역할별 분기(seller면 organizerName·storeName 추가 검증)
→ 비밀번호 정규식 → camel→snake 변환까지 흐름이 또렷합니다. E2E로 빈 본문 POST를 보내니 `400 {success:false, message:"empty_email"}`로
정확히 막혔습니다(T12). 검증 뼈대가 탄탄하네요.

- [제안·이월] `route.ts:57`의 `fail(error.message)`는 Supabase가 준 에러 원문을 클라이언트로 그대로 내려보냅니다.
  "이미 가입된 이메일입니다" 같은 정보나 영어 원문이 사용자에게 노출될 수 있어서, 우리 쪽 코드(`'email_already_exists'` 등)로 매핑해 응답하고
  원문은 서버 로그에만 남기는 편이 안전합니다. 6차부터 이어온 항목이라 이번에 같이 정리하면 좋겠습니다.

### 4-3. 마이페이지 — `/mypage/profile` · `/mypage/settings` (김연수)

프로필 폼과 설정 화면 UI가 잘 만들어졌습니다. 회원 탈퇴를 바로 링크로 두지 않고 확인 모달로 한 번 더 묻게 바꾼 판단이 좋습니다
(실수로 탈퇴되는 걸 막아주는 장치죠). 다만 이 영역 전체가 로그인 가드를 못 받고 있어서, 그 부분은 2장에서 [필수]로 정리했습니다.
가드만 얹으면 화면 자체는 바로 쓸 수 있는 완성도입니다.

### 4-4. 판매자 스토어 정보 — `/seller/storeInfo` · `/api/seller/store` (방효진)

스토어 정보 수정 API가 잘 짜였습니다. `getUser()`로 먼저 인증을 확인해 비로그인은 401로 막고,
요청 본문에서 들어온 필드만 골라(`storeName`·`businessNumber`·`description`·계좌 정보 등) 부분 업데이트하는 방식이라
일부 값만 보내도 안전하게 동작합니다. `TablesUpdate<"seller_stores">` 타입을 써서 컬럼명을 타입으로 맞춘 것도 좋네요.

- [제안] 사업자번호·계좌번호·전화번호 같은 값에 형식 검증이 없습니다. 지금은 들어온 문자열을 그대로 저장하는데,
  정산·신뢰와 직결되는 정보라 최소한의 형식 체크(숫자만, 자릿수 등)를 두면 좋습니다. 회원가입 비밀번호 정규식처럼요.

> 판매자 화면 중 대시보드·티켓 관리·정산·등록·목록(`seller/dashboard` 등)은 아직 빈 스텁(`<div></div>`) 단계입니다.
> 한꺼번에 도메인을 펼치는 중이라 자연스러운 상태이고, 이번엔 실제 동작하는 storeInfo를 중심으로 봤습니다.

## 5. 팀 공통 / 협업

- **도메인 분담이 좋습니다.** 이벤트 상세(강재훈), 가입·로그인(이선우), 마이페이지(김연수), 판매자(방효진)로 네 사람이
  실제 화면을 끝까지 끌고 갔습니다. 팀장 공백이 있었던 걸 생각하면 팀이 잘 추스르고 제 속도를 찾았다고 봅니다.
- 다만 화면이 빠르게 늘면서 **인증/로그인 상태를 다루는 방식이 페이지마다 제각각**입니다. home은 서버 가드, mypage는 무방비,
  storeInfo는 getUser는 부르지만 안 막고, Header의 loggedIn은 어디선 하드코딩입니다. "로그인 필요한 영역은 레이아웃에서 가드한다"는
  규칙을 한 번 팀에서 정해 맞추면, 지금 흩어진 것들이 한 패턴으로 모입니다(2장).
- 커밋 컨벤션은 잘 지키고 있습니다. eventDetail이 mock으로 먼저 화면을 완성하고 API를 나중에 붙이는 흐름이라면,
  PR 설명이나 주석에 "API 연결 예정"을 적어 두면 리뷰하는 사람도 의도를 바로 압니다.
