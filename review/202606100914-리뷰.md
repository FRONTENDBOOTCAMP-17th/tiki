### 방효진 (lllillly) — Header / Logo / RoleHeader

좋았던 점부터.
- `Header`를 `loggedIn`·`showCategory`·`current` props로 상태를 갈라 한 컴포넌트로 구매자/비로그인/카테고리 유무를 모두 표현한 설계가 깔끔합니다. 예제 페이지(`/example/header`)에서 네 가지 상태를 한눈에 보여준 것도 좋네요.
- 아이콘 SVG에 `aria-hidden="true"`, 아이콘 링크에 `aria-label`(검색/알림/찜 등)을 빠짐없이 넣은 접근성 처리가 특히 좋습니다. 이 습관 계속 가져갑시다.
- `categoryItem`을 cva로 빼서 active 스타일을 분리한 것도 디자인 시스템답습니다.

봐주면 좋을 점.
- **[필수] `header.tsx`의 `import Button from "@/components/Button"`가 빌드를 깨뜨립니다.** 파일 실체는 `button.tsx`(소문자)예요. 이 한 줄 때문에 `/example/header`뿐 아니라 `header.tsx`를 거치는 모든 화면이 500이 됩니다. 자세한 해결은 "팀 공통 1" 참고.
- **[제안] 로그인 여부 판단의 active 매칭.** 카테고리 active를 `path === cat.href` 완전일치로 보고 있는데, `/category/concert/123` 같은 하위 경로에 들어가면 상단 탭이 활성으로 안 잡힙니다. 지금 단계에선 큰 문제는 아니지만, 나중에 `pathname.startsWith(cat.href)` 형태나 `/category` 전체 탭은 예외 처리하는 식으로 다듬으면 좋겠습니다. ("전체"가 `/category`라서 모든 하위에 다 걸리는 점도 같이 고려해야 합니다.)
- **[사소] `roleHeader.tsx`의 `roleLabels`에 타입을 한 번 묶어두면** 안전합니다. 지금은 `role: "seller" | "admin"`인데 `roleLabels`는 그냥 객체라, 나중에 role이 늘면 라벨 누락을 타입이 안 잡아줍니다. `const roleLabels: Record<RoleHeaderProps["role"], string>` 처럼 적어두면 role 추가 시 컴파일러가 라벨 누락을 알려줍니다.
- **[사소] 파일명 casing.** `header.tsx`/`roleHeader.tsx`/`logo.tsx`는 소문자, `Header.styles.ts`는 대문자로 섞여 있습니다. 컴포넌트 파일은 PascalCase(`Header.tsx`)로 통일하길 권합니다. 이유는 "팀 공통 1".

### 이선우 (zzz664) — Toast 전체 시스템 (Provider/Container/Toast/useToast/styles/animation/position)

가장 규모 큰 작업이었는데 구조가 잘 잡혔습니다.
- Context + Provider + `useToast` 훅으로 전역 토스트를 만든 설계, `success/error/warning/info` 헬퍼 제공, `maxVisible` 초과분을 `queue`로 대기시켰다가 자리 나면 꺼내는 큐 로직까지 — 초보 단계에서 이만큼 만든 건 칭찬할 만합니다.
- `useToast`에서 context가 `null`이면 "ToastProvider 내부에서만 사용 가능"이라고 throw하는 가드, 좋습니다. 이런 방어 코드가 협업에서 디버깅 시간을 줄여줍니다.
- `position`/`animation`을 맵 객체로 분리(`Toast.position.ts`, `Toast.animation.ts`)하고 `Object.freeze`까지 — 깔끔합니다.
- E2E에서 실제로 토스트가 우측 하단에 쌓이고 X로 닫히는 것 확인했습니다(`review/images/3rd-fixed-03-toast.png`).

봐주면 좋을 점.
- **[필수] `Toast.styles.ts`의 info variant 클래스에 오타가 있습니다.**
  ```ts
  info: 'bg-[#E2EEFB] text-[#3A6BA5]]',  // ← 끝에 ] 가 하나 더 붙어 있음
  ```
  Tailwind의 arbitrary value 대괄호가 `]]`로 닫혀서 이 클래스는 무효가 됩니다. info 토스트의 글자색이 의도대로 안 나와요. `text-[#3A6BA5]`로 고쳐주세요.
- **[필수] `removeToast` 안에서 `setVisible`의 업데이터로 `setQueue`를 호출**하고 있습니다.
  ```ts
  setVisible((prev) => {
    ...
    setQueue(rest);          // setState updater 안에서 다른 setState 호출
    const timer = setTimeout(...);
    timers.current.set(...);
    return [...filtered, next];
  });
  ```
  React에서 setState 업데이터 함수는 "순수"해야 합니다(렌더링 중 여러 번 불릴 수 있음). 그 안에서 `setQueue`/`setTimeout`/`timers.current.set` 같은 부수효과를 일으키면 StrictMode(개발 모드)에서 더블 호출돼 타이머가 두 번 걸리거나 큐가 어긋날 수 있습니다. `visible`을 먼저 지우고, 큐 승격은 `useEffect`로 분리하거나 별도 함수에서 처리하는 편이 안전합니다. (해결 가이드에 패턴 적어둘게요.)
- **[제안] 예제 페이지 컴포넌트 이름이 잘못 들어가 있습니다.** `src/app/example/toast/page.tsx`의 export 이름이 `ButtonExamplePage`예요(button 예제를 복붙한 흔적). `ToastExamplePage`로 바꿔주세요. 동작엔 영향 없지만 다음 사람이 헷갈립니다.
- **[제안] `Toast.tsx`의 `variant as keyof typeof defaultTitle` 단언.** `variant`가 undefined일 수 있는 타입(cva의 VariantProps는 optional)이라 `as`로 눌러 막고 있는데, defaultVariants가 'info'이긴 하지만 타입상으론 보장이 안 됩니다. `defaultTitle[variant ?? 'info']`처럼 기본값을 주면 단언 없이 안전합니다.
- **[사소] `ToastContainer`의 `min-w-100`(25rem)** 이 모바일(390px)에서 좌우 여백을 먹고 화면을 넘길 수 있습니다. `max-w-[calc(100vw-2rem)]` 같은 상한을 같이 주면 좋겠습니다.

### 김연수 (harilog) — Sidebar(Admin/Seller/MyPage/MenuItem) / Toggle / Input

- `SidebarMenuItem`을 공통으로 빼고 Admin/Seller가 메뉴 배열만 갈아끼우는 구조, 좋습니다. `SidebarItem` 타입에 `icon: LucideIcon`, `badge?: number`로 묶은 것도 깔끔합니다.
- `badge`를 `!!badge &&`로 렌더해서 `0`과 `undefined`를 안 띄운 처리, 주석으로 의도까지 적어둔 점 좋습니다.
- 주석으로 "왜 use client인지", "활성/비활성" 설명을 달아둔 것 — 본인이 이해하고 쓴 흔적이라 좋게 봅니다.

봐주면 좋을 점.
- **[필수] `Input.tsx`에서 `useId`를 조건부로 호출하고 있습니다.** (`npx eslint .` 실측: `react-hooks/rules-of-hooks` error)
  ```ts
  const inputId = id ?? name ?? useId();   // id/name 있으면 useId가 호출 안 됨
  ```
  Hook은 렌더마다 **항상 같은 순서로** 불려야 합니다. `id`/`name`이 있을 땐 `useId()`가 호출 안 되고 없을 땐 호출되니, props가 바뀌면 hook 순서가 달라져 React가 깨집니다. 항상 `const autoId = useId();`로 먼저 부르고, `const inputId = id ?? name ?? autoId;`로 쓰면 됩니다.
- **[제안] Sidebar의 색이 디자인 토큰을 안 씁니다.** `bg-purple-100`/`text-purple-700`/`bg-pink-500`(MenuItem·Admin·Seller) 같은 Tailwind 기본 팔레트를 쓰고 있는데, 팀은 이미 `globals.css`에 `primary`/`secondary`/`accent` 100~900 토큰을 정의해 뒀습니다. Toggle도 `bg-purple-500`, Input도 `focus:border-violet-500`로 제각각이에요. 같은 "보라"인데 컴포넌트마다 다른 보라가 되면 디자인 시스템의 의미가 사라집니다. `purple→primary`, `pink→accent`로 토큰에 맞춰주세요.
- **[제안] Toggle이 `name`만 받고 `checked`/`onChange`(제어)나 `id`/`aria` 연결이 없습니다.** 지금은 `defaultChecked`만 있는 비제어 컴포넌트라 폼에서 값 읽기는 되지만, 상태에 따라 UI를 바꾸거나 검증하려면 제어가 필요할 때가 옵니다. 또 `title`과 input을 `label`로 감싸 클릭은 되지만, 스크린리더용으로 `aria-label`이나 연결을 한 번 점검하면 좋겠습니다.
- **[사소] 사이드바도 파일 casing은 PascalCase로 잘 맞췄네요**(`AdminSidebar.tsx` 등). 이 기준을 header/footer/button에도 똑같이 적용하면 팀 전체가 통일됩니다.

## 팀 공통

### 1. [필수] 파일명 casing 통일 — 지금 가장 급한 한 가지

`npm run build` / `npx tsc --noEmit` / `npx eslint .` 모두 실제로 돌려본 결과입니다.

```
# tsc --noEmit
src/app/example/button/page.tsx(3,20): error TS1261:
  Already included file name '.../Button.tsx' differs from '.../button.tsx' only in casing.
  (header.tsx, example/toast/page.tsx 도 같은 '@/components/Button' import)

# npm run build  → Build error: Turbopack build failed with 3 errors
  ./src/app/example/button/page.tsx:3  Module not found: Can't resolve '@/components/Button'
  ./src/app/example/toast/page.tsx:3   Module not found: Can't resolve '@/components/Button'
  ./src/components/header.tsx:7         Module not found: Can't resolve '@/components/Button'
```

**왜 로컬에선 멀쩡해 보였을까요?** macOS의 기본 파일시스템은 대소문자를 구분하지 않습니다. 그래서 `button.tsx`를 `Button`으로 import해도 macOS는 "같은 파일"로 찾아줍니다. 하지만 **Vercel·CI·대부분의 Linux 서버는 대소문자를 구분**해서 `Button.tsx`를 못 찾고 빌드가 깨집니다. 즉 "내 컴퓨터에선 되는데 배포하면 안 되는" 가장 흔한 사고 유형입니다.

**이번에 실측으로 확인한 핵심:** Next.js 16 + Turbopack에서는 이 casing 에러가 모듈 그래프 전체를 오염시켜서, dev 서버를 띄우면 `/example/button`뿐 아니라 **홈(`/`)까지 전 라우트가 500**이 됩니다(E2E 결과 참고). 2차 때보다 범위가 커졌어요.

**증명:** 리뷰용으로 `button.tsx` → `Button.tsx`로 파일명만 바꿔 dev를 다시 띄웠더니 **모든 라우트가 즉시 200**으로 복구됐습니다. 한 글자 고치면 앱 전체가 살아납니다. (확인 후 원복했으니 여러분 코드는 그대로입니다.)

→ 권장: 컴포넌트 파일을 **PascalCase로 통일**(`button.tsx`→`Button.tsx`, `footer.tsx`→`Footer.tsx`, `navigation.tsx`→`Navigation.tsx`, `header.tsx`→`Header.tsx`, `roleHeader.tsx`→`RoleHeader.tsx`, `logo.tsx`→`Logo.tsx`)하고 import도 맞춥니다. 해결 가이드에 단계별로 적어둘게요(git에서 casing-only rename은 주의할 점이 있습니다).

### 2. [필수] CI에 `build` + `tsc`를 걸어 이 사고를 자동으로 막읍시다

①이 두 번 연속 [필수]로 올라온 건, 사람이 매번 잡기 어렵다는 뜻입니다(특히 macOS 팀이라 로컬에선 안 보임). GitHub Actions에 PR마다 `npm run build`와 `npx tsc --noEmit`를 돌리면, casing 같은 실수가 **머지 전에 빨간불**로 잡힙니다. 해결 가이드에 워크플로 YAML 예시 넣어둘게요.

### 3. [제안] 빈 스캐폴딩 정리 (2차에서 이어짐)

`src/app/**`에 0바이트 파일 27개(`home/home.tsx`, `seller/dashboard/dashboard.tsx` …)가 그대로입니다. App Router에서 라우트가 되려면 폴더 안 파일명이 `page.tsx`여야 합니다. 지금 형태는 라우트로 안 잡혀요. 또 `src/app/productDetail.tsx/`는 **폴더 이름에 `.tsx`가 붙어** 있습니다(그 안에 또 `productDetail.tsx`). 라우트를 만들 거면 `src/app/products/[id]/page.tsx` 같은 형태로 정리합시다. 당장 안 쓸 거면 지워서 트리를 깨끗하게 두는 것도 방법입니다.
