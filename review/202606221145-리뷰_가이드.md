## 0. [시연 전 필수] `SUPABASE_SERVICE_ROLE_KEY` 환경변수 확인

홈(`/`)이 `supabaseAdmin`(service-role)을 import하는데, 이 클라이언트가 모듈 로드 시점에 `SUPABASE_SERVICE_ROLE_KEY`를 읽다 보니, 이 값이 비어 있으면 홈을 비롯한 사이트 루트가 통째로 500이 납니다.

- **확인할 것**: 시연에 쓸 환경(배포 서버나 로컬 `.env.local`)에 `SUPABASE_SERVICE_ROLE_KEY`가 들어가 있는지 보면 됩니다.
- Supabase 대시보드 → Project Settings → API에서 `service_role` 키(secret)를 복사해 환경변수로 넣어주세요.
- 로컬 시연이라면 `.env.local`에 한 줄 추가하면 됩니다.

```
SUPABASE_SERVICE_ROLE_KEY=eyJ...(service_role 키)
```

> 주의: 이 키는 RLS를 통째로 우회하는 강력한 키라 클라이언트에 노출되면 큰일 납니다. `.env.local`(gitignore 대상)에만 두고 커밋은 절대 하지 마세요. `NEXT_PUBLIC_` 접두사를 붙이는 것도 안 됩니다.

## 1. [필수] 마이페이지 로그인 가드

판매자(`/seller`)를 막던 그 목록에 마이페이지만 더하면 됩니다.

```ts
// src/lib/auth/routes.ts
export const PROTECTED_ROUTES = ["/seller", "/api/seller", "/mypage"];
```

이 한 줄만 더하면 `proxy.ts`가 비로그인 사용자의 `/mypage` 접근을 알아서 `/login`으로 돌려보냅니다. 판매자 가드와 흐름이 똑같으니 따로 손볼 건 없습니다.

### 확인 방법

로그아웃한 상태로 주소창에 `/mypage`를 직접 쳐 봐서 `/login`으로 튕기면 된 겁니다(지금은 `/mypage/profile`이 그대로 열립니다).

## 2. [제안] 홈 랭킹 집계를 DB 쪽으로 옮기기 (시연 이후)

지금은 홈에서 `supabaseAdmin`으로 `orders` 전체를 읽어 와서 애플리케이션 레벨에서 event_id별로 합산하고 있습니다. 전체 랭킹은 본인 주문 RLS를 우회해야 하니 의도 자체는 맞는데, 그 탓에 공개 페이지가 service-role 키에 묶여 버린 게 마음에 걸립니다. 집계를 DB 함수로 내려보내면 공개해도 안전한 "수량 합계"만 anon 키로 읽을 수 있어서, service-role import 자체가 사라집니다.

```sql
-- 예: 이벤트별 누적 예매 수량만 돌려주는 SECURITY DEFINER 함수
create or replace function public.event_booking_counts()
returns table (event_id uuid, total_quantity bigint)
language sql
security definer            -- 정의자(소유자) 권한으로 실행 → orders RLS 우회
set search_path = public
as $$
  select event_id, sum(quantity)::bigint
  from orders
  group by event_id;
$$;

-- anon 도 호출만 가능하게 (개별 행이 아니라 집계만 노출되므로 안전)
grant execute on function public.event_booking_counts() to anon, authenticated;
```

그러면 홈 쪽은 이렇게 정리됩니다.

```ts
// page.tsx — supabaseAdmin import 제거
const { data: counts } = await supabase.rpc("event_booking_counts");
// counts: [{ event_id, total_quantity }] → 기존 bookingCountByEvent 맵 채우기
```

이렇게 두면 개별 주문 행(누가 무엇을 샀는지)은 여전히 RLS로 가려진 채로 남고, 공개에 필요한 합계만 밖으로 나가니, 홈이 더 이상 service-role 키에 기대지 않아도 됩니다.

## 3. [참고] clone/pull 후 `npm install`

오늘 PortOne SDK(`@portone/server-sdk`, `@portone/browser-sdk`)가 새로 들어갔습니다. pull 받고 나서 `npm install`을 다시 돌리지 않으면 결제 페이지랑 confirm API가 "Module not found"로 500이 나니 주의해주세요. 팀원들이 최신 develop을 받을 때 `npm install`도 같이 돌리도록 미리 공유해 두면 좋겠습니다. (CI는 깨끗해서 배포 빌드 쪽은 걱정 안 하셔도 됩니다.)

## 4. [제안·이월] join API 원문 에러 노출

`api/auth/join/route.ts`의 `fail(error.message)`는 이벤트 상세 API에서 했던 방식 그대로 정리하면 됩니다. Supabase 원문 메시지를 그대로 내보내는 대신 우리 코드(`'email_already_exists'` 등)와 상태코드만 내려주고, 원문은 `console.error`로 서버 로그에만 남기는 식이죠. 이벤트 상세에서 이미 잘 해두셨으니 그 패턴을 그대로 옮겨오면 됩니다.
