## 지금까지 잘 된 점 (세팅)

기본기를 잘 챙겨놨습니다. 하나씩 짚어보겠습니다.

- **`.gitignore`가 꼼꼼합니다.** `node_modules`, `.next/`, `.env*`, `*.pem`, supabase 임시 파일까지 빠짐없이 들어가 있습니다. 특히 `.env` 류와 `*.pem`을 처음부터 막아둔 점이 좋습니다. 초보 팀에서 가장 자주 나는 사고가 **API 키나 비밀번호를 실수로 깃에 올리는 것**인데, 그 위험을 시작부터 닫아둔 셈입니다.

- **PR 템플릿(`PULL_REQUEST_TEMPLATE.md`)을 미리 만들어 둔 점이 좋습니다.** 관련 이슈, 작업 내용, 테스트 방법, 스크린샷, 체크리스트까지 짜임새가 있습니다. 체크리스트에 "콘솔 에러 없음", "eslint / prettier 확인", "반응형 UI 확인"이 들어가 있는 걸 보면 협업을 진지하게 생각하고 출발했다고 봅니다. 이 템플릿 하나가 앞으로 PR 품질을 꽤 끌어올려 줄 겁니다.

- **PR / 머지 흐름을 실제로 연습하고 있습니다.** 커밋 기록을 보면 `config` 브랜치 → PR #1 머지 → 문제를 발견하고 `revert-1-config` 브랜치로 되돌리는 PR #2 머지, 이런 흐름이 보입니다. 건강한 신호라 생각합니다. `develop`에 바로 푸시하지 않고 **브랜치 → PR → 머지** 순서를 지켰고, 잘못 들어간 설정도 강제로 지우지 않고 `revert`로 깔끔하게 되돌렸습니다. 초보 단계에서 이 정도면 잘하고 있다고 봅니다.

- **README의 뼈대를 미리 잡아둔 점**도 좋습니다. Team Members / Project Period / Tech Stack / Features / Folder Structure / Git Convention 섹션 제목이 자리 잡고 있어서, 이제 내용만 채우면 됩니다. `kim/docs` 브랜치에서 서비스 소개를 더 풍부하게 다듬고 있는 것도 봤습니다. "슬롯 기반 예약, 실시간 잔여석, QR 티켓" 같은 핵심 도메인을 글로 정리하기 시작한 건 좋은 출발이라 생각합니다.

정리하면, **세팅·협업 규칙·보안 습관은 이미 평균 이상**입니다. 지금 비어 있는 건 "아직 안 쓴 부분"이지 "잘못 쓴 부분"이 아닙니다.

## 팀 공통 — 컨벤션 · 구조 · 협업 시작 가이드

여기서부터는 전부 **[제안]** 입니다. "이렇게 하면 앞으로 편해진다" 정도로 받아들여 주세요.

### 1. [제안] README의 빈 섹션을 "팀의 합의문"으로 채우기

지금 비어 있는 네 섹션(Tech Stack / Features / Folder Structure / Git Convention)은 사실 **팀이 함께 정해야 할 약속**입니다. 한 사람이 채우기보다, 30분 정도 모여서 같이 정하고 적기를 권합니다.

- **Tech Stack**: 이미 정해지고 있다고 봅니다. Next.js, React, TypeScript, Tailwind CSS, Supabase 정도를 쓸 텐데, **버전과 "왜 골랐는지" 한 줄**까지 적어두면 좋습니다. (예: "Supabase — 인증·DB·스토리지를 한 번에 쓸 수 있어서")
- **Features**: 거창하지 않아도 됩니다. "티켓 검색 / 상세 / 예약(슬롯·잔여석) / 결제 / 마이페이지 / 판매자 등록" 처럼 **핵심 기능을 목록으로** 적으면, 그게 곧 역할 분담의 단위가 됩니다.
- **Folder Structure**: 아래 2번에서 따로 얘기하겠습니다.
- **Git Convention**: 아래 3번에서 따로 얘기하겠습니다.

### 2. [제안] 폴더 구조를 먼저 합의하고 시작하기

코드를 쓰기 **전에** 폴더 구조를 합의해 두면, 나중에 "이 파일 어디 둬요?" 하는 혼란과 충돌을 크게 줄일 수 있습니다. Next.js App Router 기준으로 무난한 예시를 드립니다.

```
src/
  app/            # 라우팅 (page.tsx, layout.tsx)
  components/     # 재사용 UI 컴포넌트 (ui/, common/)
  features/       # 기능 단위 (ticket/, reservation/, auth/ ...)
  lib/            # supabase client, 유틸 함수
  hooks/          # 커스텀 훅
  types/          # 공통 타입 정의
  styles/         # 전역 스타일
```

꼭 이대로일 필요는 없습니다. **중요한 건 "팀이 같은 규칙을 쓴다"는 것**이지 정답을 맞히는 게 아닙니다. 정한 내용을 README의 Folder Structure 섹션에 적어두면, 새로 합류하는 사람도(혹은 나중의 본인도) 헤매지 않습니다.

### 3. [제안] 커밋 / 브랜치 컨벤션 정하기

지금 커밋 메시지를 보면 두 가지 스타일이 섞여 있습니다.
- `chore: 레포 초기 세팅 (...)`, `docs: README 수정` — 타입을 붙인 깔끔한 스타일
- `초기세팅`, `supabase세팅` — 한글로 간단히

둘 다 의미는 통하지만, **팀이 한 가지로 통일**하면 나중에 기록을 훑어볼 때 훨씬 읽기 좋습니다. 초보 팀에 권하는 무난한 규칙은 이렇습니다.

- **커밋 메시지**: `타입: 설명` 형식
  - 타입 예시: `feat`(기능), `fix`(버그), `style`(CSS·포맷), `refactor`(리팩터링), `docs`(문서), `chore`(설정·기타)
  - 예: `feat: 티켓 상세 페이지 레이아웃 추가`
- **브랜치 이름**: `feat/티켓-상세`, `fix/로그인-에러` 처럼 `타입/작업내용`
- **흐름**: `develop`에서 브랜치 따기 → 작업 → PR → 리뷰 1명 이상 → 머지

지금 이미 브랜치-PR-머지 흐름을 잘 쓰고 있으니, **거기에 "메시지 규칙"만 하나 얹으면** 컨벤션이 완성됩니다.

### 4. [제안] 이슈 / PR로 협업을 "눈에 보이게" 시작하기

기능을 나눴다면, GitHub **Issues**에 하나씩 등록해 두세요. PR을 올릴 때 템플릿의 `close #이슈번호`를 채우면 **누가 무엇을 하고 있는지가 한눈에 보입니다.** 팀 인원이 줄어든 지금일수록 "지금 무슨 일이 어디까지 됐는지"를 서로 아는 게 중요합니다. 칸반 보드(GitHub Projects)를 가볍게 하나 만들어 두는 것도 권합니다.

### 5. [제안] 새 팀 리더 / 역할 분담 README에 명시하기

`kim/docs` 브랜치를 보면 이미 README에서 **강재훈 님을 Team Leader로 옮겨 적고** 있습니다. 잘한 결정이라 생각합니다. 팀에 변동이 생겼을 때 가장 먼저 할 일이 바로 이 "리더와 역할을 다시 정하고 글로 박아두는 것"이라 봅니다. 이 변경을 `develop`까지 머지해서 **공식 명단이 현재 팀 상황과 일치**하도록 마무리해 주세요. 각자 어떤 기능을 맡는지(Features 옆에 담당자 이름) 적어두면 더 좋습니다.

## 마무리

다시 말하지만, **TiKi는 출발이 좋습니다.** 보안 습관(`.gitignore`), 협업 도구(PR 템플릿), 건강한 git 흐름(브랜치·PR·revert)까지, 코드가 많은 다른 팀에도 없는 경우가 많은 기본기입니다. 지금 비어 있는 칸들은 "못 한 것"이 아니라 "이제 채울 것"이고, 회의 한 번이면 시작됩니다.

팀에 변동이 있어서 마음이 무거웠을 수 있습니다. 그런데 인원이 줄어든 팀이 이미 README에서 리더를 다시 정하고 문서를 다듬고 있는 걸 보면, **이미 스스로 잘 추스르고 있다**고 생각합니다. 4명이면 충분히 티켓 예약 플랫폼을 만들 수 있습니다. 한 번에 다 하려 하지 말고, 작은 PR을 자주, 서로 챙겨가며 쌓아갑시다. 다음 리뷰 때는 실제로 도는 화면을 같이 보면서 이야기할 수 있길 기대합니다. 화이팅입니다.
