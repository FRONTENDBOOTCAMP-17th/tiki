# TiKi 1차 리뷰 — 해결 가이드 (지금부터 이렇게 시작합니다)

- **날짜**: 2026-06-08
- **함께 보면 좋은 문서**: `review/202606080939-리뷰.md` (리뷰 본문)

리뷰 본문에서 "이런 걸 해보면 좋다"고 짚었던 것들을, 여기선 **따라 하기만 하면 되는 단계별 안내**로 풀어보겠습니다. TiKi는 아직 코드가 거의 없는 단계라 "버그 고치기"가 아니라 **"좋게 출발하기"** 가 목표입니다. 한 번에 다 하려 하지 말고, 위에서부터 하나씩 천천히 해봅시다. 인원이 줄었지만, 이 순서대로만 가면 충분히 매끄럽게 굴러간다고 봅니다.

---

## 1단계. 새 리더 · 역할 분담 정하기

### 왜?
팀에 변동이 생겼을 때 가장 먼저 할 일은 **"누가 무엇을 책임지는지"를 다시 정하고 글로 남기는 것**이라 생각합니다. 그래야 "이건 누가 하기로 했지?" 하는 공백이 안 생깁니다. 이미 `kim/docs` 브랜치에서 강재훈 님을 리더로 옮겨 적기 시작했는데, 잘한 일입니다. 이제 팀 전체가 동의하고 마무리만 하면 됩니다.

### 이렇게 해봅시다
1. **30~60분짜리 팀 회의를 하나 잡습니다.** (오프라인이든 디스코드든 상관없습니다)
2. 회의에서 딱 세 가지만 정합니다.
   - **새 리더**: 강재훈 님으로 확정 (이미 진행 중이니 동의만 하면 끝)
   - **역할 분담**: 아래 표처럼 "각자 맡을 기능 영역"을 나눕니다. **빈 사람 없이** 모두에게 영역이 가게.
   - **컨벤션 / 다음 작업** (2~4단계 내용)
3. 정한 걸 README의 Team Members 옆이나 노션/문서에 적습니다.

역할 분담 예시 (인원에 맞게 자유롭게 조정하세요):

| 이름 | 역할 | 담당 영역(예시) |
| ---- | ---- | --------------- |
| 강재훈 | Team Leader | 진행·조율 + 예약/슬롯 기능 |
| 김연수 | Member | 문서·README + 티켓 목록/상세 |
| 방효진 | Member | 인증/로그인 + 마이페이지 |
| 이선우 | Member | 판매자 등록 + 결제 흐름 |

> 리더라고 해서 혼자 다 끌고 가야 하는 게 아닙니다. 리더의 첫 임무는 딱 하나, **"이 회의를 소집해서 결론을 내는 것"** 입니다. 그거면 충분합니다.

## 2단계. 커밋 / 브랜치 컨벤션 합의하기

### 왜?
지금 커밋 기록을 보면 `chore: 레포 초기 세팅` 처럼 타입을 붙인 것과 `초기세팅`, `supabase세팅` 처럼 한글로만 쓴 게 섞여 있습니다. 둘 다 뜻은 통하지만, **팀이 한 가지로 통일**하면 나중에 기록을 훑을 때 "이건 기능 추가였구나 / 이건 버그 수정이었구나"가 한눈에 보입니다. 협업할 때 이게 생각보다 큰 차이를 만든다고 봅니다.

### 커밋 메시지 규칙 (`타입: 설명`)

| 타입 | 언제 쓰나 | 예시 |
| ---- | --------- | ---- |
| `feat` | 새 기능 | `feat: 티켓 상세 페이지 레이아웃 추가` |
| `fix` | 버그 수정 | `fix: 예약 시 잔여석 0일 때 버튼 비활성화` |
| `style` | CSS·포맷(동작 변화 없음) | `style: 헤더 여백 정리` |
| `refactor` | 동작은 같고 코드 정리 | `refactor: 티켓 카드 컴포넌트 분리` |
| `docs` | 문서 | `docs: README 폴더구조 작성` |
| `chore` | 설정·기타 | `chore: eslint 설정 추가` |

### 브랜치 규칙 (`타입/작업내용`)
```
feat/ticket-detail      # 티켓 상세 기능
fix/login-error         # 로그인 에러 수정
docs/readme             # 문서 작업
```

### 협업 흐름 (이미 잘 하고 있습니다)
```
develop 에서 브랜치 따기
   → 작업 & 커밋
   → GitHub에 push
   → PR 생성 (PR 템플릿 채우기)
   → 팀원 1명 이상 리뷰
   → develop 에 머지
```
> 이미 `config` → PR #1 → 문제 발견 → `revert-1-config` → PR #2 흐름을 잘 썼습니다. `develop`에 바로 안 밀고 PR로 돌린 것, 잘못된 설정을 강제로 지우지 않고 `revert`로 깔끔히 되돌린 것 모두 정석입니다. **여기에 "메시지 규칙"만 얹으면 컨벤션 완성**입니다.

## 3단계. README 빈 섹션 채우기

### 왜?
README는 **팀의 약속을 적어두는 첫 페이지**입니다. Tech Stack / Features / Folder Structure / Git Convention 네 칸이 비어 있는데, 이건 "못 한 것"이 아니라 "이제 같이 정해서 채울 것"입니다. 한 사람이 다 쓰기보다, 1단계 회의에서 정한 내용을 그대로 옮기면 됩니다. 아래 템플릿을 복사해서 채워보세요.

```markdown
## Tech Stack

| 구분       | 기술                | 이유 (한 줄) |
| ---------- | ------------------- | ------------ |
| Framework  | Next.js (App Router)| 라우팅·SSR을 한 번에 |
| Language   | TypeScript          | 타입으로 실수 줄이기 |
| Styling    | Tailwind CSS        | 빠른 스타일링 |
| Backend    | Supabase            | 인증·DB·스토리지를 한 번에 |
| Deploy     | Vercel (예정)        | Next.js 배포가 간단 |

## Features

- 티켓 검색 / 목록 / 상세 보기
- 슬롯(날짜·시간) 기반 예약, 실시간 잔여석 표시
- 결제 흐름
- 마이페이지 (예약 내역 / QR 티켓)
- 판매자 등록 및 티켓 상품 등록
- 소셜 로그인 (Supabase Auth)

## Folder Structure

\`\`\`
src/
  app/          # 라우팅 (page.tsx, layout.tsx)
  components/   # 재사용 UI 컴포넌트
  features/     # 기능 단위 (ticket/, reservation/, auth/ ...)
  lib/          # supabase client, 유틸 함수
  hooks/        # 커스텀 훅
  types/        # 공통 타입
  styles/       # 전역 스타일
\`\`\`

## Git Convention

- 커밋: \`타입: 설명\` (feat / fix / style / refactor / docs / chore)
- 브랜치: \`타입/작업내용\` (예: feat/ticket-detail)
- 흐름: develop에서 브랜치 → 작업 → PR → 리뷰 → 머지
```

> 그리고 `kim/docs` 브랜치에서 다듬던 서비스 소개 + 새 리더(강재훈) 명단을 `develop`까지 머지해서, **공식 README가 현재 팀 상황과 일치**하도록 마무리해 주세요.

## 4단계. Next.js + TypeScript + Tailwind 스캐폴드 올리기

### 왜?
지금 가장 중요한 마일스톤은 **"`npm run dev`를 쳤을 때 실제로 도는 프로젝트"** 를 `develop`에 올리는 것입니다. 빈 화면이어도 괜찮습니다. 이게 있어야 모든 팀원이 같은 출발점에서 각자 기능을 붙일 수 있습니다. 개발 기간(05.28~07.08)의 약 1/3이 지난 지금, 이번 주 안에 이걸 해두면 한결 수월해진다고 봅니다.

### 이렇게 해봅시다
한 명이 대표로 프로젝트를 생성해서 PR로 올리면 됩니다.

```bash
# 1) 프로젝트 생성 (현재 폴더에 바로 만들 땐 . 사용)
npx create-next-app@latest .

# 생성 중 질문이 나오면 이렇게 고릅니다:
#   TypeScript?            → Yes
#   ESLint?                → Yes
#   Tailwind CSS?          → Yes
#   src/ directory?        → Yes
#   App Router?            → Yes
#   import alias (@/*)?    → Yes (그대로 엔터)

# 2) 잘 도는지 확인
npm run dev
# 브라우저에서 http://localhost:3000 열어 화면이 뜨면 성공입니다.
```

생성 후 3단계에서 합의한 폴더 구조에 맞게 빈 폴더만 미리 만들어 두면 좋습니다.

```bash
mkdir -p src/components src/features src/lib src/hooks src/types
```

> 처음엔 화면이 기본 Next.js 페이지여도 문제없습니다. **"도는 상태로 develop에 올라간 것"** 자체가 큰 한 걸음입니다. 그다음 Supabase client 연결(`src/lib/supabase.ts`)도 작은 PR로 다시 시도해 보세요. 지난번 supabase 설정이 revert된 건 실패가 아니라 정상적인 시행착오였으니 부담 갖지 않아도 됩니다.

## 5단계. 기능별 이슈 / PR로 협업 시작하기

### 왜?
인원이 줄어든 지금일수록 **"지금 무슨 일이 어디까지 됐는지"를 서로 아는 것**이 중요합니다. GitHub Issue에 기능을 하나씩 적어두면 누가 무얼 하고 있는지가 눈에 보이고, 일이 겹치거나 빠지는 걸 막을 수 있습니다.

### 이렇게 해봅시다
1. 2단계에서 나눈 기능들을 **GitHub Issues에 하나씩 등록**합니다.
   - 예: `[feat] 티켓 상세 페이지`, `[feat] 로그인 기능`, `[feat] 예약 슬롯 UI` ...
   - 각 이슈에 담당자(Assignee)를 지정합니다.
2. 작업을 시작할 땐 그 이슈에 맞는 브랜치를 땁니다. (예: `feat/ticket-detail`)
3. PR을 올릴 때 **템플릿의 `close #이슈번호`** 를 채웁니다.
   - 그러면 PR이 머지될 때 이슈가 자동으로 닫혀서 진행 상황이 깔끔하게 관리됩니다.
   ```markdown
   ## 관련 이슈
   close #12
   ```
4. (선택) GitHub Projects로 칸반 보드를 가볍게 하나 만들어, `할 일 → 진행 중 → 완료`로 카드를 옮기면 한눈에 보기 좋습니다.

> 처음엔 이슈 5~10개 정도면 충분합니다. 너무 잘게 쪼개려고 애쓰지 말고, "한 사람이 며칠 안에 끝낼 수 있는 단위"로 나누면 됩니다.

## 마무리 — 이 순서대로 가면 됩니다

```
1) 회의로 리더·역할·컨벤션 정하기
2) 커밋/브랜치 규칙 합의
3) README 빈 칸 채우기 (+ kim/docs 머지)
4) create-next-app 으로 도는 스캐폴드 올리기
5) 기능별 이슈 만들고 → 브랜치 → PR 로 작업 시작
```

이미 보안 습관(`.gitignore`), 협업 도구(PR 템플릿), 건강한 git 흐름을 갖췄습니다. 비어 있는 건 "이제 채울 것"일 뿐이고, 위 다섯 단계면 시작됩니다. 팀에 변동이 있었지만, 스스로 리더를 다시 정하고 문서를 다듬는 걸 보면 이미 잘 추스르고 있다고 봅니다. 4명이면 충분히 티켓 예약 플랫폼을 만들 수 있습니다. 막히는 게 있으면 언제든 물어보세요. 화이팅입니다.
