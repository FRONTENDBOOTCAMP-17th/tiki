## 1. `[필수]` 빌드가 깨져 있습니다 — 파일명 대소문자 불일치 (실측)

`npm run build`가 실패합니다.

```
./src/app/example/button/page.tsx:3:1
Module not found: Can't resolve '@/components/Button'
```

`example/button/page.tsx`는 `@/components/Button`(대문자 B)을 import 하는데, 실제 파일은 `src/components/button.tsx`(소문자 b)입니다.

- **macOS는 파일명 대소문자를 구분하지 않아** 로컬에선 가끔 동작하는 것처럼 보입니다.
- 하지만 **Vercel·OCI 같은 Linux 서버는 대소문자를 구분**해서 빌드가 그냥 실패합니다. 실제로 로컬 `next build`(Turbopack)도 위처럼 실패했습니다.
- 더 심각한 건, dev 서버(`next dev`)에서도 이 import 오류가 한 번 컴파일되면 **앱 전체로 번져 `/example/navigation`·`/`까지 500**이 되는 걸 확인했습니다(아래 E2E). 즉 한 파일 문제가 전체를 멈춥니다.

**고치는 법(둘 중 하나로 통일):**
- (권장) 컴포넌트 파일을 PascalCase로: `button.tsx` → `Button.tsx`, `footer.tsx` → `Footer.tsx`, `navigation.tsx` → `Navigation.tsx`. import는 그대로 `@/components/Button`.
- 또는 파일명을 소문자로 두고 import를 `@/components/button`으로 맞춥니다.

지금은 `Spinner.tsx`(대문자)·`Button.styles.ts`(대문자)와 `button.tsx`·`footer.tsx`(소문자)가 섞여 있어 이런 사고가 납니다. 팀 규칙을 "컴포넌트 파일 = PascalCase"로 못 박는 걸 권합니다(가이드에 일괄 변경 절차).

> 참고: macOS에서 `git mv button.tsx Button.tsx`는 대소문자만 바뀌어 git이 못 알아챌 수 있습니다. 가이드의 2단계 rename을 따르세요.

## 2. `[제안]` App Router 라우트 규칙 — `page.tsx`가 있어야 페이지가 됩니다

현재 `src/app` 아래에 `home/home.tsx`, `login/login.tsx`, `mypage/mypage.tsx`, `seller/dashboard/dashboard.tsx` … 처럼 **빈 파일 35개**가 폴더 구조로 들어가 있습니다. 그런데 Next.js App Router에서 **실제 URL로 열리는 건 `page.tsx`라는 이름의 파일뿐**입니다.

```
실제로 열리는 라우트(page.tsx 존재):  /  ·  /example/button  ·  /example/navigation  ·  /example/spinner
열리지 않는 것: /home, /login, /mypage ...  (home.tsx 등은 라우트가 아님)
```

지금은 폴더 뼈대를 잡아 둔 단계라 이해합니다. 다만 본격적으로 페이지를 만들 때는:
- 라우트가 될 파일은 반드시 `폴더/page.tsx`로.
- 라우트가 아닌 보조 컴포넌트는 `_components/` 같은 폴더에 두면 Next가 라우트로 오해하지 않습니다.

예: `src/app/login/login.tsx`(빈 파일) → `src/app/login/page.tsx`로 만들어야 `/login`이 열립니다.

또 하나, **`src/app/productDetail.tsx/`** 는 폴더 이름에 `.tsx` 확장자가 붙어 있고 그 안에 또 `productDetail.tsx`가 있습니다. 의도치 않게 만들어진 것 같으니 `src/app/product/[id]/page.tsx` 같은 형태로 정리하세요.

## 3. 멤버별 리뷰

### 이선우 (zzz664) — `cn` 유틸 / Button / Spinner

**잘한 점**
- `cn` 유틸을 `clsx` + `tailwind-merge`로 구성 — 충돌나는 Tailwind 클래스를 안전하게 합치는, 실무에서 그대로 쓰는 방식입니다.
- `Button`을 `cva`(class-variance-authority)로 variant/size/fullWidth로 분리하고, `ComponentPropsWithRef`로 ref까지 받고, `loading` 시 `disabled || loading` + `aria-busy`로 접근성까지 챙긴 점 — 군더더기 없이 잘 만들었습니다.
- 스타일(`Button.styles.ts`)과 컴포넌트(`button.tsx`)를 분리한 구조도 깔끔합니다.

**`[필수]`** 위 1번(casing). 이 import가 빌드를 깨뜨립니다. 본인이 만든 `button.tsx`를 `@/components/Button`으로 부른 거라, 파일명/임포트만 맞추면 끝납니다.

**`[제안]`** `Button`의 로딩 표시가 직접 작성한 `<div className="... animate-spin ...">`인데, 이미 잘 만든 `<Spinner>` 컴포넌트가 있습니다. `loading`일 때 `<Spinner size="sm" />`를 쓰면 스피너 스타일이 한 곳에서 관리됩니다.

**`[사소]`** `spinner` 예제 페이지(`example/spinner/page.tsx`)에 복붙 흔적이 남아 있습니다 — 함수명이 `ButtonExamplePage`, 화면 텍스트가 "Button 컴포넌트", 섹션 제목이 "Filled Variants"인데 실제론 Spinner를 보여줍니다. 문구만 정리하면 됩니다.

**`[사소]`** `outline` variant의 기본 텍스트가 `text-white`라, 흰 배경 위에서는 hover 전까지 글자가 안 보일 수 있습니다. `text-gray-700` 등 기본색을 주는 게 안전합니다.

### 방효진 (lllillly) — `Navigation` + 예제 문서 페이지

**잘한 점**
- 모바일 하단 탭바를 `min-[744px]:hidden`으로 모바일에서만 노출, 로그인/게스트 메뉴 분기(`loggedIn`), 현재 경로 active 표시(보라 인디케이터 바 + 색)까지 — 완성도가 높습니다. E2E 캡처(`images/2nd-02-navigation-mobile.png`)에서 9가지 상태가 깔끔하게 나왔습니다.
- 예제 페이지에서 **모든 상태(로그인 5탭/게스트 4탭, 각 active)** 를 한눈에 보여준 건 디자인 시스템 문서로서 아주 좋은 습관입니다.

**`[사소]`** active 판정이 `path === menu.href` 완전일치라, `/mypage/orders/123` 같은 하위 경로에선 상위 탭이 active로 안 잡힙니다. 나중에 `pathname.startsWith(menu.href)`(단, "/"는 예외 처리) 정도로 보완하면 좋습니다.

**`[제안]`** 메뉴 링크(`/mypage/orders`, `/category`, `/search`, `/login`)가 아직 실제 폴더와 다릅니다(예: 폴더는 `mypage/history`인데 링크는 `/mypage/orders`). 라우트를 실제로 만들 때 링크와 폴더명을 맞춰 주세요. 또 SVG 아이콘 8개를 이 파일 안에 인라인으로 두니 파일이 길어집니다(207줄). `components/icons/`로 빼거나 `lucide-react` 같은 아이콘 라이브러리를 쓰면 가벼워집니다.

### 강재훈 (bbalabuya, 새 리더) — `Footer` + 색 토큰(globals.css)

**잘한 점**
- Tailwind v4 `@theme`로 primary/secondary/accent를 100~900 스케일까지 정의 — 디자인 시스템의 색 토대를 제대로 깔았습니다. 컴포넌트들이 `bg-primary-700` 식으로 일관되게 쓸 수 있는 기반이 됐습니다.
- Footer 레이아웃(서비스/고객지원/판매자 3열 + 카피라이트)도 구조가 분명합니다.

**`[사소]`** Footer가 `bg-mirage`를 쓰는데, 이 색이 `@theme`에 정의돼 있는지 확인하세요(없으면 적용 안 됨). 정의돼 있다면 OK입니다. 또 `layout.tsx`의 `metadata` title이 아직 "Create Next App"입니다 — "TiKi"로 바꿔주면 좋겠습니다.

## 4. 팀 공통

- **파일/폴더 네이밍 규칙을 지금 정하고 README에 적으세요.** 컴포넌트=PascalCase, 라우트=`page.tsx`, 보조 컴포넌트=`_components/`. 1번 빌드 사고가 규칙 부재에서 나왔습니다.

- 디자인 시스템 먼저 + 예제 문서 페이지 패턴은 계속 가져가세요. 팀원들이 같은 컴포넌트를 재사용하게 되어 협업에 큰 도움이 됩니다.
- 팀장 공백에도 강재훈님 중심으로 토대를 잡아가는 흐름이 보입니다. 좋습니다.
