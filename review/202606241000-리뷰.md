| [필수] 홈이 SERVICE_ROLE_KEY에 강결합 (키 없으면 SSR 전체 크래시) | **해결** | 홈 랭킹 집계를 `event_booking_counts` RPC(마이그레이션 `20260623120000`, SECURITY DEFINER)로 옮기고, service role 클라이언트는 lazy 생성으로 바꿨습니다. 홈(`/`)이 anon 클라이언트만으로 정상 렌더되는 걸 실측했습니다. |

- **빌드 자체는 정상입니다.** 처음엔 `npm run build`가 `qrcode.react`를 못 찾아 실패했는데, `package.json`·`package-lock.json`엔 정상으로 들어가 있고 로컬 `node_modules`만 오래된 상태였습니다. `npm install` 후 다시 빌드하니 exit 0으로 깨끗하게 통과하더군요. 코드 문제가 아니라 설치 누락이라, 작업 전에 `npm install` 한 번 돌려 주면 됩니다. CI에서 `npm ci && npm run build`가 돌고 있다면 자동으로 걸러질 부분이고요.

## 3. 팀 공통 — 이번 회차 [필수]

### 3-1. [필수] 존재하지 않는 경로가 404가 아니라 500으로 터집니다

`admin@tiki.com`으로 로그인한 뒤 `/admin`을 열면 화면이 500으로 죽습니다. 원인을 따라가 보면 이렇습니다.

- 관리자 화면은 `src/app/admin/` 아래에 `dashboard.tsx` 같은 컴포넌트 파일만 있고 `page.tsx`가 없어서, App Router가 `/admin`을 라우트로 인식하지 못합니다. 그래서 `/admin`이 최상위 동적 라우트 `src/app/[eventId]/page.tsx`로 흘러들어가 `eventId = "admin"`으로 처리됩니다.
- 이 페이지는 네 개의 쿼리를 병렬로 부릅니다.

```ts
// [eventId]/page.tsx:41
const [event, slots, grades, reviewData] = await Promise.all([
  getEventDetail(eventId),  // queries.ts:58 — UUID 아니면 return null (가드 있음)
  getSlots(eventId),        // queries.ts:122 — UUID 가드 없음
  getGrades(eventId),       // queries.ts:144 — UUID 가드 없음
  getReviews(eventId),      // queries.ts:164 — UUID 가드 없음
]);
```

`getEventDetail`엔 `if (!UUID_RE.test(eventId)) return null;` 가드가 있어서 페이지도 `{!event ? "존재하지 않는 공연입니다" : ...}`로 잘 받아냅니다. **문제는 나머지 세 함수엔 같은 가드가 없다는 점입니다.** `"admin"`처럼 UUID가 아닌 값으로 `getSlots`/`getGrades`/`getReviews`가 Supabase를 호출하면 Postgres가 `invalid input syntax for type uuid`(22P02) 에러를 던지는데, 이게 `Promise.all`을 깨뜨려서 페이지 전체가 500이 되는 겁니다.

이게 [필수]인 이유는 `/admin` 하나 때문이 아니라, **사용자가 주소를 잘못 입력하거나 만료된 링크를 타고 들어와도 똑같이 500이 나고 응답에 DB 에러가 새어 나갈 수 있기** 때문입니다. 모르는 경로는 깔끔하게 404를 보여주는 게 맞습니다. 처방은 두 가지(가드를 네 함수에 통일 / 페이지 상단에서 `notFound()`)인데, 가이드에 적어 둘게요.

> 참고로 관리자 화면은 아직 라우팅이 안 잡혀 있어서 `admin@tiki.com`으로 들어가도 보여줄 페이지가 없는 상태입니다. 관리자 영역을 본격적으로 만들 땐 각 폴더에 `page.tsx`를 두면 라우트가 잡히는데, 그때 위 [eventId] 가드까지 함께 닫으면 좋겠습니다.

### 3-2. [필수] 주문 재고·동시성·취소 복원 (이월)

`orders/route.ts:44-89`는 인증 체크는 들어가 있는데(잘 했습니다), 주문이 들어와도 `ticket_grade`의 재고를 줄이지 않습니다. SELECT로 재고를 읽고 INSERT만 하다 보니, ① 같은 좌석이 동시에 두 번 팔리는 경합을 막을 수 없고 ② 취소 핸들러가 없어서 한 번 나간 재고가 돌아오지 않습니다. 티켓 예약의 핵심이라 시연 전에 한 번은 손봐야 합니다. 가이드에 "재고 차감을 조건부 UPDATE로 원자화 + 취소 시 복원"을 RPC 예시로 정리해 뒀습니다.
