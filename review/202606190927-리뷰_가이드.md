## 1. [필수] 마이페이지 로그인 가드 — `/mypage` 닫기

가드 구조는 이미 잘 만들어 두셨습니다. `/seller`를 막던 미들웨어에 `/mypage`만 합류시키면 됩니다.

### 1단계 — 보호 경로 목록에 `/mypage` 추가

```ts
// src/lib/auth/routes.ts
export const PROTECTED_ROUTES = ['/seller', '/mypage'];   // '/mypage' 추가
export const AUTH_ROUTES = ['/login', '/join'];
```

`proxy.ts`가 이미 `PROTECTED_ROUTES.some((route) => pathname.startsWith(route))`로 검사하고 있어서,
이 한 줄이면 `/mypage`, `/mypage/profile`, `/mypage/friends` 등 하위 경로 전부가 비로그인일 때 `/login`으로 막힙니다.
추가 코드는 필요 없습니다. 이게 "보호 경로를 한곳에서 관리"하는 방식의 장점입니다.

### 2단계 — 헤더의 `loggedIn` 하드코딩 풀기

가드가 생겨도 `mypage/layout.tsx`는 여전히 `<Header loggedIn ... />`로 항상 "로그인됨"을 넘깁니다.
레이아웃을 서버 컴포넌트로 두고(현재도 서버 컴포넌트입니다), 실제 로그인 상태를 읽어 넘기면 됩니다.

```tsx
// src/app/mypage/layout.tsx
import { getCurrentUser } from "@/lib/auth";   // 이미 만들어 둔 헬퍼
// ...

export default async function MyPageLayout({
  children,
}: {
  children: React.ReactNode;
}) {
  const user = await getCurrentUser();   // 미들웨어가 이미 막아주니 여기선 표시용으로만 사용

  return (
    <div className="flex min-h-screen flex-col">
      <Header loggedIn={!!user} showCategory={false} />
      {/* ...나머지 그대로... */}
    </div>
  );
}
```

> 미들웨어와 레이아웃 둘 다에서 막을 필요는 없습니다. **차단은 미들웨어 한 곳**에서 하고,
> 레이아웃에서는 헤더 표시 같은 "보여주기"에만 user를 쓰면 역할이 깔끔하게 나뉩니다.

## 2. [필수] API 원문 에러 노출 차단 (이벤트 상세 + 회원가입)

지금 두 API가 데이터베이스/Supabase가 준 원문 에러를 그대로 클라이언트에 내려보냅니다.
원문에는 컬럼 타입(`uuid`)·계정 존재 여부 같은 내부 정보가 들어 있어, 우리 코드로 매핑해 막는 게 안전합니다.

### 이벤트 상세 API — 잘못된 id 먼저 걸러내기

```ts
// src/app/api/events/[eventId]/route.ts
export async function GET(
  _request: Request,
  { params }: { params: Promise<{ eventId: string }> },
) {
  const { eventId } = await params;

  // 1) uuid 형식이 아니면 DB에 보내기 전에 차단 (원문 에러를 애초에 안 만듦)
  const UUID = /^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$/i;
  if (!UUID.test(eventId)) {
    return fail("invalid_event_id", 400);
  }

  const supabase = await createClient();
  const { data, error } = await supabase
    .from("event")
    .select("...")
    .eq("event_id", eventId)
    .maybeSingle();

  // 2) 그래도 DB 에러가 나면 원문 대신 일반 메시지 + 서버 로그
  if (error) {
    console.error("[events/[eventId]] db error:", error.message);   // 원문은 로그만
    return fail("internal_error", 500);
  }
  if (!data) return fail("event_not_found", 404);

  // ...이하 매핑 그대로...
}
```

핵심은 **DB에 보내기 전에 형식을 먼저 검사**해서 잘못된 입력은 400으로 끊는 것, 그리고 **그래도 DB 에러가 나면 `error.message` 대신
우리 코드(`internal_error`)만 응답**하고 원문은 `console.error`로 서버에만 남기는 것입니다.

### 회원가입 API — 같은 방식으로

```ts
// src/app/api/auth/join/route.ts
const { error } = await supabase.auth.signUp({ /* ... */ });

if (error) {
  console.error("[auth/join] signUp error:", error.message);   // 원문은 로그만
  // 자주 나오는 케이스는 우리 코드로 매핑 (필요한 만큼만)
  if (error.message.includes("already registered")) {
    return fail("email_already_exists", 409);
  }
  return fail("signup_failed", 400);
}
```

클라이언트는 `email_already_exists` 같은 우리 코드를 받아 한국어 안내로 바꿔 보여주면 됩니다.
가능하면 이런 매핑을 `lib/api/` 어딘가에 함수 하나로 모아두면, 앞으로 API가 늘어도 같은 규칙으로 갑니다.

## 3. [제안] 상세 화면 등급(grades) 실연결

지금 상세 페이지는 슬롯·리뷰는 API에서 받지만 등급은 `MOCK_GRADES` 하드코딩이라, 예매 버튼이 mock 등급 id(`grd_1`)를
주문 API로 보냅니다. 실데이터가 들어오면 주문이 `ticket grade not found`로 떨어지니, 등급도 API에서 받아오게 바꿉니다.

1. `/api/events/[eventId]/slots` 라우트를 만들어 회차와 등급을 함께 내려주거나(예: `{ slots, grades }`),
   등급 전용 라우트를 따로 둡니다. (이미 `ticket_grade` 테이블을 주문 API에서 읽고 있으니 같은 패턴으로 SELECT 하면 됩니다.)
2. 상세 페이지에서 `grades`도 상태로 두고 fetch한 뒤, `<BookingWidget grades={...} />`에 mock 대신 실데이터를 넘깁니다.
3. 연결이 끝나면 `MOCK_GRADES`·`MOCK_SLOTS`를 지웁니다. 시드가 들어오면 화면이 그대로 살아납니다.

## 4. [제안] 판매자 페이지의 `user!.id` → `requireUser()`

`lib/auth.ts`에 이미 좋은 헬퍼가 있습니다. 페이지 맨 앞에서 부르면 단언(`!`) 없이 안전합니다.

```tsx
// src/app/seller/list/page.tsx, src/app/seller/events/[id]/page.tsx
import { requireUser } from "@/lib/auth";

export default async function Page() {
  const user = await requireUser();   // 없으면 알아서 /login 으로 redirect
  const supabase = await createClient();

  const { data: events } = await supabase
    .from("event")
    .select("*")
    .eq("seller_id", user.id)   // user! 가 아니라 user.id — 타입도 안전
    // ...
}
```

이벤트 등록 API(`/api/seller/event`)에는 "판매자 역할인지" 확인을 한 줄 더하면 좋습니다.

```ts
const { data: { user } } = await supabase.auth.getUser();
if (!user) return fail("unauthorized", 401);

// 가입 시 저장한 역할 확인 (테이블/컬럼명은 실제 스키마에 맞춰서)
const { data: profile } = await supabase
  .from("users").select("role").eq("id", user.id).single();
if (profile?.role !== "seller") return fail("forbidden", 403);
```

## 5. [제안] 탈퇴·비번 변경 버튼 피드백

`WithdrawButton`·`PasswordChangeButton`의 제출 핸들러가 `// TODO`로 비어 있어, 시연에서 버튼을 눌러도 조용히 닫히기만 합니다.
서버 액션 연결 전이라면, 누른 순간 토스트로 피드백만 줘도 동작이 살아 보입니다.

```tsx
const handleWithdraw = (formData: FormData) => {
  // TODO: 비밀번호 검증 + 회원 탈퇴 server action
  toast.info("회원 탈퇴 기능은 준비 중입니다");   // 임시 피드백
  setStep("none");
};
```

여유가 되면 Supabase의 비밀번호 변경은 `supabase.auth.updateUser({ password })`,
회원 탈퇴는 서버 액션에서 처리하는 식으로 이어가면 됩니다.

## 6. [제안] 로그인 화면 미구현 링크 정리

`login/page.tsx`의 "아이디 찾기"(`/find-id`)·"비밀번호 찾기"(`/find-password`)는 아직 라우트가 없어 누르면 404가 납니다.
시연 중 클릭 사고를 막으려면, 라우트를 만들 게 아니면 잠시 막아두는 편이 안전합니다.

```tsx
{/* 준비 전까지는 링크 대신 흐린 텍스트로 */}
<span className="text-gray-300 cursor-not-allowed">아이디 찾기</span>
```
