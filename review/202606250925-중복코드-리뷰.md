# TiKi 중복 코드 리뷰 (2026-06-25)

이번엔 **반복·복붙된 코드**를 한 번 훑었습니다. 같이 드린 「리팩토링 가이드」가 원칙이라면, 이 문서는 그걸 TiKi 코드에 대입한 결과입니다. 무조건 다 합치라는 얘기가 아니라, 지금 빼면 이득인 것과 둬도 되는 것을 나눠 정리했습니다.

먼저 칭찬부터. 이 팀은 공통화 의식이 꽤 있는 편입니다. `lib/supabase/{client,server,admin}.ts`로 클라이언트를 모았고, `lib/auth.ts`의 `getCurrentUser`, `lib/api/api-response.ts`의 `success/fail`, `components/modal/{Modal,Dialog}.tsx` 베이스 모달, `lib/format.ts` 포맷 유틸까지 토대가 잘 잡혀 있어요. 그래서 아래 지적은 "토대는 있는데 일부 화면이 그걸 안 쓰고 따로 복붙했다"는 양상이 많습니다.

---

## 1. [필수] 같은 DB 테이블 타입을 페이지마다 손으로 다시 적었습니다

`src/types/database.ts`에 Supabase 생성 타입(`Tables<"event">` 등)이 있고, `storeInfo/types.ts`는 `export type Store = Tables<"seller_stores">`로 **모범적으로** 쓰고 있습니다. 그런데 다른 페이지들은 같은 테이블을 snake_case로 손으로 다시 선언했어요.

- `seller/list/types.ts:1` `interface Event { event_id; title; status; venue_name; start_date; ... }`
- `seller/events/types.ts:1` `interface EventDetail { ... }` (위와 필드 대거 중복)
- `seller/ticketManagement/types.ts:1` `OrderRow` vs `seller/settlement/types.ts:1` `SettlementOrder` (event_id/amount/quantity/status 겹침)

```ts
// list/types.ts
export interface Event { event_id: string; title: string; status: "공개"|"비공개"; venue_name: string; start_date: string; /*...*/ }
// events/types.ts — 같은 테이블을 다시 선언
export interface EventDetail { event_id: string; title: string; status: "공개"|"비공개"; venue_name: string; start_date: string; /*...*/ }
```

문제는 스키마가 바뀌었을 때입니다. 손으로 적은 타입은 컬럼 추가·이름 변경을 모르고, 그래서 코드와 실제 데이터가 조용히 어긋납니다. `storeInfo`처럼 생성 타입에서 파생하면 컴파일러가 잡아줘요.

```ts
import type { Tables } from "@/types/database";
export type EventListItem = Tables<"event"> & {
  totalOrders: number; remainingSeats: number; totalRevenue: number;
};
```

`storeInfo/types.ts`(잘함)와 `list/types.ts`(손으로 적음)를 나란히 두고 보면 "스키마가 바뀔 때 컴파일러가 잡아주느냐"의 차이가 한눈에 보입니다. 화면용 카드 타입 `EventCardItem`도 `types/domain/event.ts:54`에 정식으로 있는데 `search` 컴포넌트들(`PopularList.tsx:13`, `category/[slug]/EventList.tsx:11` 등)이 비슷한 shape를 각자 인라인 선언합니다. 정식 타입을 import하면 됩니다.

## 2. [필수] 최저가 계산 블록이 4곳에 똑같이 복붙됐습니다

데이터 페칭 안에서 완전히 동일한 최저가 계산이 네 번 반복됩니다.

- `lib/event/upcoming.ts:57,65`, `lib/event/ranking.ts:87,95`, `lib/api/categories.ts`, `app/api/events/search/route.ts:68,74`

```ts
const prices = (e.ticket_grade ?? []).map((g) => g.price);
minPrice: prices.length > 0 ? Math.min(...prices) : null,
```

네 번이면 Rule of Three를 명백히 넘겼고, "매진/할인 반영" 같은 규칙이 생기면 네 곳을 다 고쳐야 합니다. 헬퍼 하나로 빼세요. (그 위의 `ticket_grade: { price: number }[] | null` 행 타입도 같은 네 파일에 재선언돼 있어 6번과 연결됩니다.)

```ts
export function minPriceOf(grades?: { price: number }[] | null) {
  const prices = (grades ?? []).map((g) => g.price);
  return prices.length ? Math.min(...prices) : null;
}
```

## 3. [필수] 포맷 유틸을 만들어 놓고 search 폴더가 또 만들었습니다

`lib/format.ts`가 있는데 `app/search/_components/utils.ts`가 거의 동일한 함수를 별도로 재정의했습니다.

```ts
// lib/format.ts
export function formatShortDate(date: string) {
  const [, month, day] = date.split("-");
  return `${parseInt(month)}월 ${parseInt(day)}일`;
}
// search/_components/utils.ts — 복붙
export function formatEventDate(date: string) {
  const [, month, day] = date.split("-");
  return `${parseInt(month)}월 ${parseInt(day)}일`;
}
```

`RankingTabs.tsx:16`에도 또 다른 인라인 날짜 포맷이 있어요. 유틸의 존재 이유는 "한 곳만 고치면 된다"인데, search만 따로 놀면 표기가 미묘하게 어긋날 수 있습니다(`~` 유무 등). `search/_components/utils.ts`를 지우고 `lib/format.ts`를 import하세요.

## 4. [제안] 세일러 API 라우트의 인증 가드가 7곳에 반복됩니다

```ts
const user = await getCurrentUser();
if (!user) return fail("unauthorized", 401);
const supabase = await createClient();
```

`event/route.ts:80`, `event/stats/route.ts:6`, `store/route.ts:26`, `orders/route.ts:38` 등 일곱 곳입니다. `getCurrentUser`/`fail`로 절반은 이미 공통화하셔서 좋은데, 이건 단순 중복을 넘어 **보안 일관성** 문제라 한 번 더 짚습니다. 한 라우트에서 이 가드를 빠뜨리면 곧 인가 누락이거든요. 묶어두면 빠질 일이 줄어듭니다.

```ts
export async function requireUserApi() {
  const user = await getCurrentUser();
  if (!user) return { error: fail("unauthorized", 401) } as const;
  return { user, supabase: await createClient() } as const;
}
// 사용: const ctx = await requireUserApi(); if ("error" in ctx) return ctx.error;
```

## 5. [제안] StatCard·EmptyState·InfoField — 인라인 마크업 모으기

베이스 `Modal`은 잘 재사용하시는데, 모달 *안쪽 콘텐츠*와 작은 UI 조각이 복붙입니다.

- **통계 타일**: `StatCard.tsx` 공용 컴포넌트가 있는데도 `seller/(dashboard)/page.tsx:91`, `EventCard.tsx:113`, `EventDetailView.tsx:117`(내부에 `Stat` 재정의 290) 등 6+곳이 "라벨 + 굵은 숫자 + 단위"를 인라인으로 다시 그립니다. `StatCard`로 통일하고 숫자 포맷은 `formatCount(n, "건")` 유틸로.
- **빈 상태**: `mypage/reservations/page.tsx:81,176`, `SellerReviewList.tsx:215` 등에 같은 안내문이 반복되는데 border 색(`gray-100` vs `gray-200`)이 이미 살짝 어긋났습니다. `<EmptyState message="..." />` 하나로.
- **모달 정보 박스**: `CancelledDetailModal.tsx:45`, `OrderTable.tsx:496` 등의 "라벨/값 한 쌍"을 `<InfoField label value />`로, 아이콘 배너(`Calendar/Clock/MapPin + 텍스트`)를 `<EventInfoBanner />`로 빼면 모달 3~4개가 정리됩니다.

## 6. [사소] 지금은 둬도 되는 것

- **이벤트 등록폼 ↔ 수정폼** (`EventCreateForm.tsx`, `EventEditForm.tsx`): 입력 클래스·헬퍼가 겹치지만, 두 폼은 "생성 vs 수정"이라 흐름이 꽤 달라서 **한 컴포넌트로 억지 통합하면 분기 지옥**이 됩니다. `inputClass`/`selectClass` 같은 공통 *조각*만 빼고 폼 자체는 둘로 두세요.
- **상태 문자열 `"공개"/"비공개"`**: 유니온 타입으로 좁혀둔 곳이 있어 큰 위험은 아닙니다. 값이 더 늘어나면 그때 `EVENT_STATUS` 상수로 모으세요.
- **`example/*` 데모 페이지 13개**: 컴포넌트 카탈로그라 중복처럼 보여도 의도된 것입니다. 배포 빌드에 꼭 필요한지만 한 번 점검하세요.

이미 잘 된 것(클라이언트, auth, success/fail, 베이스 Modal)은 **더 손대지 마세요.** `category/_categories.ts`(아이콘 매핑)와 `lib/api/categories.ts`(DB 조회)는 이름만 비슷할 뿐 다른 책임이라 합치면 안 됩니다.

## 한 줄 정리

공통 토대는 잘 깔아놓으셨는데, **그 토대를 일부 화면이 안 쓰고 따로 복붙**한 게 핵심입니다. 우선순위는 ① 생성 타입 파생으로 row 타입 단일화(1번) → ② minPrice 계산 헬퍼(2번) → ③ search 포맷 유틸 통합(3번) → ④ API 인증 가드 래퍼(4번) 순서를 권합니다. 6번은 두셔도 됩니다.
