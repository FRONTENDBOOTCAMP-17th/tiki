## 1. [제일 급함] 홈 service role 강결합 — `src/lib/supabase/admin.ts`

지금 구조는 이 파일을 import하는 순간 클라이언트를 바로 만들어버리는데요, 그러다 보니 키가 없으면 import만 해도 그 자리에서 터집니다.

```ts
// 변경 전 — 모듈 로드 시점에 즉시 실행 (키 없으면 import만 해도 크래시)
export const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
);

// 변경 후 — 필요할 때 함수로 생성 (+ 키 없을 때 명확한 에러)
export function getSupabaseAdmin() {
  const key = process.env.SUPABASE_SERVICE_ROLE_KEY;
  if (!key) throw new Error("SUPABASE_SERVICE_ROLE_KEY 미설정 (서버 전용)");
  return createClient(process.env.NEXT_PUBLIC_SUPABASE_URL!, key);
}
```

그리고 한 발 더 나아가서 **홈이 정말 service role까지 필요한지** 한번 짚어보면 좋겠습니다. 인기 랭킹은 어차피 공개 데이터라 anon으로도 충분히 가져올 수 있거든요. 홈을 anon 클라이언트로 바꿔두면 service role 의존 자체가 사라져서, 키가 없어도 루트는 정상적으로 뜨게 됩니다.

## 2. 재고 차감 + 동시성 — `src/app/api/orders/route.ts`

읽고-검사-삽입으로 이어지는 사이에 락이 없다 보니, 동시에 들어온 주문이 전부 통과해버립니다. 차감은 코드 단이 아니라 DB에서 원자적으로 처리해주세요.

```sql
-- 잔여가 충분할 때만 차감 (조건이 안 맞으면 0행 → 매진 처리)
update ticket_grade
   set quantity = quantity - p_qty
 where grade_id = p_grade_id and quantity >= p_qty
returning quantity;
```

코드 쪽에서는 이 update의 반환이 0행으로 오면 "매진"으로 보고 주문을 만들지 않으면 됩니다. 여기서 더 안전하게 가려면 주문 생성 전체를 plpgsql RPC 하나의 트랜잭션으로 묶어서, 차감과 orders insert가 한 묶음으로 같이 성공하거나 같이 실패하도록 만드는 게 좋습니다. 취소할 때는 `quantity = quantity + p_qty`로 다시 복원해주면 되고요. 잔여가 음수로 떨어지는 일이 없도록 CHECK 제약도 함께 걸어두세요.

```sql
alter table ticket_grade add constraint quantity_nonneg check (quantity >= 0);
```

## 3. orders 외래키 추가

```sql
alter table orders
  add constraint fk_orders_event   foreign key (event_id)        references event(event_id),
  add constraint fk_orders_slot    foreign key (slot_id)         references slot(slot_id),
  add constraint fk_orders_grade   foreign key (ticket_grade_id) references ticket_grade(grade_id),
  add constraint fk_orders_user    foreign key (user_id)         references users(id);
```

FK를 걸어두면 두 가지가 좋아지는데요, 하나는 잘못된 id로는 주문이 아예 안 들어간다는 점이고, 다른 하나는 Supabase 중첩 select를 쓸 수 있게 돼서 예매내역 쿼리가 한 방으로 줄어든다는 점입니다.

```ts
// FK가 생기면 4번 왕복이 1번으로
const { data } = await supabase
  .from("orders")
  .select("*, event:event_id(title, thumbnail), slot:slot_id(date,start_time), grade:ticket_grade_id(grade_name,price)")
  .eq("user_id", userId)
  .order("created_at", { ascending: false });
```

## 4. 상태값 enum 표준화

```sql
create type order_status as enum ('cart','ordered','paid','cancelled');
alter table orders alter column status type order_status using status::order_status;
```

그리고 코드에서는 문자열을 여기저기 직접 쓰지 말고, 공유 상수로 한 군데 모아두세요.

```ts
export const ORDER_STATUS = { CART:'cart', ORDERED:'ordered', PAID:'paid', CANCELLED:'cancelled' } as const;
```

라이브러리 조회도 `"ordered"`만 보지 말고 `["ordered","paid"]`처럼 결제 완료 상태까지 같이 포함하는 게 맞습니다. 매진 판정 쪽은 이벤트 status가 `"공개"`(한글)로 들어온다는 점을 반영해서 비교값을 맞춰주거나, 아예 `ticket_grade.quantity` 합으로 판단하는 편이 더 정확합니다.

## 5. 검색 인덱스 + 에러 메시지

```sql
create extension if not exists pg_trgm;
create index if not exists idx_event_title_trgm on event using gin (title gin_trgm_ops);
create index if not exists idx_event_venue_trgm on event using gin (venue_name gin_trgm_ops);
```

에러 응답은 원문 메시지를 그대로 내보내지 말고 코드로 바꿔주세요.


---

손대는 순서는 **1(홈 강결합) → 2(재고 차감) → 3(FK)**이 제일 급합니다. 1번은 사이트가 뜨느냐 마느냐가 걸린 문제라 가장 먼저 잡아야 하고, 2번과 3번은 티켓 도메인이라면 기본으로 갖춰야 할 부분이라 그다음으로 보면 됩니다. 화이팅입니다.
