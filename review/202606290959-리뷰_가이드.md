# TiKi 14차 리뷰 — 해결 가이드 (2026-06-29)

리뷰에서 짚은 [필수] 항목을 어떻게 고치는지 단계별로 정리했습니다. SEO 개념 전반(크롤러가 무엇을 보는지, Metadata API, OG, 검증법)은 따로 공유한 `seo-가이드.md`에 처음부터 끝까지 적어뒀으니, 막히면 그 글의 해당 장을 같이 보면 됩니다.

---

## 1. 루트 metadata 교체 + lang="ko"

`src/app/layout.tsx`의 기본값을 실제 서비스 정보로 바꿔줍니다. (seo-가이드.md 1장)

```tsx
// src/app/layout.tsx
import type { Metadata } from 'next';

export const metadata: Metadata = {
  metadataBase: new URL('https://tiki.example.com'), // 배포 도메인으로 교체
  title: {
    default: 'TiKi — 공연·이벤트 티켓 예약',
    template: '%s | TiKi', // 하위 페이지 제목 뒤에 자동으로 " | TiKi"
  },
  description: '보고 싶은 공연을 한 곳에서. 예매부터 좌석·등급 선택까지 간편하게.',
  openGraph: {
    type: 'website',
    siteName: 'TiKi',
    locale: 'ko_KR',
  },
};

export default function RootLayout({ children }: Readonly<{ children: React.ReactNode }>) {
  return (
    <html lang="ko" className={`...기존 클래스 유지...`}>
      {/* 기존 body 그대로 */}
    </html>
  );
}
```

핵심 세 가지만 기억하면 됩니다. `metadataBase`를 적어두면 OG 이미지 상대경로가 절대경로로 바뀌어 카카오톡 미리보기가 안 깨지고, `title.template` 덕에 하위 페이지는 `title: '제목'`만 적어도 `제목 | TiKi`가 됩니다. `lang="ko"`는 검색엔진·스크린리더가 언어를 제대로 인식하게 합니다.

## 2. 이벤트 상세 generateMetadata (+ React.cache로 중복 조회 방지)

먼저 `getEventDetail`을 `React.cache`로 감싸 같은 요청 안에서 한 번만 DB를 치게 합니다. (seo-가이드.md 2·9장)

```ts
// src/lib/event/queries.ts
import { cache } from 'react';

export const getEventDetail = cache(async (eventId: string) => {
  // ...기존 본문 그대로...
});
```

그다음 `src/app/[eventId]/page.tsx`에 `generateMetadata`를 추가합니다. 페이지 본문이 이미 `getEventDetail(eventId)`를 부르고 있으니, 같은 함수를 메타에서도 부르면 됩니다(cache 덕에 DB는 한 번).

```tsx
// src/app/[eventId]/page.tsx
import type { Metadata } from 'next';

export async function generateMetadata(
  { params }: { params: Promise<{ eventId: string }> }
): Promise<Metadata> {
  const { eventId } = await params;
  const event = await getEventDetail(eventId);

  if (!event) {
    return { title: '존재하지 않는 공연' };
  }

  const description = event.description?.slice(0, 120) ?? `${event.title} 예매 정보`;

  return {
    title: event.title, // → "○○ 콘서트 | TiKi"
    description,
    alternates: { canonical: `/${eventId}` },
    openGraph: {
      title: event.title,
      description,
      url: `/${eventId}`,
      images: event.images?.[0] ? [{ url: event.images[0] }] : undefined,
    },
  };
}
```

soft-404도 같이 정리하면 좋습니다. 지금은 `event`가 `null`이면 안내 문구를 200 OK로 내려주는데(page.tsx:80), 없는 공연은 `notFound()`로 진짜 404를 내려주는 게 SEO상 깔끔합니다.

```tsx
import { notFound } from 'next/navigation';
// ...
if (!event) notFound();
```

카테고리 상세는 고정 카테고리라 정적 `metadata`만 export하면 됩니다.

```tsx
// src/app/category/[slug]/page.tsx
export async function generateMetadata({ params }): Promise<Metadata> {
  const { slug } = await params;
  const category = categories.find((c) => c.slug === slug);
  return { title: category ? category.label : '카테고리' };
}
```

## 3. sitemap.ts / robots.ts 추가

App Router에서는 파일 하나씩이면 됩니다. (seo-가이드.md 6·7장)

```ts
// src/app/sitemap.ts
import type { MetadataRoute } from 'next';
import { createClient } from '@/lib/supabase/server';

const BASE = 'https://tiki.example.com';

export default async function sitemap(): Promise<MetadataRoute.Sitemap> {
  const supabase = await createClient();
  const { data: events } = await supabase
    .from('event')
    .select('event_id')
    .eq('status', '공개'); // 공개 이벤트만

  const eventUrls = (events ?? []).map((e) => ({
    url: `${BASE}/${e.event_id}`,
    lastModified: new Date(),
    changeFrequency: 'daily' as const,
    priority: 0.8,
  }));

  return [
    { url: BASE, changeFrequency: 'daily', priority: 1 },
    ...eventUrls,
  ];
}
```

```ts
// src/app/robots.ts
import type { MetadataRoute } from 'next';

const BASE = 'https://tiki.example.com';

export default function robots(): MetadataRoute.Robots {
  return {
    rules: {
      userAgent: '*',
      allow: '/',
      disallow: ['/admin/', '/mypage/', '/seller/', '/payment/', '/api/'],
    },
    sitemap: `${BASE}/sitemap.xml`,
  };
}
```

dev 서버에서 `curl -s http://localhost:3000/sitemap.xml | head`와 `curl -s http://localhost:3000/robots.txt`로 바로 확인됩니다.

## 4. 관리자 서버액션 권한 가드 (가장 중요)

`lib/auth.ts`에 이미 `requireUser`(미로그인 시 redirect)와 `getCurrentProfile`(users 테이블에서 role 조회)이 있으니 그대로 씁니다. 관리자 액션 함수 첫 줄에서 role을 확인하는 작은 헬퍼를 하나 만들어두면 모든 액션이 한 줄로 끝납니다.

```ts
// src/lib/auth.ts 에 추가
export async function requireAdmin() {
  const profile = await getCurrentProfile();
  if (!profile || profile.role !== 'admin') {
    throw new Error('관리자 권한이 필요합니다.');
  }
  return profile;
}
```

그리고 각 액션 첫 줄에 호출합니다.

```ts
// src/app/admin/notifications/actions.ts
import { requireAdmin } from '@/lib/auth';

export async function sendNotification({ target, userIds, title }: {...}) {
  await requireAdmin(); // ← 이 한 줄. 일반 회원 직접 호출 차단
  if (!title.trim()) return { error: '알림 내용을 입력해주세요' };
  // ...기존 본문 그대로...
}
```

```ts
// src/app/admin/events/actions.ts — 세 함수 모두 동일하게
export async function hideEvent(eventId: string) {
  await requireAdmin();
  // ...
}
export async function publishEvent(eventId: string) {
  await requireAdmin();
  // ...
}
export async function deleteEvent(eventId: string) {
  await requireAdmin();
  // ...
}
```

왜 이렇게 해야 하냐면, 서버액션은 화면 버튼에만 묶인 게 아니라 사실상 누구나 POST로 직접 부를 수 있는 엔드포인트이기 때문입니다. admin 레이아웃 가드는 "화면을 여는" 사람만 막을 뿐, 함수 자체를 직접 호출하는 요청은 막지 못합니다. 그래서 권한 검증은 **화면이 아니라 함수 안에서** 해야 합니다.

`deleteEvent`의 순차 삭제도 단계별 에러를 확인하거나, 더 깔끔하게는 FK에 `ON DELETE CASCADE`를 걸어 `event` 한 줄만 지우면 연관 행이 자동으로 따라 지워지게 하면 좋습니다.

## 5. cancelReservation await 추가

`src/app/payment/[orderId]/_components/PaymentForm.tsx:90`의 리다이렉트 취소 경로에 `await`를 붙이고, 콜백을 async로 바꿔줍니다.

```tsx
// 현재 (90줄 부근)
if (redirectErrorCode) {
  cancelReservation(orderId);              // ← await 없음
  toast.error(searchParams.get('message') || '결제가 취소되었습니다.');
  return;
}

// 수정
useEffect(() => {
  if (!redirectedPaymentId) return;

  (async () => {
    if (redirectErrorCode) {
      await cancelReservation(orderId);    // ← 취소 완료를 기다림
      toast.error(searchParams.get('message') || '결제가 취소되었습니다.');
      return;
    }
    try {
      await confirmPayment(orderId);
      toast.success('결제가 완료되었습니다.');
      router.replace('/mypage/reservations');
    } catch (error) {
      toast.error(error instanceof Error ? error.message : '결제 확인 실패');
      setSubmitting(false);
    }
  })();
  // eslint-disable-next-line react-hooks/exhaustive-deps
}, []);
```

바로 아래 데스크탑 팝업 경로(128줄)는 이미 `await cancelReservation(orderId)`로 돼 있으니, 두 경로의 처리가 같아지도록 맞추는 작업입니다. `await`가 없으면 취소 요청이 떠나기만 하고 끝나기 전에 컴포넌트가 다음 동작으로 넘어가, 취소가 서버에 반영되기 전에 흐름이 끊길 수 있습니다.
