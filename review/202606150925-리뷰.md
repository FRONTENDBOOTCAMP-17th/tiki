## 3. CI가 잘 들어왔습니다 (`.github/workflows/ci.yml`, 작성: 김연수)

4차에서 권했던 CI가, 그냥 형식만 갖춘 게 아니라 **이 팀이 실제로 겪은 사고를 잡도록** 설계돼 있어 자세히 칭찬합니다.

- **`runs-on: ubuntu-latest`가 핵심입니다.** 리눅스는 파일시스템이 대소문자를 구분합니다. 맥/윈도우 로컬에선 `Button.tsx`를 `button`으로 import해도 그냥 열리지만, 리눅스에선 못 찾고 빌드가 깨집니다. 이 팀을 2·3차 내내 괴롭힌 게 정확히 그 casing 사고였는데, CI를 ubuntu에 올려둔 덕에 **이제 같은 실수를 사람이 아니라 기계가 PR 단계에서 잡습니다.** 주석으로 그 의도를 명시해둔 것도 좋습니다. 문제를 겪고, 원인을 이해하고, 재발을 막는 장치를 만든 흐름이 깔끔합니다.
- **`tsc --noEmit` + `npm run build` 둘 다 돕니다.** 타입 오류와 빌드(use client·프리렌더) 오류를 둘 다 막습니다.
- **`setup-node@v20` 고정 + npm 캐시 + `npm ci`.** node 버전을 고정하고 lockfile로 결정적 설치를 하니, "내 로컬에선 됐는데"가 줄어듭니다. `npm ci`는 lockfile을 그대로 따르므로, lockfile을 깔끔히 재생성해둔 후속 정비도 합리적이었습니다.
- **Supabase 값을 secrets로 주입.** 빌드에 필요한 env를 코드에 하드코딩하지 않고 GitHub Secrets로 넣었습니다. `.env.example`로 필요한 변수만 안내하고, `.gitignore`에 `.env*` + `!.env.example`로 예외를 정확히 잡은 것도 키 유출 방지의 정석입니다.

작은 보완 제안 셋입니다.

- [제안] **트리거가 `on: pull_request` 뿐입니다.** PR을 열면 돌지만, `develop`에 직접 push하거나 머지한 뒤엔 CI가 안 돕니다. 머지 후에도 `develop`이 green인지 보장하려면 `push: branches: [develop]`를 같이 넣어주면 좋습니다(가이드).
- [제안] **lint 단계가 없습니다.** 지금은 `tsc`+`build`만이라 `react-hooks/exhaustive-deps` 같은 건 못 잡습니다. `npm run lint`(eslint) 단계를 추가할 여지가 있습니다.
- [사소] `tsconfig.json`의 `exclude: ["review"]`(커밋 `fbce58b`)는 리뷰용 e2e가 타입체크를 깨는 걸 피하려는 거라 의도는 타당합니다. 다만 앞으로 `exclude`에 `src` 쪽 경로가 섞여 들어가지 않게만 주의해주세요. 실제 코드가 타입체크에서 빠지면 CI가 그 파일을 못 봅니다.

## 4. 리뷰 문화가 코드를 바꾸고 있습니다

이번 회차에서 가장 인상적인 건 **PR 리뷰가 실제 설계를 바꾼 흔적**입니다. 코드만 봐도 보이는데, PR 기록에서 또렷합니다.

- **#47 (정렬):** "useSort 훅을 쓰면 페이지가 클라이언트 컴포넌트로 남으니, context로 상태를 공유하고 `use client`를 지우자"는 코멘트 직후 **SortContext가 도입**됐습니다. 리뷰 한 줄이 컴포넌트 경계를 바꿨습니다.
- **#37 (모달):** 합성 컴포넌트(`Modal.Header`/`Body`/`Footer`)와 PascalCase 파일명을 제안받은 뒤, Modal이 **정확히 그 구조로 리팩토링**됐고(`Modal.tsx:97-99`) 파일명도 통일됐습니다.
- **#56 (필터 2차):** 리뷰 후 APPROVED·머지. 그 외에도 다수 APPROVED/머지 흐름이 건강합니다.

리뷰가 "형식상 승인"이 아니라 **실제로 더 나은 설계로 이어지고 있다**는 게 이 팀의 큰 자산이라 생각합니다. 이런 루프가 한번 자리잡으면 나머지 6개월, 아니 그 이후 실무에서도 계속 갑니다.

한 가지는 짚고 싶습니다. 지금 분포를 보면 **리뷰는 이선우가, 컴포넌트 생산은 방효진·김연수가** 많이 맡은 모양새입니다. PR 생산 수는 방효진/김연수가 더 많고요. 이게 팀이 합의한 역할 분담인가요, 아니면 자연스럽게 그렇게 흘러간 걸까요? 합의된 분담이면 아주 건강한 구조라 생각합니다. 다만 리뷰 안목은 직접 코드를 짜봐야 길러지고, 컴포넌트 설계 감각은 남의 코드를 리뷰하며 넓어지니, 한두 PR쯤은 역할을 바꿔보는 것도 모두에게 도움이 됩니다. 정답이 있는 건 아니니, 팀에서 한번 이야기 나눠보면 좋겠습니다.

## 5. proxy 미들웨어가 세션 쿠키를 못 돌려보냅니다 [필수]

`proxy.ts:4-12`가 지금 대략 이렇습니다.

```ts
const response = NextResponse.next({ request });
const supabase = await createClient();   // supabase.server.ts의 클라이언트
await supabase.auth.getUser();
return response;
```

`getUser()`를 부르면 토큰이 만료됐을 때 Supabase가 토큰을 갱신하고, 갱신된 `sb-...-auth-token` 쿠키를 `setAll`로 다시 써야 합니다. 그런데 여기서 쓴 `createClient()`는 `supabase.server.ts`의 것이고, 그 `setAll`은 서버 컴포넌트용이라 `cookieStore.set`을 **try/catch로 감싸 조용히 실패**하도록 돼 있습니다(서버 컴포넌트에선 쿠키를 못 쓰니까요). 미들웨어 컨텍스트에서도 똑같이 조용히 삼켜집니다.

결과적으로 **갱신된 세션 쿠키가 브라우저로 안 나갑니다.** 토큰이 만료되는 순간 세션이 슬그머니 끊기거나, 매 요청마다 갱신을 다시 시도하는 상황이 생길 수 있습니다. 지금은 세션이 짧게 안 끊겨 잘 보일 수 있는데, 토큰 만료 시점(보통 한 시간)에 드러나는 종류라 더 까다롭습니다.

공식 패턴은 **미들웨어 전용 클라이언트**를 따로 만들어,
- `getAll`은 `request.cookies`에서 읽고,
- `setAll`은 `request`와 `response` **양쪽에** 쿠키를 쓰고,
- 그 `response`를 반환하는 것

입니다. 그래야 갱신된 쿠키가 응답에 실려 브라우저로 나갑니다. 예시 코드는 가이드에 적어뒀습니다.

- [제안] 같은 미들웨어에 **인증 가드**도 아직 없습니다. 공식 패턴은 `user`가 없고 경로가 `/login`·`/auth`가 아니면 `/login`으로 리다이렉트합니다. 지금은 `getUser()`만 부르고 보호는 각 페이지에 맡기는 구조인데, 보호 라우트가 늘어나면 한곳(미들웨어)에서 막는 편이 빠지는 곳 없이 안전합니다.

## 6. Modal 키보드 접근성 — 남은 절반 [제안]

4차에서 짚은 포커스 관리 중 닫힘/스크롤/role은 들어왔고, 키보드 쪽이 남았습니다.

1. **포커스 트랩이 없습니다.** 모달이 열려도 Tab을 누르면 포커스가 뒤 배경 버튼으로 빠져나갑니다. 모달 안에서만 Tab이 돌도록 가둬야 합니다.
2. **닫을 때 포커스 복원이 없습니다.** 모달을 연 그 버튼으로 포커스가 돌아와야 키보드/스크린리더 사용자가 흐름을 잃지 않습니다.
3. **`aria-labelledby`로 제목 연결이 안 됐습니다.** `role="dialog"`는 있는데 헤더에 `id`가 없어 "무슨 다이얼로그인지" 스크린리더가 못 읽습니다. `Modal.Header`에 `id`를 주고 모달 루트의 `aria-labelledby`로 연결하면 됩니다.

세 가지를 Modal 한 곳에 넣으면 이걸 재사용하는 모든 모달이 같이 좋아집니다. 예시는 가이드에 있습니다.

- [사소] Profile에서 아바타를 raw `<img>`로 쓰고 있는데, `next/image`로 바꾸면 최적화/레이아웃 시프트 방지가 따라옵니다. 급한 건 아닙니다.
