# TiKi 20차 리뷰 — 해결 가이드 (2026-07-07)

이번엔 새로 고칠 `[필수]`가 없어서, 발표 전날 손이 덜 드는 마무리 위주로 정리했습니다. 급한 순서가 아니라 "여유 있으면 이 순서로" 정도로 봐주세요.

---

## 1. 최종 재배포가 반영됐는지 확인하기

라이브는 이미 재배포가 돼서 `/checkin`도 살아 있고 sitemap의 `//` 이중슬래시도 사라졌습니다. 남은 건 어젯밤 마지막 머지(좌석 등급 변경·다크모드)까지 올라갔는지 눈으로 확인하는 정도입니다.

- 제일 확실한 건 라이브에서 실제로 바뀐 화면을 직접 열어 보는 방법입니다. 좌석 선택 모달에서 등급 탭을 바꿔 봤을 때 잠겨 있던 탭이 눌리고 이전 좌석이 비워지면, 최신 코드가 반영된 겁니다.
- Vercel 대시보드 Deployments에서 가장 최근 Production 배포의 커밋 해시가 develop 최신(`cdd9000`)과 같은지 봐도 됩니다.
- 자동배포가 걸려 있으면 develop에 머지되는 순간 배포가 도니까, 보통은 몇 분만 기다리면 됩니다.

## 2. 데모 동선 리허설 — 구매부터 입장 확정까지

발표에서 제일 인상적인 건 "티켓을 사서 실제로 입장까지 되는" 흐름 하나입니다. 이번에 입장 코드까지 붙었으니, 시나리오를 이렇게 고정해 두고 두세 번 돌려 보면 좋겠습니다.

1. 일반 사용자로 로그인 → 이벤트 상세 → 좌석 선택(등급 바꿔 보기 포함) → 결제.
2. 마이페이지에서 QR 티켓 열기 → 여기서 QR 아래 "입장 코드 (스캔이 안 될 때)" 8자리가 같이 보이는지 확인.
3. 판매자(또는 배정 스태프)로 로그인 → `/checkin` → 해당 공연 선택 → 스캔 화면.
4. 카메라가 안 되는 환경이면 8자리 코드를 직접 입력 → "입장 가능"이 뜨는지 확인. 같은 코드를 한 번 더 넣으면 "이미 사용된 티켓"이 떠야 정상입니다.

발표장에서는 카메라가 막히는 경우가 많으니, 4번의 코드 입력 폴백을 데모에 넣어 두면 안전합니다. 스캔이 실패하더라도 "이게 오히려 실전을 대비한 폴백"이라고 설명할 수 있으니, 오히려 좋은 소재라 생각합니다.

## 3. `[사소]` robots에 `/example` 막기

`src/app/robots.ts`의 disallow 배열에 `/example` 한 줄만 더해 주면 됩니다. `/example/*`는 개발용 컴포넌트 카탈로그라, 검색엔진에 잡힐 이유가 없거든요.

```ts
disallow: ["/admin", "/mypage", "/seller", "/api", "/payment", "/example"],
```

`/checkin`·`/staff`는 어차피 로그인으로 튕겨서 크롤러가 얻을 게 없으니 굳이 안 넣어도 되지만, 명시적으로 관리 화면 취급하고 싶으면 같이 넣어도 됩니다.

## 4. `[사소]` `issue_entry_code` 왕복 한 번 줄이기 (여유 있을 때)

지금은 `on conflict ... do update` 뒤에 코드를 다시 `select`해서 반환하는데, 이건 `returning`으로 바로 받을 수 있습니다. 동작엔 문제가 없으니, 정말 시간이 남을 때만 손대면 됩니다.

```sql
insert into public.ticket_entry_code (code, subject_type, subject_id, expires_at)
values (v_code, p_subject_type, p_subject_id, v_exp)
on conflict (subject_type, subject_id)
do update set code = excluded.code, expires_at = excluded.expires_at, created_at = now()
returning code into v_code;

return v_code;
```

## 5. (선택) 상세 조회 UUID 이중 가드

`[eventId]` 상세 페이지가 `EVENT_ID_RE`로 이미 비정상 id를 막고 있어서, 실사용상 문제는 없습니다. 다만 `queries.ts`의 `getSlots`·`getGrades`·`getReviews`도 각자 앞단에서 UUID 형식을 한 번 확인해 두면, 나중에 이 함수들이 다른 경로에서 호출되더라도 안전합니다. 당장 급한 건 아니니, 발표 뒤 정리 대상으로 남겨 두면 됩니다.
